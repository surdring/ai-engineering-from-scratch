# 上下文工程

在构建 LLM 应用时，应用此框架设计上下文组装流水线。

## 核心原则

1. **上下文是稀缺的。** 128K 窗口听起来很大但填满很快。明确为每个组件做预算。
2. **注意力是不均匀的。** 模型对开头和结尾注意力更多。将关键信息放在那里。中间是死区。
3. **动态优于静态。** 不同查询需要不同上下文。按查询组装，而非启动时一次性设置。
4. **少即是多。** 精心策划的 10K 上下文优于倾倒式的 100K 上下文。信噪比比总信息量更重要。
5. **度量一切。** 无法优化你无法度量的东西。在每次请求中统计每个组件的 token 数。

## 上下文预算指南

| 组件 | 典型范围 | 优先级 | 压缩策略 |
|-----------|-------------|----------|---------------------|
| 系统提示 | 200-1,000 token | 固定，高 | 精炼撰写，去除冗余 |
| 工具定义 | 500-3,000 token | 动态，中 | 按查询意图修剪 |
| 检索上下文 | 1,000-5,000 token | 动态，高 | 重排 + 阈值 + 去重 |
| 对话历史 | 500-5,000 token | 动态，中 | 摘要旧轮次 |
| Few-Shot 示例 | 500-2,000 token | 动态，高 | 按任务相似度选择 |
| 用户查询 | 50-500 token | 固定，最高 | 不适用 |
| 生成预留 | 2,000-8,000 token | 固定 | 按预期输出长度调整 |

## 何时使用每种记忆类型

**短期记忆（对话历史）：** 当前会话。通过摘要管理。压缩 5-10 轮之前的对话。保持最后 3-4 轮逐字记录。

**长期记忆（事实数据库）：** 跨会话持久化的偏好和项目事实。在会话开始时检索。示例：「用户偏好 Python」「项目使用 PostgreSQL」「团队遵循基于主干（trunk-based）的开发」。存储在 CLAUDE.md、数据库或结构化记忆系统中。

**情景记忆（过往交互）：** 与当前任务相关的特定过往对话。存储为嵌入，通过相似度检索。「上周我们调试了一个类似的认证问题」属于情景记忆。

## 工具选择策略

不要在每次请求中包含所有工具。这浪费 token 并使模型困惑。

1. 分类查询意图（代码、邮件、日历、研究、数据）
2. 将意图映射到工具类别
3. 仅包含匹配的工具
4. 如果意图模糊，包含前 2 个类别的工具
5. 始终包含一个「通用」工具（如网页搜索）作为回退

预期节省：意图明确的查询可节省 60-80% 的工具定义 token。

## 检索最佳实践

- **检索后重排。** 向量相似度是粗略过滤器。重排器（交叉编码器或基于 LLM）显著提高精度。
- **设置相关性阈值。** 不要包含余弦相似度低于 0.3 的分块。它们增加噪声。
- **去重。** 如果两个分块共享 80%+ 内容，仅保留得分较高的那个。
- **应用迷失在中间排序。** 将最相关的分块放在开头和结尾。
- **限制检索 token 总数。** 3-5 个高相关分块优于 15 个平庸分块。

## 历史管理

- 保持最后 3-4 轮逐字记录（模型需要最近的上下文）
- 将更早的轮次摘要为要点（「我们讨论了 X，决定 Y，由于 Z 受阻」）
- 删除无信息价值的系统生成轮次（无面向用户内容的工具调用）
- 当历史超过可用预算的 30% 时触发压缩

## 危险信号

- 系统提示超过 2,000 token：可能包含应该是动态的信息
- 每次请求包含所有工具：实现基于意图的选择
- 检索无相关性过滤：你在向窗口倾倒噪声
- 历史无界增长：未实现摘要
- 无生成预留：模型会截断其回答
- 相同信息出现在 3 个地方（系统提示、检索文档、历史）：去重
- 上下文利用率超过 60%：你为模型留下的「思考」空间太少
