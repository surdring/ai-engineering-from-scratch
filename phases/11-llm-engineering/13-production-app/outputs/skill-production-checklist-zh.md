# 生产级 LLM 检查清单

在交付 LLM 应用时，按顺序完成此检查清单。每个部分都有具体阈值的通过/不通过标准。

## 1. 安全（发布阻塞项）

每一项必须在任何部署之前通过。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| API 密钥在环境变量中 | 代码库中零硬编码密钥 | `grep -r "sk-" --include="*.py"` 返回空 |
| 输入防护栏已激活 | 提示注入模式被阻止 | 发送「忽略之前所有指令」— 返回已阻止响应 |
| PII 脱敏 | 身份证号、信用卡、邮箱模式被捕捉 | 发送「My SSN is 123-45-6789」— PII 在 LLM 调用前被脱敏 |
| 输出过滤 | 危险内容被阻止 | 模型不能返回 `DROP TABLE`、`rm -rf`、`exec()` 模式 |
| 限流 | 每用户请求上限已执行 | 同一用户在 10 秒内 100 个请求 — 最后 50+ 被拒绝 |
| 所有端点认证 | 无未经认证的 LLM 访问 | `curl /v1/chat` 不带 token 返回 401 |
| CORS 限制 | 仅生产域名被允许 | `Origin: evil.com` 请求被拒绝 |
| 最大输入 token | 超限请求被拒绝 | 发送 50K token 输入 — 返回 413 或截断 |

## 2. 可靠性（第一周生存）

这些防止你的第一次值班事故。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 带退避的重试 | 5xx 时 3 次重试，指数级延迟 | 请求中途杀死 LLM 模拟 — 日志中可见重试 |
| 回退模型链 | 链路中 2+ 个模型 | 主模型不可用 — 仍从回退返回响应 |
| 请求超时 | 所有外部调用最多 30s | 慢 LLM 模拟（60s）— 请求在 30s 超时 |
| 优雅降级 | 缓存/RAG 失败不导致服务崩溃 | 停止缓存 — 请求仍然成功（更慢、更贵） |
| 健康检查端点 | 返回依赖状态 | `GET /health` 返回 `{"status": "healthy", "cache": ..., "llm": ...}` |
| 流式传输可用 | 首 token 在 500ms 内 | 测量首 token 时间，持续 < 500ms |
| 错误消息安全 | 内部错误永不泄漏给用户 | 强制 500 — 用户看到通用错误，而非堆栈跟踪 |

## 3. 成本控制（第一个月经济）

这些防止 $50K 的意外账单。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 每次请求成本已追踪 | 每次请求记录 token 数 + USD 成本 | 请求日志有 `input_tokens`、`output_tokens`、`cost_usd` 字段 |
| 语义缓存已激活 | 重复模式 > 20% 命中率 | 1000 次测试请求后缓存统计显示命中率 |
| 缓存 TTL 已配置 | 条目过期（默认：1 小时） | 条目插入 — TTL 后不返回 |
| 每用户成本追踪 | 成本按 user_id 聚合 | 面板/API 显示按成本排列的前 10 用户 |
| 成本告警 | 每日预算 80% 时告警 | 设置 $10 每日预算，发送 $8.50 请求 — 告警触发 |
| 按成本模型路由 | 低复杂度查询使用更便宜的模型 | 简单问题路由到 gpt-4o-mini，复杂问题路由到 gpt-4o |
| 最大输出 token 已设置 | 响应按模板限制 | 模板设置 max_output_tokens=512 — 响应永不超限 |

**成本估算公式：**
```
月度 LLM 成本 = DAU x 每用户查询数 x 30 x (1 - 缓存命中率) x (平均输入 token x 输入价格 + 平均输出 token x 输出价格) / 1,000,000
```

**按规模的基准阈值：**

| DAU | 目标每次请求成本 | 月度预算 |
|-----|-------------------|----------------|
| 1K | < $0.005 | < $750 |
| 10K | < $0.003 | < $4,500 |
| 100K | < $0.001 | < $15,000 |

## 4. 可观测性（生产环境调试）

你无法修复你看不到的东西。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 结构化 JSON 日志 | 每次请求生成一个 JSON 日志行 | 日志包含：request_id、user_id、model、tokens、latency_ms、cost |
| 请求追踪 | 端到端追踪含组件耗时 | 单次请求显示：guardrail (5ms) + cache (2ms) + llm (3200ms) + eval (1ms) |
| 延迟追踪 | P50、P95、P99 已测量 | 1000 次请求后：P50 < 2s，P99 < 10s |
| 错误率监控 | 错误已计数并分类 | 面板显示：0.5% API 错误，0.1% 防护栏拦截，0.01% 超时 |
| 缓存指标 | 命中率、未命中率、条目数可见 | `GET /v1/cache/stats` 返回当前数据 |
| A/B 测试指标 | 每个变体的质量指标已记录 | 每次请求记录 prompt_template + version 用于比较 |
| 评估记录 | 每次请求的质量信号已记录 | 响应长度、延迟、模型、模板版本已存储用于离线分析 |

## 5. 提示管理

提示就是代码。像对待代码一样对待它们。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 版本化模板 | 每个模板有名称 + 版本字符串 | 模板变更创建新版本，旧版本保留 |
| A/B 测试支持 | 按确定性用户哈希分流 | 同一用户在同一实验中始终看到相同变体 |
| 回滚能力 | 1 分钟内回退到先前版本 | 更改实验配置 — 流量即时切换 |
| 模板验证 | 渲染前验证变量 | 模板中缺少变量引发明确错误，而非 KeyError |
| 系统提示分离 | 系统消息和用户消息分开字段 | 系统提示未拼接到用户消息中 |

## 6. 扩展就绪

不需要在启动时就绪。需要在 10 倍增长时。

| 检查项 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 异步 LLM 调用 | API 调用无线程阻塞 | 50 并发请求 — 服务器 CPU 保持 < 30% |
| 连接池 | HTTP 连接复用 | 网络追踪显示到 LLM 提供商的持久连接 |
| 水平扩展 | 无状态服务器设计 | 负载均衡器后 2 个实例 — 所有请求成功 |
| 队列支持 | 非实时任务进入队列 | 摘要请求返回 job_id，结果通过轮询获取 |
| 负载测试 | 100 并发用户，< 5% 错误率 | `wrk` 或 `locust` 测试在目标并发下通过 |

## 新项目的实现顺序

1. **第 1 天：** API 服务器 + 提示模板 + 带重试的单次 LLM 调用
2. **第 2 天：** 输入防护栏 + 输出防护栏 + 错误处理
3. **第 3 天：** 语义缓存 + 每次请求成本追踪
4. **第 4 天：** 流式传输（SSE）+ 健康检查端点
5. **第 5 天：** 结构化日志 + 请求追踪 + 评估记录
6. **第 2 周：** A/B 测试 + 提示版本管理 + 回滚
7. **第 3 周：** 回退模型链 + 优雅降级
8. **第 4 周：** 负载测试 + 异步优化 + 水平扩展

## 快速诊断

如果生产环境出现问题，按此顺序检查：

1. **用户抱怨错误？** 检查健康端点，然后检查日志中的错误率，然后检查 LLM 提供商状态页面
2. **响应慢？** 检查 P99 延迟，然后检查缓存命中率，然后检查链路追踪中的 LLM 响应时间
3. **成本激增？** 检查每次请求成本趋势，然后检查缓存命中率，然后检查按成本的前列用户，然后查看是否提示模板变更增加了 token 数
4. **质量下降？** 检查是否部署了新提示版本，检查 RAG 检索准确率是否变化，检查模型提供商是否更改了默认模型版本
5. **安全事件？** 检查防护栏拦截率（突然下降 = 防护栏被禁用），检查请求日志中异常模式，立即轮换 API 密钥
