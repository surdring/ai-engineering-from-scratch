# LLM 成本优化模式

在构建需要控制成本的 LLM 应用时，应用此决策框架。

## 何时优化

**立即优化的情况：**
- 月度 LLM 支出超过 $500 或基础架构预算的 10%
- 消费者产品的每次查询成本超过 $0.01
- 系统提示超过 1,000 token 且每次请求都发送
- 超过 30% 的查询是重复或近似重复的
- 正在从 100 扩展到 10,000+ 日活用户

**暂时不优化的情况：**
- 少于 100 DAU 且仍在验证产品-市场契合度
- 月度支出低于 $100 且增长缓慢
- 仍在迭代提示设计（缓存会锁定提示）

## 缓存策略选择

### 精确缓存

**适用场景：** temperature=0，相同提示重复出现，需要确定性输出。

```python
key = sha256(json.dumps({"model": m, "messages": msgs, "temp": 0}))
```

- 实现时间：30 分钟
- 命中率：大多数应用 10-25%，FAQ 机器人 40-60%
- 延迟：<1ms（字典查找）
- 风险：底层数据变化时返回过期响应

**不适用场景：** temperature > 0，每次查询都唯一，需要实时数据。

### 语义缓存

**适用场景：** 用户用不同措辞问相同问题，FAQ 为主的产品，客户支持。

- 实现时间：2-4 小时（嵌入 + 相似度 + 存储）
- 命中率：在精确缓存之上增加 15-35%
- 延迟：10-50ms（嵌入 + ANN 搜索）
- 风险：误报（对相似但不同的问题返回错误缓存答案）

**阈值指南：**
- 0.98+：非常保守，几乎没有误报，命中率较低
- 0.95：事实型问答的良好平衡
- 0.90：激进，命中率高但有错误答案风险
- 0.85：仅用于低风险应用（建议、自动补全）

**不适用场景：** 每次查询都有唯一上下文（代码生成），响应必须反映最新数据，查询空间无界。

### 提供商提示缓存

**适用场景：** 系统提示 > 1,024 token（OpenAI）或模型特定最小值，相同前缀重复发送。

| 提供商 | 操作 | 节省 |
|----------|--------|---------|
| Anthropic | 为系统消息添加 `cache_control: {"type": "ephemeral"}` | 缓存前缀 90%（扣除 25% 写入溢价） |
| OpenAI | 无需操作（自动） | 缓存前缀 50% |
| Google | 使用 Context Caching API 并显式设置 TTL | 缓存上下文约 75% |

**不适用场景：** 系统提示每次请求变化，提示长度低于最小值。

## 模型路由规则

### 基于关键词（简单、快速）

```
simple:  <= 5 词 OR 匹配 FAQ 关键词 -> gpt-4o-mini ($0.15/$0.60)
medium:  一般查询、摘要              -> claude-sonnet ($3/$15)
complex: "分析"、"比较"、"调试"      -> gpt-4o ($2.50/$10)
```

- 实现时间：1 小时
- 准确率：70-80%
- 节省：40-60% 模型成本

### 基于嵌入（更准确）

按类别嵌入 50-100 个标注查询。通过最近邻分类新查询。

- 实现时间：4-8 小时
- 准确率：85-92%
- 节省：50-70% 模型成本
- 额外成本：约 $0.02/1M tokens 分类嵌入费用（可忽略）

### 基于机器学习（生产级）

在历史查询/模型对上训练一个小型分类器（逻辑回归或小 BERT）。

- 实现时间：1-2 周
- 准确率：90-95%
- 节省：60-75% 模型成本
- 需要：来自生产流量的标注训练数据

## 限流配置

### 按层级的令牌桶参数

| 层 | 桶大小 | 补充速率 | 最大 RPM | 每日上限 |
|------|-------------|-------------|---------|-----------|
| 免费 | 50K tokens | 500/秒 | 10 | 50K |
| 专业 | 500K tokens | 5K/秒 | 60 | 500K |
| 企业 | 5M tokens | 50K/秒 | 300 | 5M |

### 实现检查清单

1. 在 Redis 中存储桶（非内存），用于多实例应用
2. 使用原子操作（MULTI/EXEC）防止竞态条件
3. 拒绝响应中返回 `Retry-After` 头
4. 将被拒绝的请求作为指标跟踪（>5% 拒绝率 = 层级限制过紧）
5. 实现优雅降级：先拒绝昂贵模型请求，保留廉价模型访问

## 预算控制

### 三级熔断器

| 阈值 | 操作 | 可逆性 |
|-----------|--------|------------|
| 月预算的 70% | 记录警告，通过 Slack/PagerDuty 告警团队 | 是（自动） |
| 月预算的 85% | 将所有流量路由到最便宜的模型 | 是（自动，下个计费周期） |
| 月预算的 95% | 仅提供缓存响应，拒绝新的 LLM 调用 | 是（手动重置或下个周期） |

### 每用户成本跟踪

跟踪每个用户的累计成本。标记超过中位数 10 倍的用户。常见原因：
- 合法的重度用户（升级其层级）
- 提示注入循环（机器人发送自动请求）
- 低效集成（客户端在每次错误时重试）

## 成本跟踪字段

记录每次 API 调用并包含以下字段：

```json
{
  "timestamp": "2026-04-02T10:30:00Z",
  "model": "gpt-4o",
  "input_tokens": 1523,
  "output_tokens": 487,
  "cached_input_tokens": 1024,
  "latency_ms": 1847,
  "cost_usd": 0.006142,
  "user_id": "user_abc123",
  "cache_status": "partial_hit",
  "request_category": "customer_support",
  "complexity_class": "medium",
  "routed_from": "gpt-4o"
}
```

### 需展示的关键指标

- **每次查询成本**（P50、P95、P99）— 按模型、按功能、按用户层级
- **缓存命中率** — 精确 vs 语义，随时间趋势
- **模型分布** — 每个模型的流量百分比、每个模型的成本
- **预算消耗速率** — 当前支出 vs 按当前速率预估的月度支出
- **拒绝率** — 被限流的请求百分比，按层级

## 常见错误

| 错误 | 为何有害 | 修复方案 |
|---------|-------------|-----|
| temperature > 0 时使用缓存 | 非确定性输出，过期缓存给出错误的变化性 | 仅缓存 temp=0 调用，或接受缓存响应失去随机性 |
| 语义缓存阈值过低 | 对表面相似的查询返回错误答案 | 从 0.95 开始，仅在测量误报率后降低 |
| 无缓存失效机制 | 底层数据变化时响应过期 | 设置 TTL（动态数据 1 小时，静态数据 24 小时），数据更新时失效 |
| 所有流量路由到最便宜模型 | 质量下降，用户会注意到 | 按复杂度路由，衡量每层质量，设置最低质量阈值 |
| 无每用户限制 | 一个滥用用户耗尽全部预算 | 始终实现每用户配额，即使很慷慨 |
| 忽略输出 token | 输出 token 成本是输入的 2-5 倍 | 适当设置 max_tokens，使用停止序列，压缩输出 |
| 提示稳定前启用缓存 | 缓存被旧提示的响应填满 | 仅在提示最终定稿后启用缓存，提示变更时刷新缓存 |
