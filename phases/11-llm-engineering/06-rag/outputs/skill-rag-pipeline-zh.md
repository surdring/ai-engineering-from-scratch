# RAG 流水线模式

每个 RAG 系统遵循此模式：

```
文档 -> 分块 -> 嵌入 -> 存储
查询 -> 嵌入 -> 搜索(top_k) -> 构建提示 -> 生成
```

索引每个文档执行一次。查询在每次用户请求时执行。

## 何时使用 RAG

- LLM 需要访问私有或最新文档
- 微调太贵或更新太慢
- 需要为答案引用来源
- 知识库频繁变化

## 何时不使用 RAG

- 答案是 LLM 已具备的通识知识
- 任务是创意类（写作、头脑风暴）而非事实类
- 你需要模型采用特定的推理风格（使用微调）

## 实现检查清单

1. 将文档分割为 256-512 token 的片段，50 token 重叠
2. 使用一致的嵌入模型嵌入每个分块
3. 将嵌入连同原始文本存储在向量数据库中
4. 查询时，使用相同模型嵌入用户问题
5. 通过余弦相似度检索 top-k（5-10）个最相似分块
6. 构建提示：系统指令 + 检索上下文 + 用户问题
7. 生成答案，将其锚定在检索到的上下文中
8. 返回答案及来源引用

## 常见错误

- 索引和查询使用不同嵌入模型（向量不兼容）
- 分块太小（失去上下文）或太大（稀释相关性）
- 分块之间未包含重叠（在边界处截断句子）
- 文档变更后忘记重新索引
- 将检索到的分块直接返回给用户而不生成连贯答案
- 对事实类 RAG 查询不设置 temperature=0（更高温度 = 更多幻觉）

## 调试检索

如果正确的分块未被检索到：
1. 打印查询嵌入并验证非零
2. 手动检查已知相关分块的余弦相似度
3. 尝试改写查询以匹配文档词汇
4. 验证索引和查询时的嵌入模型一致
5. 检查相关内容是否在分块过程中丢失

## 生产参数

- 分块大小：256-512 token
- 重叠：50 token（分块大小的 10-20%）
- Top-k：大多数用例 5-10
- 温度：事实类答案用 0
- 嵌入模型：text-embedding-3-small（经济高效）或 text-embedding-3-large（更高准确率）
