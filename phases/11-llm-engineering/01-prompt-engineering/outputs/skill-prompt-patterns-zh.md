# 提示模式选择指南

在构建 LLM 驱动的功能时，先选择提示模式再编写提示。模式决定结构。内容填充其中。

## 模式决策矩阵

| 任务类型 | 主要模式 | 辅助模式 | 温度 | 需要 Few-Shot？ |
|-----------|----------------|-------------------|-------------|-----------------|
| 数据抽取 | 模板填充 | Few-Shot | 0.0 | 是（2-3 个示例） |
| 分类 | Few-Shot | 护栏 | 0.0 | 是（3-5 个示例） |
| 摘要 | 人设 + 模板 | 受众适配 | 0.3 | 否 |
| 代码生成 | 人设 | 思维链 | 0.0 | 可选 |
| 创意写作 | 人设 | 批判 | 0.7-1.0 | 否 |
| 多步推理 | 思维链 | 分解 | 0.3 | 可选 |
| 问答 | 人设 + 护栏 | 边界 | 0.3 | 否 |
| 提示生成 | 元提示 | 批判 | 0.7 | 是（1-2 个示例） |
| 内容审核 | 护栏 + 边界 | Few-Shot | 0.0 | 是（5+ 个示例） |
| 翻译/适配 | 受众适配 | Few-Shot | 0.3 | 是（2-3 个示例） |

## 何时使用每种模式

**人设模式（Persona Pattern）**：作为基线用于每条提示。唯一的问题是角色要多具体。对于通用任务，宽泛的角色就足够了。对于领域特定任务，角色应说明领域、资历级别和上下文。

**Few-Shot 模式**：当输出格式比内容更重要时使用。如果模型需要生成特定的 JSON 形态、CSV 格式或分类标签，示例比指令更有效。经验法则：简单格式用 2-3 个示例，复杂或模糊格式用 5 个以上。

**思维链模式（Chain-of-Thought Pattern）**：用于数学、逻辑、多步分析以及任何需要模型「展示计算过程」的任务。在推理任务上将准确率提升 10-40%（Wei et al., 2022）。不要用于简单的事实查询或抽取 — 这会浪费 token。

**模板填充模式（Template Fill Pattern）**：用于结构化抽取，每个输出必须具有相同形态。配合 temperature=0.0 和缺失字段的显式「N/A」处理效果最佳。

**批判模式（Critique Pattern）**：当质量比速度更重要时使用。模型生成、批判并改进。约将 token 成本翻倍，但显著提高准确率和完整性。最适合高风险输出（报告、建议、面向公众的内容）。

**护栏模式（Guardrail Pattern）**：用于任何面向用户的系统。始终包含：范围边界、超出范围的拒绝行为、显式的「我不知道」处理。与应用侧的输入验证结合使用。

**元提示模式（Meta-Prompt Pattern）**：用于为新任务生成提示。不需要从头编写提示，而是描述任务并让模型编写提示。然后测试和迭代。节省初始提示开发的时间。

**分解模式（Decomposition Pattern）**：用于受益于分而治之的复杂问题。模型将问题分解成若干部分，逐一求解，然后合并。对 3-7 个子问题的任务最有效。

**受众适配模式（Audience Adaptation Pattern）**：当相同内容需要服务于不同受众时使用。显式指定受众 — 不要依赖模型从上下文中猜测。

**边界模式（Boundary Pattern）**：用于绝不能回答特定类型问题的生产系统。比护栏更强，因为它定义了一个带有精确拒绝消息的硬边界。对合规敏感领域至关重要。

## 跨模型兼容性

各模式在 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro 和 Llama 3 上的一致性排名：

| 模式 | 跨模型一致性 | 备注 |
|---------|------------------------|-------|
| Few-Shot | 非常高 | 示例在所有模型间迁移良好 |
| 模板填充 | 非常高 | 显式结构留下的差异空间很小 |
| 思维链 | 高 | 所有主流模型均支持「逐步思考」 |
| 人设 | 高 | 各处有效，但不同模型对不同角色具体程度响应不同 |
| 护栏 | 中等 | Claude 最严格遵循护栏；GPT-4o 在长对话中有时漂移 |
| 批判 | 中等 | 自我批判的质量因模型差异显著 |
| 元提示 | 中等 | GPT-4o 和 Claude 产生不同风格的提示 |
| 边界 | 低-中等 | 拒绝行为各异；按模型逐一测试 |

## 常见错误

1. **对所有任务使用思维链**：CoT 增加 token 和延迟。仅在需要推理步骤时使用。
2. **约束过多**：超过 5-7 个约束后模型开始丢弃部分。优先最重要的 3 个。
3. **矛盾的人设 + 约束**：「你是一个创意作家」+「永远不要使用比喻」会让模型困惑。
4. **未指定温度**：在你需要确定性输出时，将温度留在默认值（通常为 1.0）。
5. **在模型间复制粘贴提示**：始终要测试。为 GPT-4o 优化的提示可能在 Claude 上表现不佳，反之亦然。
6. **忽略系统消息**：将所有内容放在用户消息中，而非利用系统消息存放持久规则。
7. **过度依赖负向约束**：「不要做 X, Y, Z, A, B, C」不如「只做 W」有效。正向框架给模型一个清晰的目标。

## 可靠性目标

| 用例 | 模式组合 | 预期准确率 | Token 成本 |
|----------|-------------------|-------------------|------------|
| 生产级抽取 | 模板 + Few-Shot | 95% 以上 | 低（500-1K） |
| 面向用户问答 | 人设 + 护栏 + 边界 | 90% 以上 | 中（1-2K） |
| 代码生成 | 人设 + 思维链 | 85% 以上 | 中（1-3K） |
| 内容生成 | 人设 + 批判 | 90% 以上质量 | 高（2-4K，双次通过） |
| 分类 | Few-Shot + 护栏 | 95% 以上 | 低（300-800） |
| 复杂分析 | 分解 + 思维链 | 85% 以上 | 高（3-5K） |
