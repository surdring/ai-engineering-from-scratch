# 微调决策指南

在微调之前，按顺序尝试以下：

```
1. 提示工程（分钟级，$0）
2. 提示中添加 Few-Shot 示例（分钟级，$0）
3. RAG 知识检索（天级，$10-100/月）
4. 使用 LoRA/QLoRA 微调（天级，$5-50/每次实验）
5. 全量微调（周级，$100-10,000/每次运行）
```

仅在上一步可测量地不足时进入下一步。

## 何时微调

- 模型需要提示工程无法实现的一致输出风格或格式
- 你在蒸馏更大的模型（用 8B 模型达到 GPT-4 质量）
- 延迟重要且 Few-Shot 示例增加过多 token
- 你需要模型可靠地遵循复杂的推理模式
- 你有 1,000+ 个所需输入输出行为的高质量示例

## 何时不微调

- 模型用正确的提示已经能做到你需要的
- 你需要模型了解事实（改用 RAG）
- 你只有不到 500 个训练样本（可能过拟合）
- 任务频繁变化（重新训练成本高）
- 你需要审计哪些数据影响了特定输出（微调是黑盒）

## 方法选择

| GPU VRAM | 7B 模型 | 13B 模型 | 70B 模型 |
|----------|----------|-----------|-----------|
| 16GB (T4) | QLoRA | 不可行 | 不可行 |
| 24GB (3090/4090) | QLoRA 或 LoRA | QLoRA | 不可行 |
| 40GB (A100) | LoRA 或全量 | QLoRA 或 LoRA | QLoRA |
| 80GB (A100/H100) | 全量 | LoRA 或全量 | QLoRA 或 LoRA |

## LoRA 配置检查清单

1. 从 r=16，alpha=32 开始（大多数任务的安全默认值）
2. 首先目标 q_proj 和 v_proj（最小可行 LoRA）
3. QLoRA 使用学习率 2e-4，LoRA fp16 使用 5e-5
4. 设置 lora_dropout=0.05
5. 训练 1-3 轮（更多轮次有过拟合风险）
6. 每 100 步在留出集上评估
7. 保存检查点并按评估损失选择最佳

## 常见错误

- 训练轮次过多（小数据集上第 2-3 轮后过拟合）
- 使用与全量微调相同的学习率（LoRA 需要更高学习率）
- 忘记设置 pad token（Llama 模型下会导致 NaN 损失）
- 没有冻结基础模型（违背 LoRA 的目的）
- 仅在训练数据上评估（始终留出 10-20% 用于评估）
- 跳过提示工程基线（微调一个提示工程已经解决的问题）

## 质量验证

训练后，在 200+ 留出样本上比较：
1. 带最佳提示的基础模型（基线）
2. 带 LoRA 适配器的基础模型（你微调的模型）
3. 带相同提示的 GPT-4 或 Claude（上限）

如果 LoRA 模型没有击败提示基线，你的训练数据或配置需要改进，而非更多计算。

## 适配器管理

- 多任务服务时保持适配器独立（每次请求切换适配器）
- 单任务部署时将适配器合并到基础权重中
- 将适配器存储在 Hugging Face Hub（10-100MB，易于版本管理和分享）
- 部署前验证合并模型输出与非合并输出匹配
- 使用 TIES-Merging 或 DARE 将多个适配器合并为一个

## 调试训练

如果损失不下降：
1. 检查学习率（LoRA 太低了，尝试 2e-4）
2. 验证 LoRA 层确实在接收梯度
3. 确认基础模型权重已冻结
4. 检查数据格式（分词器必须与模型预期格式匹配）

如果损失下降但评估质量差：
1. 训练数据质量问题（垃圾进，垃圾出）
2. 过拟合（减少轮次，增加 dropout，增加数据）
3. 错误的目标模块（复杂任务添加 MLP 层）
4. 秩太低（尝试 r=32 或 r=64）
