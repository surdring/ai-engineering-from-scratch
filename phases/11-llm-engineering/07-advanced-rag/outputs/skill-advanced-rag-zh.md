# 高级 RAG 模式

基础 RAG：嵌入查询 -> 向量搜索 -> top-k -> 生成。
高级 RAG：嵌入查询 + BM25 -> 融合排名 -> 重排 -> top-k -> 生成。

```
查询 -> [向量搜索 (top-50)] -+-> RRF 融合 -> 重排器 (top-5) -> 提示 -> LLM
                                 |
查询 -> [BM25 搜索 (top-50)]  --+
```

## 何时从基础 RAG 升级

- 检索质量低于 70% Recall@5
- 用户报告错误或不相关的答案
- 语料库增长超过 100K 个分块
- 查询使用与文档不同的词汇
- 多跳问题持续失败

## 实现检查清单

1. 在向量索引旁添加 BM25 索引
2. 并行运行两种搜索（各 top-50）
3. 使用倒数排名融合（RRF，k=60）合并
4. 用交叉编码器重新排序 top 候选
5. 取 top-5 用于最终提示
6. 在测试集上添加忠实性评估

## 技术选择指南

- **混合搜索**：生产环境始终使用。查询时无额外成本。
- **重排**：当 Recall@50 好但 Recall@5 差时使用。增加 50-200ms 延迟。
- **HyDE**：当查询模糊或使用与文档不同的词汇时使用。增加一次 LLM 调用。
- **父子分块**：当小分块缺乏上下文但大分块稀释相关性时使用。
- **元数据过滤**：当语料库有清晰类别（日期、来源类型、部门）时使用。
- **查询分解**：用于需要来自多个文档信息的多跳问题。

## 常见错误

- 在不同分块集上运行 BM25 和向量搜索（必须搜索相同语料库）
- 重排的候选池太小（top-10 太少；使用 top-50）
- 对每次查询都添加 HyDE（仅在词汇不匹配是瓶颈时有用）
- 不评估变更（在每种技术前后测量 Recall@k）
- 在测量失败位置之前过度工程化流水线

## 评估工作流

1. 创建 50+ 个带已知答案分块的测试问题
2. 测量每种检索方法的 Recall@5 和 Recall@10
3. 对检索成功的查询，测量生成答案的忠实性
4. 随语料库增长每周跟踪指标
5. 在添加更多技术之前调查个别失败
