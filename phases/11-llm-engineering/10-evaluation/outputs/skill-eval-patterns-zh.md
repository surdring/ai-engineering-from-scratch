# 评估模式

在为 LLM 应用构建评估时，应用此决策框架。

## 选择评估方法

**使用自动化指标（BLEU、ROUGE、BERTScore）的场景：**
- 每个测试用例都有参考答案
- 速度比细微差别更重要（10,000+ 用例）
- 在昂贵评估之前需要一个廉价的首轮过滤器
- 专门评估翻译或摘要任务

**使用 LLM 作为评判员的场景：**
- 质量是主观的（有帮助性、语气、完整性）
- 并非每个用例都有参考答案
- 需要评估安全性、偏见或策略合规性
- 比较不同的提示版本或模型版本
- 预算允许约 $20/每 1,000 次评估调用

**使用人工评估的场景：**
- 校准 LLM 评判员（同时运行两者，测量相关性）
- 评估评判员可能出错的边缘情况
- 高风险领域（医疗、法律、金融）
- 初始量规设计 — 人类定义「好」的含义
- 需要为利益相关者提供可辩护的结果

**三者结合使用的场景：**
- 启动新应用（随着规模扩大：人工 -> LLM 评判 -> 自动化）
- 季度审计（每日自动化，PR 上 LLM 评判，季度人工）

## 量规设计原则

### 锚定量规优于非锚定量规

非锚定：「请按 1-5 分评价回答质量。」
锚定：「5：事实正确，直接回答问题，包含具体示例。」

锚定量规可将评分者间分歧降低 30-40%。每个级别必须描述具体、可观察的行为。

### 三种量规架构

**逐点评分（每标准 1-5）**：独立评分每个输出。简单、可扩展、适用于 CI。存在尺度漂移 — 评判员今天称为「4」的明天可能变成「3」。

**成对比较（A vs B）**：展示两个输出，选择更好的。消除尺度校准问题。最适合比较两个特定版本。不生成绝对质量分数。

**N 选最佳**：生成 N 个输出，评判员选择最佳。衡量系统的上限。如果 5 选最佳显著优于 1 选最佳，则推理时适合使用采样 + 选择。

### 标准选择指南

| 应用 | 推荐标准 |
|------------|---------------------|
| 客户支持聊天机器人 | 相关性、正确性、有帮助性、安全性、语气 |
| 代码生成 | 正确性、完整性、代码质量、安全性 |
| RAG/问答 | 相关性、忠实性、正确性、完整性 |
| 摘要 | 忠实性、完整性、简洁性 |
| 创意写作 | 相关性、创意性、风格、连贯性 |
| 分类 | 准确率、校准度（置信度 vs 正确性） |
| 多轮对话 | 连贯性、记忆、有帮助性、安全性 |

## 测试套件规模

### 最小样本量

| 决策 | 最小用例数 | 原因 |
|----------|-------------|-----|
| 快速健全检查 | 20-50 | 仅捕捉灾难性故障 |
| PR 级回归测试 | 100-200 | 检测 5-10% 的质量变化 |
| 部署决策 | 200-500 | 对 5% 差异具有统计显著性 |
| 模型比较 | 500-1000 | 区分表现相近的系统 |
| 发表级 | 1000+ | 窄置信区间，按类别分析 |

### 数学基础

N 个测试用例，观察准确率为 p，95% Wilson 置信区间宽度约为：

- N=50, p=0.9：宽度 = 0.19（对接近的比较无用）
- N=200, p=0.9：宽度 = 0.09（适用于部署决策）
- N=500, p=0.9：宽度 = 0.05（适用于模型比较）
- N=1000, p=0.9：宽度 = 0.03（发表级）

如果两个系统的置信区间重叠，你不能声称其中一个更好。

## 回归测试工作流

### 对每个涉及提示或 LLM 代码的 PR

1. 加载黄金测试集（100-200 用例）
2. 运行基线提示 — 如有缓存分数则加载
3. 运行新提示
4. 用 LLM 评判员按 4 个标准评分
5. 计算每个标准的均值和 Bootstrap 置信区间
6. 标记任何均值回归 > 0.3 分的标准
7. 标记任何新置信区间下界低于基线置信区间下界的标准
8. 如果没有标记 — 自动批准评估检查
9. 如果有标记 — 需要人工审查被标记的测试用例

### 每周全面评估

1. 从生产流量中抽样 500 用例
2. 对当前生产提示运行
3. 与上周基线比较
4. 计算每个类别的分数
5. 任何类别回归 > 5% 时告警
6. 如果分数稳定或提升则更新基线

### 每月校准

1. 从每周评估中抽样 50 用例
2. 让 2 名人工评分员评分
3. 计算 LLM 评判员与人工分数的相关性
4. 如果相关性低于 0.75 — 重新调整量规或更换评判模型
5. 归档校准结果用于审计追踪

## 成本管理

### 按评估频率的预算

| 评估类型 | 频率 | 用例数 | 评判每次成本 | 月成本（每周 10 PR） |
|-----------|-----------|-------|--------------------|---------------------------|
| PR 评估 | 每次 PR | 200 | ~$16 (GPT-4o) | ~$640 |
| 每周全面 | 每周 | 500 | ~$40 | ~$160 |
| 每月校准 | 每月 | 50 (人工) | ~$25 (人工时间) | ~$25 |
| **总计** | | | | **~$825/月** |

### 成本降低策略

- **缓存基线分数**：仅在测试套件变化时重新评分基线，而非每次运行
- **使用更便宜的评判员进行筛选**：首先运行 GPT-4o-mini，将边界用例（2-4 分）升级到 GPT-4o
- **分层评估**：首先运行 ROUGE-L（免费），仅对通过 ROUGE 阈值的用例进行评判评分
- **稳定标准子采样**：如果安全性分数持续 5/5，仅抽取 20% 用例进行安全性评估而非 100%
- **批量 API 定价**：OpenAI Batch API 便宜 50% — 用于非时间敏感的每周/每月评估

## CI/CD 集成模式

### GitHub Actions

触发条件：任何修改 `prompts/`、`src/llm/` 或 `config/model*.yaml` 的 PR

步骤：
1. 检出代码
2. 安装评估依赖（deepeval、promptfoo 或自定义）
3. 对 PR 分支运行评估套件
4. 与缓存的基线分数比较
5. 将结果作为 PR 评论发布（标准表格、通过/失败、差异）
6. 设置检查状态：如无回归则通过，任何标准回归则失败

### 评估作为合并门禁

评估检查应为合并的**必选项**，而非建议项。对待它就像对待失败的测试套件。如果评估显示 BLOCK，PR 不合并，直到修复回归或更新测试用例并附上理由。

### 存储结果

将评估结果存储为 JSON 构件：
- PR 编号、commit SHA、时间戳
- 每个测试用例的分数及评判理由
- 带置信区间的聚合指标
- 与基线的比较差异

使用这些构件进行趋势分析。连续 8 周每周下降 0.1 分，累积就是 0.8 分的回归，而任何单个 PR 检查都无法捕捉到。

## 应避免的反模式

| 反模式 | 为何失败 | 修复方案 |
|-------------|-------------|-----|
| 凭感觉评估 | 人类无法感知 5% 的回归 | 使用统计检验的自动化评分 |
| 在提示示例上测试 | 衡量的是记忆而非泛化 | 将评估数据与提示示例分开 |
| 单一指标 | 优化正确性会损害有帮助性 | 至少评分 3-5 个标准 |
| 无基线 | 「4.2/5」没有比较就没有意义 | 始终与已知良好的版本比较 |
| 评判模型太弱 | GPT-3.5 产生嘈杂、不一致的分数 | 使用 GPT-4o 或 Claude Sonnet 作为评判 |
| 测试用例太少 | 50 个用例置信区间宽度 19 分 — 无用 | 部署决策至少 200 |
| 静态测试套件 | 分布漂移使旧测试无关 | 每月从生产流量刷新 |
| 忽略按类别分数 | 整体改进可能掩盖类别回归 | 按类别报告带置信区间 |
| 仅在发布时评估 | 质量随时间退化（模型更新、数据漂移） | 持续评估 — 至少每周 |

## 评判提示模板

### 逐点评分

```
你正在评估一个 AI 助手回复的质量。

输入：{input}
参考答案（黄金标准）：{reference}
AI 回复：{output}

使用提供的量规为 AI 回复的每个标准评分。

{量规}

以 JSON 返回评估结果：
{"scores": {"criterion": {"score": N, "reasoning": "一句话理由"}}}
```
