给定提议的 MCP 服务器的能力（以自然语言或工具列表草案形式），将每个归类为工具、资源或提示，附一句话论证。

生成：

1. 每能力归类。对每个项，返回 `{name, primitive: tool | resource | prompt, rationale}`。
2. 资源 URI 方案。如果有任何能力成为资源，提议 URI 方案（`notes://`、`gh://`、`db://`）和模板模式。
3. 提示参数骨架。如果有任何能力成为提示，提议参数列表和必需/可选标志。
4. 订阅候选者。标记经常变化且会受益于 `resources/subscribe` 的资源。
5. 反模式标志。指出旧设计将读取包装在工具中的案例（如 `notes_read(id)`），而资源更适合。

硬拒绝：
- 任何归类为「既是工具又是资源」而没有拆分的能力。选一个或搭建一对。
- 任何没有确定必需参数的提示。暴露在斜杠命令 UI 中需要参数 Schema。
- 任何不可寻址的资源 URI 方案（自由格式字符串，非 URI）。

拒绝规则：
- 如果所有能力都归为工具，拒绝并询问服务器是否有可成为资源的只读数据。
- 如果没有能力适合提示，这没问题；提示是可选的。不要编造。
- 如果服务器的领域更适合 A2A（智能体间协作、不透明状态），拒绝并重定向到 Phase 13 · 19。

输出：一页决策报告，包含归类表、URI 方案提议、提示骨架和订阅标志。以对该服务器最有效的工具 -> 资源转换建议结尾。
