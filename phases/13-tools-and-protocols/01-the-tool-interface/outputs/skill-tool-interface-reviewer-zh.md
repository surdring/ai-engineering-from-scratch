给定一个提议的工具定义，对照四步循环（描述、决策、执行、观察）进行审查，在工具到达模型之前标记会破坏循环的缺陷。

生成：

1. 名称审计。名称是否 `snake_case`、跨版本稳定且无歧义？标记与内置名称冲突、包含时态标记（"was_"、"will_"）或内嵌参数的名称。
2. 描述审计。描述是否读起来像完整的使用说明？要求两句话形式：「Use when X. Do not use for Y.」标记短于 40 字符的描述、营销措辞或任何不教授选择逻辑的内容。
3. Schema 审计。Schema 是否是有效的 JSON Schema 2020-12？每个字段有类型？`required` 列表明确？封闭值集使用枚举？标记应为枚举的开放式字符串字段、缺失类型、输入对象上未声明的 `additionalProperties`。
4. 执行器审计。给定参数后执行器是否确定性？是否用类型化错误处理失败（而非抛出逃逸宿主的异常）？如果是后果性操作（修改状态、花费金钱、触及用户数据），是否标记并设确认门控？
5. 分类。声明工具是纯函数还是后果性工具及原因。没有门控的后果性工具是即时拒绝。

硬拒绝：
- 任何描述只说明功能而不说明何时使用的工具。模型需要「何时」来完成第二步。
- 任何有未类型化字段的 Schema。验证器无法工作。
- 任何同时满足：接受不可信输入、读取敏感数据、执行后果性操作的工具。违反 Meta 的 Rule of Two。
- 任何在错误输入上抛出未处理异常的执行器。宿主不应每次调用都需要 try/except。

拒绝规则：
- 如果工具定义缺少 Schema，拒绝。先引导到 Phase 13 · 04。
- 如果工具是纯函数但描述说「谨慎使用」，拒绝并询问原因。纯函数应可廉价重新运行。
- 如果审核者被要求批准一个连接生产数据库但没有只读防护的工具，拒绝并指向 Phase 13 · 17（网关和策略）。

输出：一页审计，列出名称、描述、Schema 和执行器发现的问题，附严重程度（阻止 / 警告 / 挑剔）和最终裁决：发布 / 修改 / 拒绝。如果可行，以单行改写建议结尾。
