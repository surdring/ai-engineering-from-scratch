# 流水线预算规划器

将延迟/吞吐量目标转化为逐阶段预算，让每个团队成员都知道自己的工程目标数据。

## 使用场景

- 在构建新的视觉服务之前，为每个阶段设定预期
- 首次基准测试后，查看哪个阶段最远偏离预算
- 当 SLA 变更需要重新协商预算时

## 输入

- `p95_latency_target_ms`：每个请求的预算
- `target_qps`：每个副本的吞吐量
- `stages`：`{ name: str, current_ms: float }` 列表

## 分配规则

如未提供当前测量值，七个标准阶段的默认分配如下：

| 阶段 | 占比 |
|------|------|
| 解码 + 预处理 | 15% |
| 检测器前向传播 | 55% |
| 后处理检测（NMS，钳制） | 5% |
| 裁剪 + 缩放（用于分类器） | 5% |
| 分类器前向传播 | 15% |
| Schema 验证 | <1% |
| 响应序列化 | 4% |

在 GPU 受限的流水线（云）中，检测器占比通常升至 70%。在 CPU 上，预处理和分类器批量处理消耗更多。

## 报告

```
[预算计划]
  p95 目标:   <ms>
  吞吐量:     <每副本 qps>

| 阶段                | 目标_ms | 当前_ms | 余量 | 状态 |
|---------------------|---------|---------|------|------|
| 解码+预处理         | ...     | ...     | ...  | ok|X |
| 检测器              | ...     | ...     | ...  | ok|X |
| ...                 | ...     | ...     | ...  |      |

[瓶颈]
  阶段:  <名称>
  超出:  <超出预算毫秒数>
  杠杆:  <具体行动>

[可用的杠杆]
  解码+预处理:     Pillow-SIMD, libjpeg-turbo, 通过 NVJPEG 在 GPU 上解码
  检测器:          更小的骨干网络, 更低的输入分辨率, INT8, TensorRT
  后处理:          GPU 端 NMS（torchvision.ops）, 融合掩码
  裁剪+缩放:       GPU 裁剪（grid_sample）, 批量插值
  分类器:          更小的骨干网络, INT8, 热缓存, 批量
  schema:          在热路径中跳过验证，仅在边界验证
  响应:            orjson, stream protobuf
```

## 规则

- 绝不建议从生产路径中移除 schema 验证；建议将其移到边界
- 如果预处理超出预算，始终先尝试 Pillow-SIMD 或 NVJPEG，再更换模型
- 如果检测器超出目标 30% 以上，直接切换模型而非优化当前模型
- 当 current_ms > 1.1 * target_ms 时标记状态为 `X`；在预算 10% 以内标记 `ok`
