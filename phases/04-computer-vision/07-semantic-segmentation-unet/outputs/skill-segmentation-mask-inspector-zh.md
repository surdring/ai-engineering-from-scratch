# 分割掩码检查器

一个诊断工具，用于弥合「损失下降了」和「掩码看起来确实是对的」之间的差距。

## 使用场景

- 训练运行结束后，mIoU 看起来不错但视觉检查显示并非如此时
- 部署前：检查预测与真实标签之间的类别平衡
- 当大型物体的逐类别 IoU 很高但小型物体很低时
- 调试不会在 IoU 中体现的边界伪影（因为像素数量少）

## 输入

- `preds`：预测类别 ID 的 (N, H, W) 张量
- `targets`：真实类别 ID 的 (N, H, W) 张量
- `num_classes`：整数
- 可选 `class_names`：C 个字符串的列表

## 步骤

1. **类别像素直方图。**计算 `preds` 和 `targets` 中每个类别的像素百分比。标记任何 `|pred% - gt%| / max(gt%, 1e-6) > 0.30`（相对偏差超过 30%）的类别。对于在真实标签中不存在的类别（`gt% == 0`），直接标记预测占比超过 `0.3` 的情况

2. **逐类别 IoU** 和 **逐类别边界 F1**。边界 F1 通过将每个掩码膨胀 3 像素、求交集并评分来计算。IoU > 0.7 但边界 F1 < 0.5 的类别存在边缘模糊问题

3. **小目标召回率。**将每个真实标签的连通分量（Connected Component）按尺寸桶分类（微小 < 100 px，小 < 1000 px，中 < 10000 px，大 >= 10000 px）。按桶和类别报告召回率。小目标召回率低于 0.3 而大目标召回率高于 0.9 表明存在分辨率/感受野问题

4. **混淆配对。**对每个类别，找出其最常混淆的类别（在其真实掩码内最常见的错误预测类别）。报告前三对

5. **饱和检查（需要 `probs` 或 `logits`，而非仅 `preds`）。**如果调用者传入原始逐像素概率分布 `probs: (N, C, H, W)`，计算每个类别中 `probs.max(dim=1) > 0.99` 的像素比例。高饱和度（某类别像素的 >0.9）表明过度自信——可考虑使用标签平滑或校准。当仅传入 argmax 后的 `preds` 时，跳过此步骤并在报告中记录

## 报告格式

```
[mask-inspector]
  类别数: C

[类别分布]
  名称       真实%    预测%   差值
  ...

[指标]
  类别        IoU     边界F1   召回_微小   召回_小   召回_中   召回_大
  ...

[混淆配对]
  类别 A 与 类别 B 混淆: <N> 像素（最常见）
  类别 B 与 类别 A 混淆: <N> 像素
  ...

[结论]
  最有影响的问题: <一句话>
```

## 规则

- 按真实像素占比降序排列类别行，让出现最频繁的类别排在最前面
- 将 IoU < 0.4 或边界 F1 < 0.3 的类别标记为 `严重`
- 当小目标召回率是主要失败来源时，建议：使用更高分辨率训练、减小最后编码器阶段的步幅，或使用特征金字塔解码器
- 当边界 F1 是主要失败来源时，建议：使用边界感知损失（Lovasz 或 BoundaryLoss）、水平翻转的 TTA 以及无步幅的解码器
- 绝不将类别索引作为唯一标识符；如果提供了 `class_names`，在每一行中都使用它
