# 推测式解码选择器

帮助工程师在普通推测式解码、Medusa、EAGLE 或 lookahead 解码之间选择，并针对特定工作负载调优 `N`（草稿长度）。

## 需收集的输入

1. **验证器模型**：哪个 LLM 产生最终输出。模型大小很重要（草稿成本必须 < 验证器成本才能加速）。
2. **工作负载类型**：代码、聊天、结构化输出、摘要。决定接受率。
3. **采样策略**：贪心、低温度、高温度、束搜索。高温度采样会降低接受率。
4. **硬件目标**：内存预算决定是否能容纳单独的草稿模型。
5. **工程预算**：Medusa 和 EAGLE 需要微调；普通和 lookahead 不需要。
6. **延迟目标**：交互式聊天（< 500ms TTFT，< 50ms 每 token）vs 批处理（吞吐量优先）。

## 决策规则

- **快速起步，无训练**：同家族的 1B–3B 模型做普通草稿。典型 2× 加速。
- **可以微调**：EAGLE-2 或 EAGLE-3 使用验证器的隐藏状态。典型 3–4× 加速。
- **可以微调但不能运行两个模型**：Medusa（验证器上附加额外头）。典型 2–3× 加速。
- **无训练预算，无草稿模型**：lookahead 解码。1.3–1.6× 加速。
- **批处理密集型服务**：连续批处理更重要；推测式加速随批次增大而减弱，因为验证器已经饱和。
- **高温度或随机采样**：接受率急剧下降。考虑降低 N（2-3）或禁用。
- **结构化输出（JSON、代码）**：接受率高。将 N 推至 7+ 以获最大加速。

## 调优

- **N（草稿长度）**：从 5 开始。测量接受率。如果 α > 0.9，推到 7。如果 α < 0.6，降至 3。
- **草稿温度**：与验证器温度匹配。不匹配的草稿采样会降低 α。
- **树深度（EAGLE-2 / Medusa）**：3-5 个分支；更宽的树仅在 α > 0.8 时有益。
- **草稿模型大小**：满足 α > 0.7 的最小模型。70B 验证器用 1B 草稿模型是典型的；不要低于验证器的分词器/嵌入兼容性要求。
