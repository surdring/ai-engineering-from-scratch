给定模型大小、SLA（TTFT P99）、流量形态（稳定 vs 突发）和预算姿态，生成冷启动缓解计划。

生成：

1. **冷启动预算。** 分解原始冷启动路径（节点供应、镜像拉取、权重到 HBM、引擎初始化、首次前向）。对所述模型大小使用 2026 年标称秒数。
2. **层级选择。** 选择将总时间降到 SLA 以下的最少层级数：预种子镜像（L1）、模型流式加载器（L2）、GPU 快照（L3）、暖池（L4）、分层加载（L5）。论证每层针对的特定阶段。
3. **暖池规模。** 说明主路径的 `min_workers`。如果 SLA 是 TTFT P99 < 60s 在 70B+ 模型上，无论成本如何都强制暖池。
4. **成本估算。** 所选暖池的月度 GPU 成本和每天预期的冷启动次数。
5. **尾延迟策略。** 新副本上的第一个用户会发生什么——他们被排队到暖副本，还是承受冷启动代价？指定具体策略（例如「在 10s 内将第一个请求路由到任何暖副本；回退到冷启动」）。
6. **失败模式。** 如果暖副本在会话中途死亡会发生什么。恢复是自动的（实时迁移），还是在下一次请求时进行冷启动？

硬性拒绝：
- 提议「加暖池」而不计算月度成本。
- 声称缓解措施而不指定它针对的特定阶段（例如「使用 Bottlerocket」而不说明它消除了 180s 镜像拉取）。
- 忽略 GPU 快照的每 GPU 拓扑约束——如果平台迁移 SKU，快照无效。

拒绝规则：
- 如果 SLA 是 TTFT P99 < 5s 在全新的 70B 冷启动上且没有暖池，拒绝——在 2026 年基础设施速度下数学上不可能。
- 如果预算禁止暖池但 SLA 要求亚 30s 冷启动，指定平台特定修复（Modal GPU 快照、Baseten 预热）并拒绝在没有它的情况下在其他平台上承诺 SLA。
- 如果运维人员要求在突发流量和 70B 模型的情况下缩容到零，拒绝承诺 SLA——没有快照或暖池时数学不成立。

输出：一页计划，列出阶段、层级、`min_workers`、月度成本、尾延迟策略、失败模式。以要告警的单一指标结尾：过去滚动一小时的 P99 冷启动持续时间。
