# LLM FinOps — 单位经济学与多租户归因

> 传统的 FinOps 在 LLM 支出上失效。成本是 Token 交易，而非资源运行时间。标签无法映射——API 调用是交易，而非资产。工程决策（提示词设计、上下文窗口、输出长度）是财务决策。2026 年的操作手册有三个归因维度，需在第一天就进行插桩：按用户（`user_id`）用于座位定价和扩展，按任务（`task_id` + `route`）用于产品功能表面成本和优先级，按租户（`tenant_id`）用于单位经济学和续约。四个 Token 层级——提示词、工具、记忆、响应——合并在一个桶里隐藏了支出。多租户产品的执行层次：每个租户的速率限制（预期峰值的 2-3 倍，明确的 429 + retry-after）；每日支出上限（合同上限的 1.5-3 倍；触发收紧速率 + 告警）；支出 z-score > 4 时的终止开关（自动暂停 + 呼叫值班）。归因模式：标记并聚合、遥测链接器（追踪-ID → 计费；最高准确率）、采样和外推、基于模型的分配、事件溯源、实时流式。单位指标：每次解决查询的成本、每次生成产出物的成本——而非 $/M token。追溯性标记总是遗漏；在请求创建时进行插桩。

**类型：** 学习
**语言：** Python（标准库，带终止开关的玩具级成本归因模拟器）
**前置知识：** 第 17 阶段 · 13（可观测性），第 17 阶段 · 14（缓存）
**时间：** 约 60 分钟

## 学习目标

- 解释为什么传统 FinOps（标签 + 层级）在 LLM 支出上失效，并指出三个新的归因维度。
- 列举四个 Token 层级（提示词、工具、记忆、响应）以及为什么单桶计费隐藏了成本。
- 为多租户产品设计执行层次（速率限制 → 支出上限 → 终止开关）。
- 选择单位指标（每次解决查询 / 产出物成本）而非 $/M token。

## 问题

你的账单显示 $40,000。你无从知晓：
- 哪个租户花了这笔钱。
- 哪个产品功能驱动了它。
- 是否有任何个人用户是滥用的。
- 提示词膨胀、工具调用还是记忆放大是罪魁祸首。

在供应商侧进行标记并聚合对云资源（EC2、S3）有效，标签传播到账单项。LLM API 调用不会自动标记——你必须在调用现场加盖用户/任务/租户标签并贯穿始终。追溯性归因总是遗漏边界情况。

## 概念

### 三个归因维度

**按用户**（`user_id`）：谁花了多少。驱动座位定价、扩展对话、识别深度用户。

**按任务**（`task_id` + `route`）：哪个产品功能表面花了多少。驱动功能优先级、"砍掉昂贵功能"的决定。

**按租户**（`tenant_id`）：哪个客户是盈利的。驱动单位经济学、续约定价、层级阈值。

在第一天就在调用现场对三者进行插桩。追溯性永远更差。

### 四个 Token 层级

| 层级 | 示例 | 典型占比 |
|-------|---------|---------------------|
| 提示词 | 系统 + 用户输入 | 40-60% |
| 工具 | 反馈的工具调用结果 | 20-40%（Agent 工作负载） |
| 记忆 | 先前对话 / 检索到的文档 | 10-30% |
| 响应 | 模型输出 | 10-30% |

将四者合并在一起使优化失去方向。在你的归因模式中将它们区分出来。

### 执行层次

1. **速率限制** per 租户。预期峰值的 2-3 倍。返回 429 带 `Retry-After`。租户看到摩擦；无意外账单。

2. **每日支出上限** per 租户。合同上限的 1.5-3 倍。触发：收紧速率限制 + 通知客户成功。

3. **终止开关** 在支出 z-score > 4 时（相对租户基线）。自动暂停租户；呼叫值班；升级至运维 + CS。

### 归因模式

- **标记并聚合**：加盖元数据头；后续聚合。简单；粗糙。
- **遥测链接器**：通过追踪 ID 将追踪链接到计费。最高准确率。成熟团队的做法。
- **采样 + 外推**：采样 5-10%，乘以倍数。对粗略支出成本有效；遗漏长尾。
- **基于模型的分配**：回归推断成本驱动因素。用于无标签的遗留数据。
- **事件溯源**：成本作为流（Kafka / Kinesis）中的事件。实时。
- **实时流式**：仪表板亚秒级更新。

### 单位成本的指标

$/M token 是供应商语言。产品指标：

- 每次解决支持工单的成本。
- 每次生成文章的成本。
- 每次成功 Agent 任务的成本。
- 每次用户会话分钟的成本。

将成本与产品成果绑定。否则优化无锚点。

### 成本归因追踪形态

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用时发送。存储在数据湖中。按维度聚合。存在于第 17 阶段 · 13 的可观测性栈中。

### 复合节省栈

叠加：缓存 + 批处理 + 路由 + 网关。四者全用：
- 缓存 L2（第 17 阶段 · 14）：输入约便宜 10 倍。
- 批处理（第 17 阶段 · 15）：5 折。
- 路由到便宜模型（第 17 阶段 · 16）：60% 成本削减。
- 网关效率（第 17 阶段 · 19）：冗余 + 重试。

最佳情况叠加：约朴素基线的 5-10%。大多数团队使用了 2-3 个杠杆；很少叠满全部四个。

### 应记住的数字

- 归因维度：按用户、按任务、按租户。
- 四个 Token 层级：提示词、工具、记忆、响应。
- 终止开关：支出 z-score > 4。
- 单位指标：每次解决查询的成本，而非 $/M token。
- 叠加优化：可能达到基线的 ~5-10%。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Per-user attribution | "用户级成本" | 每次调用加盖 `user_id` |
| Per-task attribution | "功能成本" | `task_id` + `route` 识别产品功能表面 |
| Per-tenant attribution | "客户成本" | `tenant_id`；驱动单位经济学 |
| Four token layers | "成本层级" | 提示词 + 工具 + 记忆 + 响应 |
| Rate limit | "429 守卫" | 网关强制执行的每租户上限 |
| Daily spend cap | "每日上限" | 带告警的租户范围预算 |
| Kill switch | "自动暂停" | 支出 z-score > 4 触发自动暂停 |
| Cost per resolved | "产品单位指标" | 绑定产品成果的成本，非 token |
| Telemetry joiner | "追踪到计费" | 最高准确率的归因模式 |
| Stacked optimization | "缓存+批处理+路由+网关" | 复合节省至 ~5-10% 基线 |
