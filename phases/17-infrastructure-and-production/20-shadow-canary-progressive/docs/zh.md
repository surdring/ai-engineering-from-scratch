# 影子流量、金丝雀发布与渐进式 LLM 部署

> LLM 发布结合了软件部署中最难的部分：没有单元测试、扩散式的失败模式、延迟的信号。顺序是（1）影子模式（Shadow Mode）——将生产请求复制到候选模型、记录、对比，零用户影响；可捕获明显的分布问题但不是质量保证；（2）金丝雀发布（Canary Rollout）——渐进式流量转移 10% → 25% → 50% → 75% → 100%，每步设置关卡（Gate）；跟踪延迟百分位、成本/请求、错误/拒绝率、输出长度分布、用户反馈率；（3）A/B 测试用于确认稳定性后的不同备选方案。非确定性（Non-Determinism）不可消除——相同的输入在不同运行间最多有 15% 的准确率差异，原因包括 GPU FP 非结合律加上批次大小方差。成本是一个变量，而不是常数——好 20% 的模型每次调用可能贵 3 倍。回滚速度是决定性的：如果回滚需要重新部署，那你就太慢了。策略存在于配置/开关中；模型存在于带固定摘要（Digest）的注册表中；回滚 = 翻转策略 + 恢复阈值 + 钉住（Pin）旧模型，几秒内完成。

**类型：** 学习
**语言：** Python（标准库，玩具级金丝雀渐进模拟器）
**前置知识：** 第 17 阶段 · 13（可观测性），第 17 阶段 · 21（A/B 测试）
**时间：** 约 60 分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（实时流量渐进）和 A/B 测试（确认稳定性后的对比）。
- 列举五个 LLM 专属的金丝雀指标（延迟、成本/请求、错误/拒绝、输出长度分布、用户反馈）。
- 解释为什么 LLM 的非确定性（高达 15%）改变了发布中"稳定"的含义。
- 设计一个只需几秒（策略翻转）而非几小时（重新部署）的回滚路径。

## 问题

你发布一个新模型。离线评估显示 3% 的准确率提升。你在生产环境中切换它。24 小时内，成本上涨 40%，用户踩赞率上涨 8%，三个客户工单报告"奇怪的回答"。你回滚。重新部署需要 3 小时。你的周末被毁了。

这一切本来都可以避免。影子模式会在任何用户看到之前就捕捉到 40% 的成本飙升。金丝雀会在踩赞率变化的 10% 处停止。策略标志回滚只需 30 秒。这门纪律填补了"离线评估看起来不错"与"真实用户满意"之间的空白。

## 概念

### 影子模式

候选模型接收与生产相同的请求；输出被记录，不返回给用户。零用户影响。记录：

- 输出内容（与生产对比的差异）。
- Token 数量（成本增量）。
- 延迟。
- 拒绝和错误。

可捕捉：成本爆炸、长度退化、明显的拒绝变化、硬错误。不可捕捉：用户可感知的质量增量。影子是烟雾测试，不是质量测试。

### 金丝雀发布

渐进式流量转移带关卡。典型进程：1% → 10% → 25% → 50% → 75% → 100%。在每步在 5 个指标上设关卡：

1. **延迟百分位** — P50、P95、P99。违反：金丝雀的 P99 > 基线的 1.5 倍。
2. **每请求成本** — 混合 $。违反：高于基线 >20%。
3. **错误 / 拒绝率** — 5xx 加显式拒绝。违反：基线的 2 倍。
4. **输出长度分布** — 均值 + P99。违反：分布漂移。
5. **用户反馈率** — 踩赞 / 工单提交。违反：基线的 1.5 倍。

### 非确定性是新的方差

相同的输入产生不完全相同的输出。原因：

- GPU FP 非结合律（浮点归约顺序因批次而异）。
- 批次大小方差（相同提示词在 128 的批次 vs 16 的批次中）。
- 采样（温度 > 0）。

测量：相同的评估集在不同运行间高达 15% 的准确率差异。发布中的"稳定"意味着指标在预期方差范围内，而非与基线完全相同。将关卡设到噪声地板之上。

### 成本是一个变量

好 20% 的模型每次调用可能贵 3 倍。每请求成本是五个关卡之一。交付一个"更好"但打破单位经济学的模型是回滚案例。

### 回滚是武器

- 策略标志（特性开关系统）：在配置中翻转百分比；只需几秒。
- 模型钉住（Model Pinning）（注册摘要）：钉住的模型不会自动升级。
- 回滚 = 恢复标志 + 将钉住的摘要设置为之前版本。几秒，而非几小时。

如果你的技术栈需要重新部署来回滚，在发布之前修复这个问题。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。集成 Istio/Linkerd 加权路由。

**Istio weighted routing** — 服务网格层的流量拆分。

**KServe / Seldon Core** — 模型推理的内置金丝雀发布。

**Feature flags** — LaunchDarkly、Flagsmith、Unleash。策略层切换，无需重新部署。

### 指标节奏

金丝雀关卡视流量量每 5-15 分钟检查一次。1% 流量在 10 req/min 下给出 50-150 个数据点/窗口——对延迟足够但对用户反馈噪声较大。10% 提供约 10 倍数据点。进程应在每一步停留足够长时间积累足够的样本。

### A/B 步骤是可选的

如果新模型与旧模型有显著差异（不同行为、不同成本曲线、不同语气），在金丝雀通过后进行 50% 的 A/B 测试。如果只是一个改进版本，当金丝雀关卡通过后跳过直接到 100%。

### 应记住的数字

- 金丝雀进程：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性天花板：相同输入高达 15% 的运行间方差。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本关卡：高于基线 >20% 即违反。
- 回滚：秒级，非小时级。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Shadow mode | "复制到新版" | 零影响发送到候选模型，仅用于日志记录 |
| Canary | "渐进式流量" | 带关卡的逐步用户暴露式发布 |
| Gates | "发布检查" | 阻止进度推进的指标阈值 |
| Non-determinism | "LLM 方差" | 不可消除的运行间差异 |
| Policy flag | "标志翻转回滚" | 配置级回滚，秒级非小时级 |
| Model pin | "注册摘要" | 对模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进式" | Kubernetes 原生的金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 带金丝雀发布原语的模型推理 |
| Istio weighted | "网格流量拆分" | 服务网格流量拆分器 |
