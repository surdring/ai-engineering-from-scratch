# SGLang 与前缀密集型工作负载的 RadixAttention

> SGLang 将 KV 缓存视为一等、可复用的资源，存储在基数树（Radix Tree）中。vLLM 按 FCFS（先到先服务）调度请求，而 SGLang 的缓存感知调度器优先处理共享更长前缀的请求——实际上是一种深度优先基数遍历，使热分支驻留在 HBM 中。在 ShareGPT 风格 1K 提示的 Llama 3.1 8B 上，SGLang 达到约 16,200 tok/s，vLLM 约 12,500，约 29% 的优势。在前缀密集型 RAG 工作负载上优势达到 6.4 倍。在语音克隆形态的工作负载上缓存命中率超过 86%。2026 年部署在 400,000+ GPU 上，覆盖 xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS。陷阱是当前缀顺序不一致时 6.4 倍的数字会蒸发——顺序是工程师的杠杆。

**类型：** 学习
**语言：** Python（标准库，玩具级基数树缓存 + 缓存感知调度器）
**前置条件：** Phase 17 · 04（vLLM 推理内部）、Phase 14（Agent RAG）
**时间：** ~75 分钟

## 学习目标

- 画出 RadixAttention：前缀如何存储在基数树中，以及 KV 块如何在属于同一分支的序列之间共享。
- 解释缓存感知调度，以及为什么 FCFS 对前缀密集型流量是错误的。
- 给定前缀缓存命中率和提示长度分布，计算工作负载的预期加速比。
- 说出使 6.4 倍数字成为现实而非失去优势的提示排序纪律。

## 问题

经典推理将每个请求的提示视为不透明的。即使 5,000 个 RAG 请求都以相同的 2,000 Token 系统提示加上相同的检索前导开始，vLLM 也会对该 2,000 Token 前缀进行 5,000 次预填充。GPU 一遍又一遍地做相同的工作。

观察：Agent 和 RAG 工作负载中的提示几乎总是共享长前缀。系统提示、工具模式、少样本示例、检索头、对话历史——全部跨请求重复。如果你存储该前缀的 KV 缓存一次并复用它，你就不需要再次预填充。

RadixAttention 正是这样做的。Token 被索引在基数树中；每个节点拥有其从根路径上的 Token 序列的 KV 块。新请求遍历树：任何 Token 匹配的节点复用该节点的 KV 块。预填充成本与"新"后缀成正比，而非全量提示。

挑战是调度。如果两个请求共享 2,000 Token 前缀，第三个只共享同一前缀的 200 Token，你希望将两个长共享请求一起服务，使长前缀留在 HBM 中。FCFS 做相反的事——它服务先到达的人，可能在下一个长前缀请求到达前驱逐热分支。

## 概念

### 基数树作为 KV 索引

基数树（紧凑 Trie）存储 Token 序列。每个节点拥有一个 Token 范围和为该范围计算的 KV 块。子节点扩展一个或多个 Token 的序列。

```
root
 |- "You are a helpful assistant..."  (2,000 tokens, 124 KV 块)
      |- "Context: <doc A>..."        (500 tokens, 31 块)
           |- "Question: Alice..."    (80 tokens, 5 块)
           |- "Question: Bob..."      (95 tokens, 6 块)
      |- "Context: <doc B>..."        (520 tokens, 33 块)
```

一个新请求以系统提示 + "Context: <doc A>" + "Question: Carol" 到达。调度器遍历：系统前缀匹配（124 块复用），doc-A 分支匹配（31 块复用），然后仅为 "Question: Carol" 分配新块（4 块）。预填充成本：4 块新 Token。没有树：160 块。预填充上约 40 倍节省。

### 缓存感知调度

如果缓存搅动（Churn），基于基数树的复用是无意义的。两个关键策略：

1. **深度优先分发**。从队列中选择下一个请求时，优先选择根植于与当前运行集相同分支的请求。这保持热分支被钉住。
2. **分支级别 LRU，而非块级别**。驱逐整个分支（从最短使用的叶子开始），而非单个块，使缓存形状匹配基数形状。

FCFS 违反两者。共享 2,000 Token 的请求排在共享 50 Token 的请求后面，然后 2,000 Token 分支被驱逐以接纳 50 Token 的分支。

### 你应该记住的基准数字

- Llama 3.1 8B，H100，ShareGPT 1K 提示：SGLang ~16,200 tok/s vs vLLM ~12,500（~29% 优势）。
- 前缀密集型 RAG（相同系统 + 相同文档，不同问题）：SGLang 上高达 6.4 倍。
- 语音克隆工作负载：86.4% 前缀缓存命中率。
- SGLang 客户的生产命中率：50-99%，取决于提示纪律。
- 2026 年部署在 400,000+ GPU 上。

### 排序陷阱

6.4 倍数字依赖一致的提示模板排序。如果你的客户端在某些请求中构建提示为 `[system, tools, context, history, question]`，在另一些中为 `[system, context, tools, history, question]`，树无法找到共享前缀。对人类来说看起来像共享前缀的，对基数树是两个不同的序列。

工程师的杠杆：你的提示模板是一个缓存键。固定顺序。将所有不可变的（系统、工具、模式）放在最前面。将检索上下文放在其次。将用户问题放在最后。不要将动态内容交错插入前缀中。

真实案例研究：将动态内容移出可缓存前缀，一个部署在一次变更中从 7% 命中率提升到 74%。

### RadixAttention 胜出和失败的地方

胜出：
- RAG（相同检索前导，不同问题）。
- Agent（相同工具模式，不同查询）。
- 带长系统提示的聊天。
- 带重复前导的语音/视觉工作负载。

失败（回到 vLLM 级吞吐量）：
- 具有唯一提示的单次生成（代码补全、无系统提示的开放式聊天）。
- 每个请求将唯一内容交错插入前缀的动态提示。

### 为什么这是一个调度器问题，而非仅仅是内核问题

你可以将 KV 复用实现为内核技巧。SGLang 的洞察是复用仅在调度器保持热分支驻留时才有回报。naive 的"如果可用则复用"策略会在混合负载下搅动缓存。基数树索引调度器是将内核技巧转化为 29% 生产优势的关键。

### 与 vLLM 的交互

这两个系统不是严格竞争者。2026 年 vLLM 增加了前缀缓存（`--enable-prefix-caching`）和缓存感知路由器（vLLM Router in Rust）。差距缩小但未完全消失——SGLang 的整个栈是基数优先的；vLLM 是嫁接上去的。对于前缀复用占主导的工作负载，SGLang 仍然是默认选择。对于没有强前缀模式的通用推理，vLLM 仍然相当或更好。

## 使用它

`code/main.py` 实现一个玩具基数树 KV 缓存，加上一个具有两种策略的调度器：FCFS 和缓存感知。通过两者运行相同工作负载，报告前缀缓存命中率和吞吐量增量。然后运行"打乱排序"工作负载以展示 6.4 倍的崩溃。

## 交付它

本课产出 `outputs/skill-radix-scheduler-advisor.md`。给定工作负载描述（提示模板形状、检索模式、并发租户数），它产生一个提示排序处方和 SGLang 采纳的 go/no-go。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| RadixAttention | "SGLang 的东西" | KV 缓存索引为基数树，使共享前缀复用块 |
| 基数树 | "紧凑 Trie" | 每个节点拥有一个 Token 范围及其 KV 块的树 |
| 缓存感知调度器 | "热分支优先" | 优先选择共享驻留分支请求的调度器 |
| 前缀缓存命中率 | "你的提示有多少是免费的" | 从复用 KV 块中服务的提示 Token 比例 |
| FCFS | "先到先服务" | 破坏前缀局部性的默认调度 |
| 分支级 LRU | "驱逐叶子" | 与基数形状匹配的驱逐策略 |
| 提示模板排序 | "缓存键" | 提示的组件顺序决定了树能共享什么 |
| 系统提示钉住 | "驻留前缀" | 保持不可变系统部分钉住以避免驱逐颠簸 |
