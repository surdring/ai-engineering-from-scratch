给定技术栈（Kubernetes / VM / 托管）、SLI/SLO 成熟度、可观测性质量和团队值班成熟度，生成混沌计划。

生成：

1. **前提检查。** 验证 SLI/SLO 已定义、可观测性已接通、回滚已自动化、操作手册已结构化、值班轮转已建立。如果任何一项缺失，拒绝在生产中运行混沌。
2. **四个平面。** 为每个平面（控制、目标、安全、可观测性）指定工具。可观测性指向阶段 17 · 13。
3. **三个初始实验。** 从销毁 Pod 开始。然后是提供商 429。然后是内存过载。每个附带爆炸半径上限、持续时间、成功标准。
4. **安全门槛。** 燃烧率（>2x 预期）、爆炸半径（< 舰队 30%）、追踪 ID 标记、抑制窗口。
5. **频率。** 每周小型金丝雀。每月游戏日（跨团队）。每季度韧性审计。
6. **工具。** LitmusChaos（OSS，CNCF 毕业）、Chaos Mesh（OSS，CNCF 沙箱）、Harness Chaos（商业 AI 辅助）、AWS FIS / Azure Chaos Studio（托管云原生）。

硬性拒绝：
- 在没有五个前提条件的情况下在生产中运行混沌。拒绝——会变成真实事件。
- 没有爆炸半径上限的实验。拒绝。
- 没有追踪 ID 标记的实验。拒绝——不可能去重告警。

拒绝规则：
- 如果团队从未在预发环境中成功运行过一次实验，拒绝生产混沌，直到在预发中绿色通过一次。
- 如果事件量已经很高（>2/周），拒绝增加混沌——先稳定化。
- 如果团队没有 SLO，在任何实验之前要求定义 SLO。

输出：一页计划，包含前提检查、四平面工具、三个初始实验、安全门槛、频率。以每季度依赖图更新承诺结尾。
