给定一个目标运行时、一个主模型和一个（可能更强的）休眠时模型，生成一个具有明确块类型和异步整合的三层记忆系统。

生成内容：

1. `Block` 类型，包含 `label`、`value`、`limit`、`description`、`version`、`history`。每次写入递增版本并记录旧值。暴露 `near_limit(threshold=0.8)`。
2. 一个 `BlockStore`，至少包含三个默认块：`human`（关于用户的事实）、`persona`（智能体自我概念）和 `task`（当前范围）。允许用户自定义块。
3. 一个 `Recall` 存储——按会话分页的轮次日志。每轮自动写入。尾部在达到容量时驱逐但保持可检索。
4. 一个 `Archival` 存储——至少两种后端（向量、KV）。插入返回记录 id。在矛盾时使无效而非删除。
5. 一个 `PrimaryAgent`，处理轮次并仅发出原始写入。关键路径上不进行摘要。
6. 一个 `SleepTimeAgent`，在轮次之间运行：对超过阈值的块进行摘要、使矛盾的归档记录无效、将 `learned_context` 写入共享块。

硬性拒绝：

- 在面向用户的轮次期间同步运行任何记忆操作，除了直接查找。摘要、整合、失效属于休眠时步骤。
- 在矛盾时删除归档记录。使无效以保持历史可审计。
- 在没有审核步骤的情况下写入 Persona 或 Safety 块。这些块全局性地影响行为；静默写入会掩盖错误。

拒绝规则：

- 如果运行时无法跨会话持久化块，拒绝交付描述为具有「记忆」的产品。降级声明。
- 如果休眠时智能体没有追踪输出，拒绝。静默整合是调试的死区。
- 如果用户要求「不进行失效，始终信任最新写入」，对任何历史声明重要的领域（合规、医疗、法律）拒绝。

输出：每个组件一个文件，外加一个 `README.md`，命名默认块、休眠时节奏和矛盾解决策略。结尾的「下一步阅读」指向第 09 课如果智能体需要基于图的记忆推理，或第 23 课如果产品需要记忆操作上的 OTel span。
