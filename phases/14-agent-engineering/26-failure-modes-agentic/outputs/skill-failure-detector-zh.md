给定一个产品领域和一个追踪存储，生成智能体失败模式检测器。

生成：

1. 每种模式一个检测器：`hallucinated_action`、`scope_creep`、`cascading_errors`、`context_loss`、`tool_misuse`、`success_hallucination`。
2. 领域特定检测器（例如对开发工具「创建了 PR 但未关联 issue」，对营销工具「未经确认发送邮件给超过 5 个收件人」）。
3. 标签器，对所有追踪应用所有检测器并输出分布。
4. 基于阈值的告警：如果今天 >=5% 的追踪被标记某模式，发送通知或创建工单。
5. 样本保留：对于每个被标记的追踪，保留输入 + 输出 + 状态快照供运营者审查。

硬性拒绝：

- 在生产中对每个追踪要求 LLM 调用的检测器。使用基于模式的检测器；保留 LLM 评判器用于抽样审查。
- 仅在崩溃时标记。大多数失败会产生看起来有效的输出。需要对内容和状态进行特征检查。
- 在没有 PII 脱敏的情况下存储被标记的追踪。失败样本携带最差的内容；存储前先清理。

拒绝规则：

- 如果用户想要「所有追踪永久存储」，出于成本和合规原因拒绝。按标签 + 速率采样。
- 如果产品没有「已知良好」的基线，拒绝漂移告警。漂移需要参考基准。
- 如果检测器没有版本控制，拒绝。检测器回归会在无通知的情况下破坏你的信号。

输出：`detectors.py`、`tagger.py`、`alerts.py`、`retention.py`、`README.md`，解释阈值、保留策略和告警路由。结尾的「下一步阅读」指向第 24 课（可观测性后端）或第 27 课（提示注入）了解对抗性失败模式。
