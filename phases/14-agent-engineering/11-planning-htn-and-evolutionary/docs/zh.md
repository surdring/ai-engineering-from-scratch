# HTN 规划与进化搜索

> 符号规划处理那些计划必须是可证明正确的场景。进化代码搜索处理那些适应度函数是机器可检查的场景。ChatHTN（2025）和 AlphaEvolve（2025）展示了当每个与 LLM 配对时所能解锁的能力。

**类型：** 构建
**语言：** Python（标准库）
**前置要求：** Phase 14 · 02（ReWOO 和计划-执行）
**时间：** ~75 分钟

## 学习目标

- 解释层次任务网络（HTN）：任务、方法、操作符、前提条件、效果。
- 描述 ChatHTN 的混合循环 —— 带 LLM 回退分解的符号搜索。
- 解释 AlphaEvolve 的进化循环以及为什么它只在有程序化评估器时才有效。
- 在标准库中实现一个玩具 HTN 规划器加一个玩具进化搜索。

## 问题

ReWOO（第 02 课）、计划-执行和 ReAct 涵盖了大多数 Agent 规划。它们无法很好覆盖的两种情况：

1. **需要可证明正确性的计划。** 排程、飞行路径规划、合规工作流 —— 计划必须通过构造是可证明正确的。一个流利的 LLM 计划只要偶尔幻觉一个步骤就是不可接受的。
2. **具有机器可检查适应度函数的优化。** 矩阵乘法、调度启发式、编译器遍历 —— 目标不是"一个正确的计划"而是"最好的计划"。

HTN 规划和 AlphaEvolve 解决了这两个不同的问题。两者都将 LLM 用作放大器，而非替代品。

## 概念

### 层次任务网络

HTN 是：

- **任务（Tasks）** —— 复合任务（待分解）和原语任务（直接可执行）。
- **方法（Methods）** —— 将复合任务分解为子任务的方式，带有前提条件。
- **操作符（Operators）** —— 带前提条件和效果的原语动作。
- **状态（State）** —— 一组事实。

规划：给定一个目标任务和一个初始状态，找到分解为串行前提条件得到满足的原语操作符。

HTN 比 LLM 更古老，仍然是可证明正确计划的参考标准。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交织：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："你会在状态 `s` 中如何分解 `task`？"
3. 将 LLM 响应转换为候选子任务。
4. 根据操作符 schema 验证；拒绝无效的分解。
5. 递归。

论文的核心声明：每个生成的计划都是可证明正确的，因为 LLM 的建议仅作为候选分解进入，从不作为直接的计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续工作）添加了一个学习器，通过回归泛化 LLM 产生的分解 —— 将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是另一种东西：由 Gemini 2.0 Flash/Pro 集合编排的进化代码搜索。

循环：

1. 从种子程序 + 程序化评估器开始（返回适应度分数）。
2. LLM 集合提出变异。
3. 通过评估器运行变异。
4. 保留最好的；再次变异。

已发表的胜利：

- 56 年来首次超越 Strassen 算法的 4x4 复数矩阵乘法改进（48 次标量乘法）。
- 通过 Borg 调度启发式回收了 0.7% 的 Google 计算资源。
- 在前沿工作负载上实现 32% 的 FlashAttention 加速。

硬约束：适应度函数必须是机器可检查的。对散文答案进行进化搜索不会收敛。

### 何时使用哪个

| 问题类别 | 使用 | 原因 |
|----------|------|------|
| 带硬约束的调度 | HTN + ChatHTN | 可证明的正确性 |
| 编译器优化 | AlphaEvolve | 机器可检查的适应度 |
| 多步骤任务执行 | ReAct / ReWOO | 循环中有 LLM，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略约束的自动化 | HTN | 前提条件编码策略 |

### 此模式出错的地方

- **没有操作符的 HTN。** 没有前提条件/效果 schema，正确性声明就会崩溃。ChatHTN 的"LLM 建议分解"需要 schema 来拒绝无效的动作。
- **AlphaEvolve 没有真正的评估器。** "问 LLM 代码是否更好"不是一个适应度函数。评估器必须是确定性的和快速的。
- **过度工程化。** 大多数 Agent 任务不需要这两者。首选 ReAct 或 ReWOO。

## 构建

`code/main.py` 实现两个玩具：

- 一个标准库 HTN 规划器，带有操作符、方法、前提条件、效果和一个在没有方法匹配复合任务时触发的 `LLMFallback`。"LLM"是一个脚本化分解器，使规划器可离线运行。
- 一个算术程序上的标准库进化搜索：生成表达式，其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行：

```
python3 code/main.py
```

轨迹显示 HTN 规划器分解一个复合任务（带有一个中途的 LLM 回退）以及进化循环收敛到一个目标表达式。

## 使用

- **HTN 规划器** —— `pyhop`、`SHOP3`，或为领域特定策略执行构建自己的。
- **ChatHTN** —— 研究代码；模式（符号 + LLM 回退）可以干净地移植到任何 HTN 规划器。
- **AlphaEvolve** —— DeepMind 论文；模式（集合 + 评估器）是可重现的。OpenEvolve 和类似的开源分支正在涌现。
- **Agent 框架** —— 尚未提供一等 HTN 或 AlphaEvolve。将其构建为子 Agent 或后台工作器。

## 交付物

`outputs/skill-hybrid-planner.md` 生成一个混合规划器支架（HTN 或进化的），显式限定了 LLM 角色的范围。

## 练习

1. 扩展 HTN 规划器支持回溯：当操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 向 ChatHTN 添加 LLM-方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时首先重新检查方法库。
3. 将进化搜索评估器替换为真实测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛所需代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合使用：使用 HTN 将复合任务分解为子任务，然后在每个子任务的原语操作符上使用进化搜索。在哪些场景表现优异，哪些场景过度工程化？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------|----------|
| HTN | "层次规划器" | 带操作符、前提条件、效果的任务分解 |
| Method（方法） | "分解规则" | 将复合任务分解为子任务的方式 |
| Operator（操作符） | "原语动作" | 带前提条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | LLM 集合变异代码；确定性评估器选择 |
| Fitness function（适应度函数） | "评估器" | 对输出的确定性、机器可检查的分数 |
| Online method learning（在线方法学习） | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
