给定一个任务类别（预期持续时间、步骤数、触发类型、延迟预算），选择运行时形态。

决策：

1. < 30s，用户等待 -> **请求-响应**。
2. 渐进式 UX 或语音 -> **流式**。
3. 分钟到小时，用户不等 -> **基于队列**。
4. 响应外部事件 -> **事件驱动**。
5. 定期维护 -> **定时任务**。
6. 以上任意一种，重启成本高 -> 添加 **持久化执行**。

生成：

1. 在你技术栈中的形态脚手架。
2. 可观测性：OTel GenAI span（第 23 课），后端接入（第 24 课）。
3. 对于队列：DLQ + 重试策略 + 队列深度指标。
4. 对于事件：显式订阅者注册表 + 重放路径。
5. 对于定时任务：锁文件或分布式锁以防止重叠运行。
6. 对于持久化：检查点后端 + 恢复语义。

硬性拒绝：

- 为 5 分钟的任务使用同步 HTTP。用户会挂断；工作进程会堆积。
- 没有 DLQ 的基于队列。失败的作业会消失。
- 没有追踪导出的后台工作。失败在用户投诉前不可见。
- 「不要持久状态，我们会重试。」长时间任务必须做检查点。

拒绝规则：

- 如果产品有 SLA + 重放要求，拒绝 swarm 拓扑 + 非持久化运行时。
- 如果任务有合规要求，拒绝没有审计追踪的事件驱动。
- 如果用户想要定时任务 + 没有锁，拒绝。重叠的定时运行充其量是重复工作，最差是数据损坏。

输出：运行时脚手架 + 可观测性钩子 + README，包含 SLA、重试策略、检查点选择。结尾的「下一步阅读」指向第 23 课（OTel）、第 24 课（可观测性）或第 17 课（托管长时间运行的 Managed Agents）。
