给定一个仓库路径和将在其中运行的智能体产品，审计七个工作台表面并生成准备就绪报告。

七个表面：

1. 指令（Instructions）：智能体首先读取的根文件（如 `AGENTS.md`），简短，路由到更深层的规则。
2. 状态（State）：一个持久的、机器可读的文件，记录任务、涉及的文件、阻塞项、下一步动作。
3. 范围（Scope）：每个任务的合约，列出允许的文件、禁止的文件、验收标准、回滚计划。
4. 反馈（Feedback）：一个运行器，捕获命令、stdout、stderr、退出码，并将结果反馈回循环。
5. 验证（Verification）：一个门控，运行测试、lint、类型检查、冒烟运行，并确认验收标准。
6. 审查（Review）：以不同角色进行第二轮检查，构建者不能标记自己的工作。
7. 交接（Handoff）：一个产物，总结变更了什么、为什么、还剩什么以及下一步最佳动作。

生成：

- 每个表面的分数：0 缺失，1 部分存在，2 健康。将每个分数关联到你观察到的文件或流程。
- 按杠杆排序的三个优先级：哪个缺失的表面如果最先添加，能消除最多的失败模式。
- 一个机器可读的 `workbench_audit.json` 报告和一个人类可读的 `workbench_audit.md` 摘要。
- 最弱表面的起始补丁：最小的文件更改，将分数从 0 提升到 1。

硬性拒绝：

- 没有文件路径或流程引用的「健康」分数。没有证据的审计会腐烂。
- 单个合并的「智能体配置」表面。合并表面会在任务出错时隐藏具体哪个表面失败。
- 因为测试慢而跳过验证。如果验证不在工作台上，构建者就是在批改自己的作业。

拒绝规则：

- 如果仓库根本没有测试命令，拒绝验证分数并将其作为阻塞性发现提出。
- 如果仓库没有版本控制历史，拒绝交接分数并将其作为阻塞性发现提出。
- 如果智能体产品以 root 身份运行或具有不受限制的文件访问权限，在定义沙箱或写入列表之前拒绝范围分数。

输出结构：

```
workbench-audit/
├── workbench_audit.json
├── workbench_audit.md
├── patches/
│   └── <weakest-surface>.patch
└── README.md
```
