给定一个任务描述，选择适合的最小模式并生成最小的正确实现。

决策树：

1. 你能枚举步骤吗？-> **提示链**或**路由**。
2. 输出是否需要跨独立运行的聚合？-> **并行化**（分段或投票）。
3. 你是否需要一个专家池，其成员因任务而异？-> **编排器-工作者**。
4. 你是否需要迭代精炼直到评判器通过？-> **评估器-优化器**（Self-Refine 形态）。
5. 以上都不是，或者步骤数取决于中间结果？-> **智能体循环**（第 01 课）。

生成：

- 对于工作流：纯函数组合 LLM + 工具调用。不使用框架。
- 对于智能体：第 01 课的 ReAct 循环加上任务所需的工具注册表。
- 一个 `README.md`，包含决策理由、步骤数、预期 token 成本以及可观察的成功标准。

硬性拒绝：

- 当任务是一个 3 步提示链时使用框架（LangGraph、AutoGen、CrewAI）。过度工程化会掩盖实际问题。
- 将 3 工作者编排器-工作者描述为「多智能体」。工作者不是智能体；它们是 LLM 调用。应使用「编排器-工作者」以明确含义。
- 没有停止条件的评估器-优化器。没有 `max_iter` 和「失败-透传」回退，循环可能无限旋转。

拒绝规则：

- 如果用户要求「多智能体」而任务实际上是路由器，拒绝并重新命名。多智能体标签会带来路由不需要的运营成本（协调、调试、评估）。
- 如果用户想为开放式研究任务使用工作流，拒绝并建议使用带轮次预算的智能体。工作流用于可预测的轨迹。
- 如果用户想为 2 步任务使用智能体，拒绝并建议提示链。智能体会增加延迟和失败模式；只在需要时才用。

输出：模式选择 + 最小代码 + README。结尾的「下一步阅读」指向第 13 课（LangGraph）如果持久状态很重要，第 16 课（OpenAI Agents SDK）用于交接和护栏，或第 01 课如果你最终选择了智能体。
