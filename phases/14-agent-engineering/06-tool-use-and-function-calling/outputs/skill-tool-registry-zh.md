给定一个任务领域，生成一个智能体可以在 BFCL V4 各维度（智能体式、多轮、实时、非实时、幻觉）上可靠使用的工具目录。

生成内容：

1. 工具定义。每个工具包含：`name`（snake_case）、`description`（告诉模型何时使用它以及何时不使用）、带类型属性的 JSON Schema 输入、必填字段、适用的枚举值、数值的最小/最大值、每个工具的超时时间、每个工具的沙箱策略（文件系统表面、网络、内存上限）。
2. 描述质量检查。对每个描述进行「这是否告诉模型何时选择此工具而非其他工具？」的检查。如果两个工具的描述重叠，拒绝并重写。
3. 并行分发计划。对每个实际任务，识别哪些工具调用是独立的（可并行化），哪些必须顺序执行。生成预期的分发图。
4. 验证策略。枚举检查、类型强制规则（例如「接受 int 作为字符串，拒绝 float 作为字符串」）、必填字段强制。每次失败返回结构化观察字符串，绝不抛出到循环层。
5. 可观测性。每个工具发出 OpenTelemetry GenAI `tool_call` span，属性包含 `gen_ai.tool.name`、`gen_ai.tool.call.id`、`gen_ai.tool.call.arguments`、`gen_ai.tool.call.result`（当内容策略要求时使用引用而非内联）。

硬性拒绝：

- 通用 shell/命令执行工具。拒绝并拆分为具体动词（`git_status`、`fs_read`、`npm_test`）。
- 当参数具有封闭值集合时缺少枚举。枚举验证是捕获漂移最廉价的方式。
- 两个不同工具使用相同描述。模型无法可靠地在它们之间选择。
- `description` 仅命名工具（「将两个数相加」）。应包含何时选择它而非替代工具。
- 没有超时时间。每个工具调用必须有上限。

拒绝规则：

- 如果单个智能体的工具列表超过 30 个工具，拒绝并建议使用子智能体委派（第 17 课）。
- 如果任何工具在没有确认门控的情况下执行破坏性操作，拒绝并指向第 09 课（权限、沙箱）。
- 如果任务是计算机使用（点击、输入、截图），拒绝并指向第 21 课——这是一种独立的工具形态，涉及基于视觉的动作。

输出：一个可直接粘贴到 Anthropic / OpenAI / Gemini SDK 调用的 JSON 工具目录、一个分发图、一个验证策略文档，以及一个注册表应通过的 BFCL 风格迷你评估。

结尾的「下一步阅读」指向：第 09 课（沙箱）、第 23 课（OTel GenAI span）或第 30 课（评估驱动）。
