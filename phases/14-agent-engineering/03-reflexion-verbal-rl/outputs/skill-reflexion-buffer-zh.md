给定一个任务类别（重复类型的智能体运行——例如「重构一个函数」、「关闭一个支持工单」），维护一个情景记忆反思缓冲区。每个反思以自然语言记录一种失败模式和纠正性见解。缓冲区会在同一任务类别的下一次试验中前置。

生成内容：

1. 反思捕获。在一次试验以低于阈值的评估分数结束后，生成一行反思，格式为「我未能做到 X，因为 Y；下次应该 Z」。丢弃外部失败（网络故障、上游 500 错误）的反思，除非这些失败是可复现的。
2. TTL 和去重。反思默认在 N 次试验后过期（建议 10 次）。完全重复的条目合并。近似重复（小嵌入模型上余弦相似度 >0.9，或共享子串 >= 80%）只保留最新的一条。
3. 作用域策略。三种作用域：task-class（按任务名称）、user（跨用户同一任务）、agent（跨所有用户）。默认为 task-class。仅当反思涉及用户特定偏好时才升级到 user 作用域；绝不自定升级到 agent 作用域。
4. 压缩。当缓冲区超出预算时，运行休眠时压缩：聚类近似重复项、总结、合并。压缩在热路径之外运行——不要延迟主要智能体的响应。
5. 提示集成。生成一个标题为「我从之前的试验中学到了什么」的单独块，包含项目符号列表。提示中最多 6 条；超出部分放入单独的摘要条目（「……及 4 条关于超时的旧反思」）。

硬性拒绝：

- 反思写成「下次更小心一些」。这是不可操作的。用强制生成具体下次指令的提示重新运行反思器。
- 基于墙上时钟时间而非试验次数来过期反思。TTL 应以试验次数为范围，而非时间范围，以支持离线可重放的运行。
- 存储包含密钥（API 密钥、令牌、PII）的反思。在提交到缓冲区之前以特定的「包含密钥」类错误拒绝。

拒绝规则：

- 如果没有附加评估器，拒绝并推荐第 05 课（Self-Refine/CRITIC）——反思需要信号，而非直觉。
- 如果任务类别是一次性的（永不重复），拒绝；情景记忆对永不重复的任务毫无意义。

输出：一个结构化缓冲区文件（JSON，包含反思对象：trial id、task class、scope、text、created_at、ttl_remaining），一个用于下一次试验的提示块，以及一份「即将过期的反思」报告，列出即将过期的条目。

结尾的「下一步阅读」指向第 06 课（上下文压缩）如果缓冲区持续达到上限，或第 08 课（Letta 休眠时计算）以将压缩移出热路径。
