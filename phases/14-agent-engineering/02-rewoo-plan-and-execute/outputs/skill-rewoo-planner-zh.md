给定一个用户请求和一个工具目录（名称、输入模式、描述），生成一个 ReWOO 计划：一个包含工具调用和证据引用（`#E1`、`#E2`……）的 DAG。在执行前验证该计划。

生成内容：

1. 一个计划 DAG。每个节点包含 id（`E1`、`E2`……）、工具名称、参数字典（字符串可以包含 `#E<k>` 引用）以及可选的 `parallel_group` 标签。
2. 验证输出。通过拓扑排序进行无环性检查；引用解析检查（每个 `#E<k>` 都有一个前置生产者）；工具存在性检查（每个工具名称都在目录中）；参数模式检查（每个参数都匹配工具的输入模式）。
3. 并行性提示。对于每个拓扑层级，列出可以并发执行的节点。
4. 规划器/求解器拆分建议。如果计划少于 3 步，推荐 ReAct。如果计划需要无界循环（每一步都重新规划），推荐带重新规划器的 Plan-and-Execute。如果计划超过 30 步或面向 web/mobile，推荐带合成计划数据的 Plan-and-Act。

硬性拒绝：

- 包含循环的计划。ReWOO 假设是 DAG；循环属于 ReAct 或 LATS 的范畴。
- 引用 `#E<k>` 而 `k` 在拓扑顺序中尚不存在的计划。明确标注失败的边。
- 调用目录中不存在的工具的计划。不要为凑计划而编造工具。
- 引用参数类型与工具模式不匹配的计划（例如，`#E1` 替换为字符串，但工具期望整数）。

拒绝规则：

- 如果任务是开放式探索（需要的工具未知、步骤未知），拒绝并推荐 ReAct 或 LATS（第 04 课）。
- 如果工具目录中包含没有门控审批工具的破坏性工具，拒绝并指向第 09 课（权限、沙箱）。

输出：一个结构化计划（JSON 或 YAML）、一个验证报告、一个并行性映射，以及一个指向执行器（ReWOO Worker）、重新规划器（Plan-and-Execute）或更大的轨迹采样循环（Plan-and-Act）的后续行动。

结尾的「下一步阅读」指向第 03 课（Reflexion）如果此类任务之前尝试过，或第 04 课（LATS）如果计划能从搜索中受益。
