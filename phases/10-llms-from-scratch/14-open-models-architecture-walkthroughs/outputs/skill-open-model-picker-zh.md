给定部署目标（GPU 类型、每 GPU VRAM、GPU 数量、目标上下文长度、目标 p50/p99 延迟、峰值并发请求）和任务概况（聊天、代码、推理、长上下文检索、工具使用），推荐一个开源模型加服务栈，并对第 14 课的六个架构旋钮逐一进行明确论证。

输出：

1. 模型短名单。三个候选模型，每个包含总参数、激活参数（考虑 MoE）、架构标志（归一化/激活/位置/注意力/MoE/上下文），以及入选短名单的唯一理由。
2. 内存预算检查。对首选候选：BF16 和所选量化的权重内存；目标上下文和目标批次大小的 KV 缓存；激活余量。如果权重 + KV 缓存 + 激活超过可用 VRAM，停止推荐。
3. 量化选择。GPTQ-4bit、AWQ-4bit、FP8 或 BF16。对照任务对准确性的敏感度进行论证（代码/数学/推理任务受激进量化的影响比聊天或检索更大）。
4. 推理栈。vLLM、TensorRT-LLM、SGLang 或 llama.cpp。对照以下进行论证：连续批处理需求、推测式解码支持、量化格式兼容性、以及单节点 vs 多节点拓扑。
5. 吞吐量健全性检查。基于 GPU 内存带宽（解码）和 TFLOPS（预填充）的预填充 tokens/秒和解码 tokens/秒估算。如果解码吞吐量低于目标并发用户下限，拒绝推荐。
6. 回退方案。如果首选超出 VRAM 或吞吐量预算，提供第二选择。始终指定一个。

直接拒绝项：
- 在单张 24GB 消费级 GPU 上不使用卸载或激进量化的 30B 以上密集模型。
- 在没有专家并行支持的服务栈上使用 MoE 模型。
- 在没有 GQA 或 MLA 的架构上使用长上下文（128k+）（KV 缓存会爆炸）。
- 任何不指定具体模型版本的推荐（例如"Llama 3 8B Instruct v3.1"，而非"Llama 3"）。

输出：一页推荐，列出模型、量化、栈，每个决策带上编号证据。以"值得重新考虑的条件是……"段落结尾，说明会改变选择的具体能力或部署参数。
