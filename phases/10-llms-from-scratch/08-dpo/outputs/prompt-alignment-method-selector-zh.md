# 对齐方法选择器

在为语言模型选择对齐方法时，使用此框架评估你的数据、计算和质量要求，然后选择最适合约束条件的方法。

## 输入要求

提供：
- **基础模型**（例如 Llama 3 8B、Mistral 7B、Qwen 2.5 72B）
- **起始点**（基础模型还是已经过 SFT？）
- **可用数据**（指令对、偏好对、非配对评分，或无）
- **计算预算**（GPU 小时、GPU 数量）
- **质量目标**（原型可用、与开源竞争、最先进）
- **时间线**（天、周、月）

## 决策矩阵

### 快速选择

| 你的情况 | 推荐方法 | 原因 |
|---------------|-------------------|-----|
| 无偏好数据，仅指令对 | 仅 SFT | 无偏好信号无法对齐 |
| < 5,000 偏好对，有限计算 | DPO | 更简单的流水线，小数据也有效 |
| 非配对反馈（仅有赞/踩） | KTO | 唯一不需要成对比较的方法 |
| 想在单次训练运行中完成对齐 | ORPO | 结合 SFT + 对齐，无需参考模型 |
| 内存受限（无法容纳参考模型） | SimPO | 无需参考模型 |
| 大规模、多目标对齐 | RLHF (PPO) | 独立奖励模型捕捉复杂偏好 |
| 带在线数据的迭代对齐 | RLHF (PPO) | 可在循环中生成、评分和重新训练 |
| 后 RLHF 精炼 | DPO | 在定向偏好上微调 RLHF 模型 |

### 详细对比

| 方法 | 数据需求 | 内存中模型 | 训练循环 | 稳定性 | 最佳规模 |
|--------|-----------------|-----------------|----------------|-----------|------------|
| SFT | 指令对 (10K+) | 1 | 1 | 高 | 任意 |
| RLHF | 偏好对 (20K+) | 3-4 | 3 | 低 | 大 (70B+) |
| DPO | 偏好对 (5K+) | 2 | 2 (SFT + DPO) | 高 | 小-中 (7B-70B) |
| KTO | 非配对评分 (5K+) | 2 | 2 (SFT + KTO) | 高 | 任意 |
| ORPO | 偏好对 (10K+) | 1 | 1 | 高 | 小-中 |
| SimPO | 偏好对 (5K+) | 1 | 2 (SFT + SimPO) | 高 | 小-中 |

## 方法特定配置

### SFT

- **何时停止**：1-3 轮后或验证损失不再下降时
- **关键超参数**：学习率（1e-5 到 5e-5，更大模型用更低值）
- **关键细节**：在损失中掩码掉指令 token
- **陷阱**：超过 3 轮导致记忆化；混入 2-5% 预训练数据

### RLHF (PPO)

- **何时使用**：你有 20K+ 比较对，需要多目标对齐，或想迭代在线学习
- **关键超参数**：KL 系数（0.01-0.05）、PPO 裁剪比例（0.1-0.3）、学习率（5e-6 到 3e-5）
- **关键细节**：奖励模型应 >= 策略模型大小
- **陷阱**：PPO 不稳定；持续监控 KL 散度和奖励曲线

### DPO

- **何时使用**：你有偏好对且需要比 RLHF 更简单的流水线
- **关键超参数**：Beta（0.1-0.5；越低 = 允许偏离参考模型越多）
- **关键细节**：参考模型必须是 SFT 检查点的冻结副本
- **陷阱**：对 beta 非常敏感；在 [0.05, 0.1, 0.2, 0.5] 上扫描

### KTO

- **何时使用**：你仅有"好"或"坏"标签，没有成对比较
- **关键超参数**：Beta（同 DPO），损失厌恶系数（对坏回答 1.5 倍）
- **关键细节**：需要大致平衡的好/坏示例（40-60% 划分）
- **陷阱**：没有成对比较，梯度信号更弱；可能需要比 DPO 更多数据

### ORPO

- **何时使用**：你想完全跳过 SFT，直接从基础模型到对齐模型
- **关键超参数**：Lambda（偏好项对 SFT 项的权重）
- **关键细节**：需要一个数据集中既有指令标签又有偏好对
- **陷阱**：组合目标难以平衡；如 SFT 损失占主导，对齐很弱

### SimPO

- **何时使用**：内存受限无法容纳参考模型
- **关键超参数**：Beta、gamma（长度归一化指数）
- **关键细节**：长度归一化防止模型偏好短回答
- **陷阱**：无参考模型锚点，模型可能漂移更远；仔细监控

## 流水线模板

### 模板 1：快速原型（1-2 天）

```
基础模型 -> SFT（1 轮，10K 示例）-> DPO（3 轮，5K 对）
```

计算：A100 上 7B 模型约 4 GPU 小时
质量：扎实的指令遵从，基础偏好对齐

### 模板 2：生产质量（1-2 周）

```
基础模型 -> SFT（2 轮，50K 示例）-> DPO（5 轮，20K 对）-> 评估 -> 迭代
```

计算：7B 约 40 GPU 小时，70B 约 200 GPU 小时
质量：与开源 RLHF 模型竞争

### 模板 3：最先进（1-3 个月）

```
基础模型 -> SFT（2 轮，100K+ 示例）-> RLHF（PPO，50K+ 对）-> DPO（定向精炼）-> 评估 -> 迭代
```

计算：70B 约 500+ GPU 小时
质量：接近前沿模型对齐水平

### 模板 4：最小数据（1-2 天）

```
基础模型 -> SFT（1 轮，5K 示例）-> KTO（来自用户的非配对赞/踩）
```

计算：7B 约 2 GPU 小时
质量：比仅 SFT 更好，数据收集开销最小

## 评估协议

对齐后，从以下维度评估：

1. **偏好胜率**：在 200+ 测试提示上用人工裁判对比对齐模型 vs SFT 模型。目标：> 60% 胜率。
2. **基准保持**：MMLU、HumanEval 或领域特定基准。不应相比 SFT 基线下降 > 5%。
3. **MT-Bench 或 AlpacaEval**：标准对齐质量基准。对比已发布基线。
4. **安全评估**：测试对抗提示、越狱和有害请求类别。
5. **回答多样性**：在 100 个提示上测量回答熵。低熵 = 模式崩塌。

## 常见失败模式

| 症状 | 原因 | 方法特定修复 |
|---------|-------|-------------------|
| 啰嗦、填充式回答 | 奖励模型/隐式偏好奖励长度 | DPO：增加 beta。RLHF：添加长度惩罚。SimPO：调整 gamma。 |
| 模型同意一切 | 偏好数据偏差导致的谄媚 | 添加正确答案与用户不一致的偏好对 |
| 拒绝良性请求 | 安全数据过度对齐 | 降低安全示例比例，添加更多良性拒绝对 |
| 输出几乎与 SFT 相同 | Beta 过高（DPO/KTO）或 KL 系数过高（PPO） | 降低 beta / KL 系数；模型没有学习 |
| 训练损失震荡 | 学习率过高或数据不足 | 降低 lr 2-3 倍，增加偏好数据 |
