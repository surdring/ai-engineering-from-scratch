# 原生稀疏注意力（DeepSeek NSA）

> 在 64k token 时，注意力消耗 70-80% 的解码延迟。每个开放模型实验室都有修复计划。DeepSeek 的 NSA（ACL 2025 最佳论文）是被采纳的解决方案：三个并行的注意力分支——压缩的粗粒度 token、选择性保留的细粒度 token 和用于局部上下文的滑动窗口——通过学习的门组合在一起。它硬件对齐（内核友好），原生可训练（在预训练中使用，而非在推理时外挂），并且在 64k 解码上运行比 FlashAttention 更快，同时匹配或超过全注意力质量。本课从头到尾构建三个分支，并展示为什么稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 第七阶段 · 12（KV 缓存、flash-attention），第七阶段 · 15（注意力变体），第十阶段 · 16（差分注意力）
**时间：** 约 60 分钟

## 学习目标

- 陈述三个 NSA 注意力分支以及每个分支捕获什么。
- 解释为什么 NSA 是"原生可训练的"，而之前的稀疏注意力方法仅是推理时使用的。
- 计算 NSA 与全注意力在 64k 上下文下的注意力计算节省，作为压缩块大小和选择 top-k 的函数。
- 在标准库 Python 中在短合成序列上实现三分支组合，并验证门权重行为正确。

## 问题

序列长度 N 的全注意力花费 `O(N^2)` 时间和每层 `O(N)` KV 缓存。在 64k token 时，计算和内存带宽数字是灾难性的。NSA 论文中的理论估计：在 64k 时注意力占总解码延迟的 70-80%。下游的一切——TTFT、tokens/sec、每百万 token 成本——都由注意力成本主导。

稀疏注意力是显而易见的答案。之前的尝试分为两类。固定模式稀疏（滑动窗口、步长、块局部）丢弃信息并在长距离召回任务上失败。推理时稀疏（KV 缓存剪枝、H2O、StreamingLLM）应用于在稠密注意力上预训练的模型，只能恢复潜在加速的一小部分，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等人，DeepSeek + 北大 + 华盛顿大学，ACL 2025 最佳论文，arXiv:2502.11089）做到了两者：模型在预训练期间学习的稀疏模式，实现为在推理时实际提供计算节省的内核对齐算法。两年后，NSA 或其直接后代将是每个前沿长上下文模型上的默认注意力。

## 核心概念

### 三个并行分支

对于每个查询，NSA 运行三次注意力，针对 KV 缓存的三个不同视图：

1. **压缩分支。** Token 被分组为大小为 `l`（通常为 32 或 64）的块。每个块通过一个小的可学习 MLP 压缩成一个摘要 token。查询对这些压缩 token 进行注意力操作，获得整个序列的粗粒度视图。

2. **选择分支。** 使用来自压缩分支的注意力分数，识别与当前查询最相关的 top-k 个块。读取来自这些块的细粒度（未压缩）token，查询对所有它们进行注意力操作。将压缩分支注意力视为选择的路由信号。

3. **滑动窗口分支。** 查询对最近的 `W` 个 token（通常为 512）进行注意力操作以获取局部上下文。此分支捕获其他两个分支可能遗漏的结构密集的短程模式（语法、局部共指）。

三个分支输出通过学习的逐位置门组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自查询上的小型 MLP 的门权重。它们不必总和为 1——它们可以独立加权分支。

### 为什么这是"原生可训练的"

选择步骤（top-k 块）是离散的。离散操作打断梯度流。之前的稀疏注意力工作要么跳过选择的反向传播（限制训练），要么使用在推理时不给出真正稀疏性的连续松弛。

NSA 规避了这一点：压缩分支注意力本身就是整个序列上可微的粗粒度注意力。top-k 操作只是重用来自压缩分支的最高注意力分数来选择加载哪些细粒度块。梯度流过压缩分支分数（影响压缩输出和选择逻辑），被选择块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图上是一个无操作——它只控制从内存加载哪些块。

这就是为什么 NSA 可以在预训练中端到端使用。模型学会通过三个分支联合路由信息，产生在推理时实际提供承诺加速的稀疏模式。

### 硬件对齐内核

NSA 的内核是为现代 GPU 内存层次结构设计的。内核按 GQA 组加载查询（外循环），为每个组获取对应的稀疏 KV 块（内循环），并在 SRAM 上运行注意力。因为每个查询组看到相同的选择块（选择是按查询组而非按查询头的），KV 加载在各组间分摊。算术强度保持高。

论文报告 Triton 内核在 64k 解码上运行速度比 FlashAttention 快 9x，加速比随序列长度增长。前向和后向内核均已提供。

### 计算预算

令 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择计数，`w` 为滑动窗口，`b` 为选择块大小（通常等于 `l`）。

- 压缩分支：每个查询 `O(N/l)` 个键，总计 `O(N * N / l)`。
- 选择分支：每个查询 `O(k * b)` 个键，总计 `O(N * k * b)`。
- 滑动分支：每个查询 `O(w)` 个键，总计 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当 `N = 64k, l = 64, k = 16, b = 64, w = 512` 时：每个查询成本为 `1000 + 1024 + 512 = 2536 个键`。全注意力为 `64000 个键`。25x 计算减少。

当 `N = 128k, l = 64, k = 16, b = 64, w = 512` 时：每个查询成本为 `2000 + 1024 + 512 = 3536 个键`。全注意力为 `128000 个键`。36x 减少。收益随序列长度增长，这正是全部要点。

### 如何比较

| 方法 | 可微 | 真实推理加速 | 长距离召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 步长/块稀疏 | 是 | 是 | 部分 |
| KV 剪枝（H2O、StreamingLLM） | 不适用（推理时） | 是 | 部分 |
| MoBA（月之暗面） | 部分 | 是 | 好 |
| NSA | 是（原生） | 是（64k 时 9x） | 匹配全注意力 |

MoBA（月之暗面，arXiv:2502.13189）是并发发表的，采用类似的三路优于一路方法，将 MoE 原理应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练需要了解的两种架构。

## 构建实现

`code/main.py` 在短合成序列上实现三个分支，并显示：

- 压缩 MLP（使用简单的均值池化基线以保持教学清晰；真实 NSA 使用可学习的 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 对最后 `w` 个 token 的滑动窗口注意力。
- 门控组合。
- 与全注意力比较的计算量计数打印。

### 步骤 1：将 token 压缩为块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤 2：压缩分支注意力

对查询与压缩键进行 softmax 注意力。压缩分支分数同时作为 top-k 选择的信号。

### 步骤 3：top-k 块选择

选择 `k` 个最高分压缩块的索引。加载来自这些块的原始未压缩 token 并对它们运行注意力。

### 步骤 4：滑动窗口注意力

取最后 `w` 个 token 并对它们运行标准注意力。

### 步骤 5：门控 + 组合

查询上的小型 MLP 产生三个门权重。最终输出是三个分支输出的加权和。

### 步骤 6：计算量计数

打印每个分支每个查询关注的键数和总数。与 `N`（全注意力）比较。在 `l = 32, k = 4, w = 128` 的 1024 token 合成数据上，NSA 每个查询看到 `32 + 128 + 128 = 288` 个键，而全注意力为 1024——减少 3.5x。

## 使用方式

NSA 正在 DeepSeek 自身的长上下文预训练流水线中使用。截至 2026 年 4 月，在公共推理栈中的集成状态：

- **DeepSeek 内部**：原生，已发布的权重使用 NSA 或其后续 DSA（Deepseek Sparse Attention）。
- **vLLM**：为 DeepSeek-V3.x 权重的实验性 NSA 支持正在开发中。
- **SGLang**：NSA 基准测试已发表；生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在 CPU 吞吐量下不值得。

何时使用 NSA：

- 以 64k 以上上下文为目标且有大规模计算预算的预训练或继续训练运行。
- DeepSeek 自身的长上下文检查点的推理。权重是 NSA 原生的。

何时不使用：

- 服务于已有稠密注意力预训练模型。没有继续训练就无法改装 NSA。
- 上下文在 16k 以下。三分支开销超过节省。
- 批量-1 交互式聊天。延迟敏感的解码能受益，但仅在长上下文下。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 压缩分支 | "粗视图" | 对块平均键的注意力，以每个查询 O(N/l) 个键提供全局上下文 |
| 选择分支 | "top-k 块" | 对具有最高压缩分支分数的 `k` 个块进行细粒度注意力 |
| 滑动窗口 | "局部上下文" | 对最后 `W` 个 token 的注意力，用于短程模式 |
| 原生可训练 | "在稀疏性开启时预训练" | 稀疏模式在预训练期间学习，而非在推理时外挂 |
| 压缩块大小 l | "粗视图的分组大小" | 多少个 token 合并为一个摘要；典型值 32-64 |
| Top-k | "保留的块数" | 其未压缩 token 被读取的压缩块数量；典型值 16 |
| 滑动窗口 W | "局部注意力半径" | 通常 512；更短损害局部一致性，更长浪费计算 |
| 分支门 | "如何混合三者" | 加权三个分支贡献的逐位置 MLP 输出 |
| 硬件对齐 | "内核友好的稀疏性" | 稀疏模式选择使得实际 GPU 内核达到理论加速比 |
| DSA | "NSA 的后继" | Deepseek Sparse Attention，DeepSeek 谱系中跟随 NSA 的架构 |
