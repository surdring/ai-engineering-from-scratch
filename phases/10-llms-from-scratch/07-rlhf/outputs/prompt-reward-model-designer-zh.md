# 奖励模型设计器

在构建 RLHF 流水线将语言模型对齐到目标行为（有用性、编程能力、安全性、诚实性）时，使用此框架设计数据收集协议、训练奖励模型并配置 PPO。

## 输入要求

提供：
- **目标行为**（例如"有用且无害的助手"、"专业 Python 编码器"、"带安全的医疗问答"）
- **基础模型**（例如经过 SFT 的 Llama 3 8B、Mistral 7B Chat）
- **奖励模型大小**（通常与策略模型相同或更大）
- **标注预算**（人工工时或可用的比较对数量）
- **计算预算**（奖励模型训练 + PPO 的 GPU 小时数）

## 第一步：偏好数据收集

### 标注协议

1. **提示选择**：从 SFT 训练分布中采样，加上分布外提示（10-20% 新颖）
2. **回答生成**：使用不同温度（0.3、0.7、1.0）的 SFT 模型为每个提示生成 2-4 个回答
3. **比较格式**：向标注者展示恰好 2 个回答，并询问"哪个回答更好？"
4. **评分标准**：定义"更好"对你的用例意味着什么

### 评分标准模板

| 标准 | 权重 | 描述 |
|-----------|--------|-------------|
| 有用性（Helpfulness） | 40% | 是否完整且正确地回答了问题？ |
| 无害性（Harmlessness） | 25% | 是否避免了有害、有偏见或误导性内容？ |
| 诚实性（Honesty） | 20% | 是否承认不确定性而非产生幻觉？ |
| 简洁性（Conciseness） | 15% | 回答长度是否适合该问题？ |

为你的用例调整权重。编程助手可能将正确性设为 60%，简洁性设为 20%。

### 数据量指南

| 规模 | 比较对 | 标注工时 | 预期 RM 准确率 |
|-------|-----------------|-----------------|---------------------|
| 最小可用 | 5,000-10,000 | 400-800 | 60-65% |
| 生产 v1 | 20,000-50,000 | 1,600-4,000 | 65-72% |
| 生产 v2 | 100,000-500,000 | 8,000-40,000 | 72-78% |

InstructGPT 使用了来自 40 位承包商的 33,000 个比较对。Anthropic 初始论文使用了来自 20 位标注者的 22,000 个。标注者间一致率通常为 70-75% — 奖励模型无法超过人类一致水平。

### 质量控制

- **一致性过滤**：丢弃少于 70% 标注者同意的对
- **标注者校准**：在正式标注前用已知好对进行校准轮次
- **偏差检测**：监控标注者是否一贯偏好更长回答、正式语言或特定模式
- **对抗样本**：包含 5-10% 旨在抓住未仔细阅读的标注者的示例

## 第二步：奖励模型架构

### 架构决策

| 决策 | 推荐 | 理由 |
|----------|---------------|-----------|
| 基础架构 | 与策略相同的 Transformer | 从 SFT 检查点初始化权重提供强大的起始特征 |
| 输出头 | 从最后隐藏状态的单线性投影 | 来自最完整位置表示的标量奖励 |
| 模型大小 | >= 策略模型大小 | 较小的 RM 产生不可靠信号，会使 PPO 不稳定 |
| 初始化 | SFT 检查点 + 新输出头 | 预训练特征已捕捉语言质量 |

### 训练配置

| 参数 | 范围 | 备注 |
|-----------|-------|-------|
| 学习率 | 1e-5 到 5e-5 | 低于 SFT，因为任务更简单 |
| 轮数 | 1-3 | 在有限比较数据上过拟合是主要风险 |
| 批次大小 | 64-256 | 每个"示例"是一对，所以有效数据是 2 倍 |
| 损失函数 | Bradley-Terry: -log(sigmoid(r_preferred - r_rejected)) | 成对比较的标准 |
| 验证划分 | 10-20% | 在留出对上监控准确率 |

### 评估指标

1. **成对准确率**：RM 在留出偏好对上正确排名的比例。目标：> 65%
2. **边际分布**：绘制 (r_preferred - r_rejected) 的分布。应中心大于 0，少量负值。
3. **校准度**：sigmoid(r_preferred - r_rejected) 是否接近实际人类偏好概率？
4. **OOD 泛化**：在与训练不同分布的提示上测试。准确率下降应 < 10%。

## 第三步：PPO 配置

### 超参数

| 参数 | 典型值 | 过高影响 | 过低影响 |
|-----------|--------------|-------------------------|------------------------|
| KL 系数 (beta) | 0.01-0.05 | 模型几乎不学习，过于接近 SFT | 奖励黑客，退化输出 |
| 学习率 | 5e-6 到 3e-5 | 训练不稳定，发散 | 收敛慢，浪费计算 |
| 裁剪比例 (epsilon) | 0.1-0.3 | 可能带来大幅不稳定性更新 | 非常保守的更新，学习缓慢 |
| 每批 PPO 轮数 | 1-4 | 对当前批次过拟合 | 未充分利用每个批次 |
| 生成批次大小 | 128-512 | 内存问题 | 梯度估计噪声大 |
| 最大回答长度 | 256-1024 | 生成慢，内存问题 | 截断有用回答 |

### 监控仪表盘

在 PPO 训练期间跟踪以下指标：

1. **平均奖励（Mean reward）**：应随训练增加。平缓可接受，下降表示不稳定。
2. **KL 散度**：应保持在 10-20 nats 以下。飙升 = 奖励黑客。
3. **回答长度**：应保持稳定。单调增长 = 啰嗦型奖励黑客。
4. **熵**：Token 分布熵应缓慢下降。快速下降 = 模式崩塌。
5. **奖励模型一致率**：用奖励模型评分 PPO 回答，一致率应提高。

### PPO 期间的危险信号

| 症状 | 可能原因 | 修复 |
|---------|-------------|-----|
| 奖励增加但输出退化 | 奖励黑客 | 增加 KL 系数，用对抗样本重新训练 RM |
| KL 散度爆炸 | 学习率过高或 KL 系数过低 | 降低 lr，增加 beta |
| 回答长度单调增长 | RM 奖励啰嗦 | 在奖励中加长度惩罚，用长度控制对重新训练 RM |
| 所有回答变得相同 | 模式崩塌 | 增加生成温度，减少 PPO 轮数 |
| 奖励剧烈波动 | PPO 不稳定 | 降低学习率，增加裁剪比例 |

## 第四步：端到端验证

部署 RLHF 训练模型前：

1. **A/B 测试 vs SFT**：在 200+ 测试提示上运行 SFT 和 RLHF 模型。让 3+ 评估者比较回答。RLHF 模型胜率应 > 60%。
2. **安全评估**：在已知对抗提示（越狱、有害请求）上测试。RLHF 模型应适当拒绝。
3. **退化检查**：运行标准基准（MMLU、HumanEval、MT-Bench）确认 RLHF 模型未丢失核心能力。
4. **遗忘检查**：在通用文本语料上测量困惑度。与 SFT 模型相比增加应 < 10%。
5. **长度分析**：比较 SFT 和 RLHF 模型的平均回答长度。如 RLHF > 50% 更长，奖励模型可能存在啰嗦偏差。
