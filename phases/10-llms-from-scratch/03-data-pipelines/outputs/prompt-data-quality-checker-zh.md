# LLM 预训练数据质量检查器

在构建或审计 LLM 预训练的数据流水线时，使用此框架在问题影响模型之前发现它们。

## 流水线输出的危险信号

**去重移除了不到 20% 的网页数据。** Common Crawl 通常包含 30-40% 的重复。如果你的去重步骤移除不到 20%，则 MinHash 参数过于保守或阈值过高。检查：shingle 大小 k、哈希函数数量、LSH 带数、Jaccard 阈值。

**压缩率低于 2.0 字符/token。** 这意味着分词器拆分过于激进。要么用更多合并重新训练、增加词汇量，要么检查预分词是否不必要地打碎文本。

**压缩率高于 6.0 字符/token。** 分词器学到了非常特定领域的合并，可能无法泛化。对领域特定模型来说可以接受，但对通用模型是警告信号。

**序列利用率低于 90%。** 填充过多。要么文档太短（过滤它们或提高最小文档长度），要么序列打包效率低（从简单填充切换到多文档打包）。

**词汇表利用率低于 50%。** 超过一半词汇表在该语料库上未使用。要么词汇表对领域来说太大，要么分词器是在非常不同的数据上训练的。

## 质量过滤器校准

在流水线的每个阶段对随机 1,000 份文档运行以下检查：

1. **随机阅读 20 份清洗后文档。** 它们是否包含残留 HTML、JavaScript、导航文本或样板内容？如果是，HTML 剥离不完整。

2. **随机阅读 20 份通过了质量过滤的文档。** 是否有垃圾信息、关键词列表或机器生成内容？如果是，收紧过滤阈值。

3. **随机阅读 20 份未通过质量过滤的文档。** 是否有真正优质内容？如果是，过滤过于激进。放宽阈值或为特定模式添加例外。

4. **从去重结果中随机阅读 20 对近似重复。** 它们真的相似吗？如果不是，降低 Jaccard 阈值或增加哈希函数数量。

## 数据混合比例

没有通用公式。从以下基线开始，根据评估结果调整：

| 类别 | Llama 3 比例 | 起点 |
|----------|--------------|----------------|
| 网页文本 | 50% | 50% |
| 代码 | 25% | 15-25% |
| 书籍/学术 | 13% | 10-15% |
| 数学 | 8% | 5-10% |
| 多语言网页 | 4% | 5-10% |

如果模型应擅长编程则增加代码比例。如果推理重要则增加数学比例。如果需要更少噪声则减少网页比例。更改比例后始终在基准上评估。

## 规模估算

对于给定的目标 token 数量：

- 从网页获取 1T token：预计约 3-5TB 原始文本，清洗和去重后约 1.5-2TB
- 分词速度（Rust）：每核每秒约 100M token
- 分词速度（Python）：每核每秒约 1-10M token
- MinHash 去重（128 哈希，16 带）：每核每秒约 10K 文档
- 序列打包：I/O 密集型，对 10GB 以上语料使用内存映射文件

对于 15T token（Llama 3 规模），规划约 30-50TB 原始输入数据、在 64 核机器上 1-2 周预处理时间，以及 100TB+ 的中间文件存储。

## 训练前检查清单

1. 总 token 数量匹配计算预算（使用 Chinchilla 缩放法则或 Llama 3 过度训练比例作为指导）
2. 去重移除了 30-40% 的网页数据
3. 质量过滤移除了 10-20% 的剩余数据
4. 英语压缩率为 3-5 字符/token
5. 序列利用率高于 95%
6. 随机抽查显示流水线每个阶段都有干净连贯的文本
7. 数据混合比例已在小型训练运行中验证
8. PII（个人身份信息）移除已在样本上验证
9. 所有二进制格式（打包序列、token ID 数组）通过往返编解码测试
10. 流水线可复现：相同输入在用固定随机种子的情况下产生相同输出
