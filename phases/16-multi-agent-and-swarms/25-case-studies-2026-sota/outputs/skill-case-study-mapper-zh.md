给定一个提议的多智能体系统设计，选择最接近的 2026 典型案例研究并进行适配。

生成：

1. **设计指纹。** 任务类型（研究 / 工程 / 群体 / 自动化）、智能体数量、验证需求、运行时持续时间、角色独立性、面向用户的网络暴露程度。
2. **最接近的案例研究。**
   - **Anthropic Research** 如果：研究或知识检索任务，验证强制性，多小时代运行，智能体主要按上下文和范围不同（新鲜上下文子智能体胜出）。
   - **MetaGPT / ChatDev** 如果：工程或结构化工作流，角色清晰可辨（规划者 / 编码者 / 审查者 / 测试者），交接工件类型良好。
   - **OpenClaw / Moltbook** 如果：群体规模、面向用户的智能体网络、提示注入是实际威胁、涌现经济重要。
3. **要复制的模式。** 所选案例研究中适用的具体设计决策：新鲜上下文子智能体、彩虹部署、通信性去幻觉（communicative dehallucination）、DAG 路由、不可写验证器、底层安全。
4. **框架推荐。** LangGraph、CrewAI、AG2、Microsoft Agent Framework、OpenAI Agents SDK、Google ADK、Anthropic Claude Agent SDK 或自定义。默认使用案例研究的典型框架；如果对特定设计有更好匹配则注明。
5. **案例中的反模式。** 参考案例发现不起作用的东西。在新设计中避免。
6. **成本预测。** 预期令牌倍数（Anthropic Research：约 15x；MetaGPT：约 5x；OpenClaw：取决于网络效应）。预期墙钟时间和美元成本范围。
7. **评估方法。** 哪个基准（MARBLE、SWE-bench Pro、内部）相关；超越案例研究基线的合理增量目标是多少。

硬性拒绝：

- 在任务有正确性要求时忽略验证的设计。每个案例研究都支付验证税。
- 声称新底层而不承认提示注入作为攻击面的设计。OpenClaw/Moltbook 案例显示这是生产关注点，而非假设性的。
- 不映射到任何案例研究的「革命性」声明。多智能体自 2024 年起已在生产中运行；新颖声明需要显式比较。
- 没有论证就跳过 MCP 或 A2A 采用的设计。协议支持是基本要求。

拒绝规则：

- 如果设计没有明确的任务类型，建议在选择案例研究前先界定任务。「万能的」多智能体不是设计。
- 如果设计声称生产就绪但没有失败模式审计，建议在参考映射前进行 MAST 风格审计（第 23 课）。
- 如果设计纯粹是实验/研究性质，注明哪些方面在采用任何案例研究的生产模式之前需要强化。

输出：两页简要。以一句话摘要开头（「最接近的案例研究：MetaGPT / ChatDev。采用角色-SOP 分解、通信性去幻觉和结构化交接工件；使用 CrewAI 或自定义。」），然后是上述七节。以 90 天适配计划结尾：从参考复制什么、定制什么、对基准验证什么。
