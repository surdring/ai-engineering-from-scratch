# 角色专业化——规划者、批判者、执行者、验证者

> 2026 年最常见的多 Agent 分解：一个代理规划，一个执行，一个批判或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过「聊天链」将设计者、程序员、审查者、测试者串联起来，并带有「交际去幻觉化」（Communicative Dehallucination）（代理显式请求缺失的细节）。验证者是承重的：Cemri 等人（MAST，arXiv:2503.13657）表明每个多 Agent 失败都可追溯到缺失或损坏的验证。PwC 报告在 CrewAI 中通过结构化验证循环获得了 7 倍准确率提升（10% → 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第 16 阶段 · 04（原语模型）、第 16 阶段 · 05（监督者）
**时间：** 约 60 分钟

## 问题

通用的多 Agent 系统产生通用的输出。群聊中的三个程序员写出三种同样平庸的代码变体。你可以添加更多代理、更多轮次，仍然无法跨越质量阈值。

修复方案不是更多的代理——而是*不同的*代理。分配不同的角色。给批判者规划者没有的工具。给验证者一个客观的测试套件。现在系统有了带有扎实纠正的内部分歧，而不仅仅是并行猜测。

## 概念

### 四个规范角色

**规划者（Planner）。** 阅读目标，产生步骤列表或规格。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）。** 一次读取一个计划步骤，产生工件。工具：实际工作工具（代码编译器、Shell、API 客户端）。输出：工件。

**批判者（Critic）。** 对照规划者的意图阅读执行者的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者（Verifier）。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、Schema 验证器。输出：通过/失败及证据。

批判者是主观的、有观点的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理**编写 PRD。
- **架构师**产生系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA 工程师**运行测试。

每个角色都有严格的输入/输出 Schema。角色提示说明角色*是什么*以及它*必须产生什么*。`Code = SOP(Team)` 的表述——确定性的 SOP 将 LLM 团队转变为可预测的管道。

### ChatDev 的交际去幻觉化

ChatDev 添加了一个关键动作：当执行者需要一个规划中没有的具体细节时，它在继续之前显式地向设计者询问。这防止了经典的 LLM 失败：看似合理地编造细节。

实现：角色提示包含「当你需要你没有获得的具体信息时，在产生输出之前显式地按名称询问相关角色。」

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 个多 Agent 执行失败。21.3% 是验证缺口（Verification Gap）——系统交付了一个没人检查过的答案。其余 79% 通常追溯到「有一个检查悄悄失败了或者从未运行过」。验证是承重角色。

PwC 报告（CrewAI 部署，2025 年）添加结构化验证循环将准确率从 10% 提高到 70%。来自一个角色的 7 倍收益。

### 批判者 vs 验证者

- 批判者是一个审查工件质量的 LLM。主观的。可能被看似合理的文字所欺骗。
- 验证者是一个运行在工件上的确定性程序。客观的。以证据给出通过/失败。

两者都使用。批判者捕捉验证者无法表述的品味问题。验证者捕捉批判者看不到的 Bug，因为它们只在运行时显现。

### 反模式

系统中每个角色都是 LLM，每个角色的输出都是「看起来不错」。经典的 MAST 失败模式。至少添加一个通过/失败由代码而非 LLM 决定的验证者。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书的专业化表面。
- **LangGraph** — 节点可以有专业化的提示；边强制执行管道。
- **AutoGen** — GroupChat 中具有角色特定的 ConversableAgent，带有单一词名称。
- **OpenAI Agents SDK** — 角色专业化 Agent 之间的交接工具。

## 构建

`code/main.py` 实现了一个构建简单 Python 函数的四角色管道：

- **规划者**产生规格。
- **执行者**生成代码字符串。
- **批判者**（LLM 模拟）标记明显问题。
- **验证者**在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行者产生正确代码（批判者 + 验证者都通过），一次执行者产生不符合规格的代码（批判者遗漏了 Bug，因为它看起来合理，验证者捕获了它，因为测试失败）。

运行：

```
python3 code/main.py
```

## 实践

`outputs/skill-role-designer.md` 接收一个任务并生成角色花名册（3-5 个角色）、每个角色的输入/输出 Schema 以及验证者检查。在将代理连接到框架之前使用此工具。

## 交付

检查清单：

- **至少一个确定性验证者。** 永远不要全部 LLM。
- **每个角色的显式 I/O Schema。** 规划者返回规格，而非散文；执行者读取该 Schema。
- **交际去幻觉化。** 执行者在信息缺失时必须询问规划者；永远不要编造。
- **批判者/验证者顺序。** 先运行批判者（便宜，捕获设计问题），再运行验证者（慢，捕获 Bug）。
- **循环预算。** 在升级到人类之前最多 2 轮批判者-执行者修订。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批判者遗漏的 Bug。添加一个静态分析检查（计数 `return` 的出现次数）作为额外的验证者。它捕获了什么运行时测试遗漏的内容？
2. 添加第 5 个角色：「需求分析师」，将用户愿望转化为规划者就绪的规格。哪些交际去幻觉化请求应该流向它？
3. 阅读 MetaGPT 第 3 节（「Agents」）。列出 MetaGPT 的 5 个角色中每个的输入/输出 Schema。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。识别交际去幻觉化在哪里打破了一个原本会是无限的循环。
5. PwC 的 7 倍准确率提升来自验证循环。假设三个添加验证者不会有帮助的任务——其中正确性的确定性检查是不可能的或代价过高。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Role specialization | 「不同代理，不同工作」 | 为规划者/执行者/批判者/验证者角色调优的不同系统提示。 |
| SOP pattern | 「编码的标准操作流程」 | MetaGPT 的框架：每个角色严格的 I/O Schema 将团队转变为管道。 |
| Communicative dehallucination | 「在编造之前询问」 | ChatDev 模式：执行者在细节缺失时询问规划者而非编造。 |
| Critic | 「LLM 审查者」 | 主观的、有观点的审查者。捕捉品味问题。可能被看似合理的文字欺骗。 |
| Verifier | 「确定性检查」 | 基于代码的通过/失败。测试运行器、类型检查器、Schema 验证器。无法被欺骗。 |
| Verification gap | 「没人检查」 | 21.3% 的 MAST 失败。答案在没有检查的情况下交付，而检查本应捕获 Bug。 |
| Revision loop | 「批判者退回」 | 批判者拒绝触发执行者重新运行并带反馈。需要预算。 |
| All-LLM anti-pattern | 「看起来不错」 | 每个角色都是 LLM，没有确定性检查。经典的 MAST 失败。 |
