给定一个任务及其吞吐量/确定性需求，推荐群体（swarm）或监督者（supervisor），并列出具体的队列和护栏选择。

生成：

1. **任务独立性检查。** 子任务是独立的还是相互依赖？群体仅在独立性高时适合。
2. **时长分布。** 均匀 vs 可变。群体主要在可变时长工作负载上获胜。
3. **排序要求。** 严格、宽松或无需。群体不保留顺序；监督者保留。
4. **可调试性需求。** 高（金融、医疗）→ 监督者。中等 → 带每任务追踪 ID 的群体。
5. **队列选择。** 内存（`queue.Queue`）用于演示；Kafka / Redis Streams / NATS / 持久数据库支持用于生产。
6. **Worker 设计要求。** 必须幂等；必须发出每任务追踪；必须处理背压。
7. **防饥饿计划。** 优先级老化、worker 专业化、有界队列。
8. **可观测性计划。** 每任务 ID、开始/结束事件、结果池模式。

硬性拒绝：

- 对具有硬排序要求的任务推荐群体。
- 没有幂等 worker 的群体。
- 在生产中没有持久队列的群体。

拒绝规则：

- 如果任务每秒独立单元少于 10，拒绝群体并推荐监督者。在低吞吐量下群体开销不合理。
- 如果可观测性需求要求单一连贯追踪（审计、合规），拒绝群体并推荐 LangGraph 确定性图。

输出：一页架构简要。以适配裁决开头，以针对目标吞吐量的具体消息代理建议结尾。
