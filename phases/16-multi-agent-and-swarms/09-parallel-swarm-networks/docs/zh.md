# 并行 / 集群 / 网络化架构

> 与监督者形成对比：没有中央决策者。代理读取共享事件总线，异步领取工作，将结果写回。LangGraph 明确支持去中心化、动态环境的「集群架构（Swarm Architecture）」。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排者瓶颈。权衡是明确的：以确定性和可追溯性换取可扩展性。集群适合有许多独立子问题的任务；它不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`、`queue`）
**前置知识：** 第 16 阶段 · 05（监督者模式）、第 16 阶段 · 04（原语模型）
**时间：** 约 75 分钟

## 问题

监督者可以扩展到几个工作者。那成百上千个呢？监督者本身成为瓶颈：关于谁做什么的每个决策都通过一个代理。一个慢的计划步骤会停滞整个系统。

集群架构翻转了设计。不是中央规划者派发工作，而是工作者从共享队列中领取工作。「协调」内建于事件总线语义中。没有编排者；系统在队列能承受的范围内扩展。

## 概念

### 形状

```
                ┌──── 共享队列 ────┐
                │                  │
       ┌────────┼────────┐  ◄──┬───┘
       ▼        ▼        ▼     │
     工作者   工作者   工作者  工作者
       A       B       C       D
       │        │        │     │
       └────────┴────────┴─────┘
                 │
                 ▼
            结果池
```

没有编排者。每个工作者重复：拉取任务、处理、写入结果（可选地将后续任务入队）。

### 什么时候集群合适

- **许多独立任务。** 抓取、转换、分类。任务互不依赖。
- **可变时长的工作。** 如果某些任务需要 100 毫秒而其他需要 10 秒，集群自动平衡负载——快的工作者拉取下一个任务。监督者必须预测时长。
- **吞吐量优先于确定性。** 你关心总完成时间，而非严格排序。

### 什么时候集群失败

- **有顺序的工作流。** 如果步骤 3 需要步骤 2 的输出，集群可能让步骤 3 在步骤 2 完成之前触发。
- **全局计划任务。** 复杂的研究问题受益于规划者。一个研究者集群产生独立的事实，而非连贯的报告。
- **调试。** 没有中央日志和异步工作，复现 Bug 代价高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年将集群推向其自然结论的论文：控制流和数据流都是分布式队列上的序列化消息。没有中央协调者。容错来自消息持久性。可扩展性是消息代理的问题，而非系统的问题。

贡献：一种编程模型，其中多代理协调是「此代理订阅什么消息主题？」而不是「监督者选择哪个代理下一个？」这使系统看起来像一个发布/订阅事件网格。

### LangGraph 的集群架构

LangGraph 2025 年文档明确将「集群架构」描述为多代理模式之一：代理是节点，但边形成带有循环的有向图，任何节点都可以从池中被激活。工作者通过条件从可用工作中选取，而非通过监督者分配。

### 失败模式：饥饿与热点

如果所有工作者都拉取最快可用的任务，长时间运行的任务永远不会被选中，直到它们是唯一剩下的。经典的队列饥饿。

缓解措施：
- 带有显式老化（Aging）的优先级队列（等待时间增加优先级）。
- 工作者专业化：某些工作者只接受「长」任务。
- 反压（Back-Pressure）：限制进入队列的快速任务数量。

### 基于内容的路由链接

集群自然与基于内容的路由配对（第 22 课）。不是通用队列，而是每种消息类型一个队列。专业工作者只订阅它们的类型。这是能扩展到数千个代理的消息总线架构的基础。

## 构建

`code/main.py` 实现了一个 4 个工作线程的集群，从共享的 `queue.Queue` 拉取任务。任务具有可变时长（有些快、有些慢）。演示对比：

- **顺序基线：** 一个工作者串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定工作者（监督者风格）。
- **集群：** 工作者从共享队列拉取。

集群自动平衡负载；固定分配在分配任务慢时让快工作者空闲。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务计数（集群不均匀但最优地分配）和墙钟时间。

## 实践

`outputs/skill-swarm-fit.md` 评估任务应使用集群还是监督者。输入：任务独立性、时长方差、排序要求、可调试性需求。

## 交付

检查清单：

- **带老化的优先级队列。** 防止长任务饥饿。
- **工作者幂等性（Idempotency）。** 如果工作者在运行中崩溃，一个任务可能被拉取多次。工作者必须是幂等的。
- **持久队列。** 生产中使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 仅在内存中。
- **每个任务的可观测性。** 每个任务有一个追踪 ID；每个工作者记录开始/结束。
- **反压。** 如果队列增长速度超过工作者消耗速度，减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长工作负载上，集群比顺序快多少？比固定分配快多少？
2. 添加优先级队列变体（使用 `queue.PriorityQueue`）。按任务「重要性」字段分配优先级。观察低优先级任务在持续负载下是否会饿死。
3. 实现热点检测器：记录任何工作者处理的任务数是最慢工作者的 3 倍以上。这说明了任务时长分布的什么情况？
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。识别 Matrix 接受的一个具体权衡（可扩展性收益）和放弃的一个（可追溯性、确定性）。
5. 将集群演示转换为使用 `(task_type, payload)` 元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构时，什么路由规则有意义？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Swarm architecture | 「去中心化代理」 | 工作者从共享队列拉取；没有中央编排者。 |
| Event bus | 「代理订阅主题」 | 按类型或内容将任务路由到工作者的消息代理。 |
| Starvation | 「任务永不运行」 | 低优先级任务从不会被选中，因为更高优先级的工作持续到达。 |
| Hot-spotting | 「一个工作者被淹没」 | 负载不平衡，一个工作者获得大多数任务。 |
| Back-pressure | 「减慢生产者」 | 当队列填满时信号上游停止生产的机制。 |
| Idempotent worker | 「安全重跑」 | 处理两次的任务产生相同结果。因为工作者可能在运行中崩溃，所以需要。 |
| Durable queue | 「在崩溃中存活」 | 由磁盘或复制存储支持的队列；工作者崩溃时任务不丢失。 |
| Matrix framework | 「全消息传递集群」 | 数据和控制流都是分布式队列上的序列化消息。 |
