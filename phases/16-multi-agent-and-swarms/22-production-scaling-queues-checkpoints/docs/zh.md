# 生产级扩展——队列、检查点与持久化

> 将多代理系统扩展到数千并发运行需要**持久化执行（Durable Execution）**。LangGraph 的运行时在每个超级步骤（Super-step）之后写入一个检查点（Checkpoint），以 `thread_id` 为键（默认使用 Postgres）；工作进程崩溃后释放租约，另一个工作进程恢复执行。代理可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）为每个代理运行生产者-消费者队列，具有三种状态（空闲 / 处理中 / 响应）和两层协调（组内聊天 + 组间管理聊天）。**纤程（Fiber）/ async** 在 LLM 流式传输中优于每任务一线程：线程在等待 Token 时 99% 的时间空闲，纤程在 I/O 上协作式让出（Yield）。反面观点：Ashpreet Bedi 的"扩展代理软件"主张**FastAPI + Postgres + 别无其他**，直到负载证明不足——简单架构比预期走得更远。本课构建一个持久化检查点日志、一个带状态转换的每代理工作队列、一个 async vs 线程演示，并确立务实的"从简单开始"规则。

**类型：** 学习 + 构建
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置条件：** Phase 16 · 09（并行集群网络）、Phase 16 · 13（共享记忆）
**时间：** ~75 分钟

## 问题

原型多代理系统在一台笔记本电脑上运行，三个代理在内存事件循环中工作。你要上生产环境：

- 代理有时运行数小时（长时间研究、人在回路中等待）。
- 工作进程崩溃。重启会丢失状态。
- 峰值负载是平均负载的 10 倍；需要水平扩展。
- 用户按代理运行付费；需要精确一次（Exactly-once）语义来计费。

内存事件循环无法做到这些。你需要在底层有一个持久化执行层。2026 年的规范选项有：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每代理生产者-消费者）。
4. 手写 FastAPI + Postgres（Bedi 的论点）。

本课构建每个选项的微型版本。

## 概念

### 持久化执行：模式

持久化执行引擎在每个"步骤"之后持久化完整程序状态（在 LangGraph 的术语中称为超级步骤）。崩溃时：

```
工作进程在步骤中途崩溃
  -> 租约超时
  -> 另一个工作进程接管 thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这需要满足以下条件：

- **可序列化状态。** 所有代理状态必须可持久化。带有活跃数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同状态和相同输入，代理产生相同动作（或将 LLM 调用委托给外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须幂等或使用去重键。

LangGraph 在每个超级步骤后写入检查点；Temporal 在每个活动后写入；Restate 使用事件溯源日志。三者实现相同的模式。

### LangGraph 的运行时

每个代理有一个 `thread_id`；状态是类型化字典；每个超级步骤向检查点表写入一行。恢复时，运行时从最后一个检查点重放，而非从头开始。代理可以 `interrupt()` 等待人类输入；运行时持久化并释放工作进程。当输入到达时，任何工作进程都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每代理队列

arXiv:2408.09955 描述了一个规模实验：一个集群中数千个并发代理。架构：

```
代理 i:
  状态 ∈ {空闲, 处理中, 响应}
  入站队列   <- 发送给代理 i 的消息
  出站队列   -> 回复 + 副作用

协调者:
  组内聊天  （同一组中的代理）
  组间管理聊天  （高层路由）
```

两层协调使组内对话密集进行，而组间保持稀疏——这种模式用于在数千代理下保持成本线性。

### Async vs 每任务一线程

LLM 调用是 I/O 密集型的。等待下一个 Token 的线程 99% 的时间空闲。每个线程约占用 1MB RAM；10000 个并发调用仅栈就需要 10GB。

纤程（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作式让出。同样的 10000 次调用可以舒适地容纳在一个进程中。在 LLM 代理规模下，async 不是优化——它是架构。

例外：CPU 密集型的后处理（嵌入、分词器技巧）仍然需要线程或进程。将 I/O 层与 CPU 层分离。

### Bedi 的反面观点

"扩展代理软件"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度工程化了。务实的默认方案：

- FastAPI + Postgres。
- 每次代理运行是一行；状态通过乐观并发原地更新。
- 后台任务通过 `pg_notify` 或简单的 Celery 工作进程。
- 重试策略在应用代码中。

对于负载在约 100 个并发代理运行以下且任务可管理的情况下，这通常就是所需的一切。当你测量到它失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，再采用持久化执行框架。过早采用会耗费时间在不会产生回报的仪式上。

### 精确一次语义

对于付费代理运行，你需要"有效精确一次"（至少一次投递 + 幂等消费者）。工程手段：

- **每次运行的唯一去重键。** 在每个副作用调用中包含它。
- **发件箱模式（Outbox Pattern）。** 副作用先写入一个表，然后单独进程执行它们。两个步骤都幂等。
- **补偿事务（Compensating Transactions）。** 当副作用成功但其跟踪写入失败时，调度补偿操作。

这些是数据库工程模式，并非 LLM 特有的。LLM 的额外成本仅在于 LLM 调用慢；其他一切都是标准分布式系统。

### 彩虹部署（Rainbow Deployment）

Anthropic 的多代理研究系统使用"彩虹部署"：多个版本的代理运行时同时运行，这样长时间运行的代理不必在每次代码部署时被杀死。在新版本上对部分流量进行金丝雀（Canary）测试；当旧版本代理完成后再退役旧版本。

这对长时间运行的有状态系统是标准做法；2026 年的适配是代理可以存活数小时，因此部署周期必须适应。

### 规范的生产检查清单

- 持久化状态（检查点、快照，或发件箱 + 可重放日志）。
- 幂等副作用。
- 用于 LLM 调用的异步 I/O 层。
- 至少一次投递 + 去重。
- 针对有状态工作负载的彩虹/金丝雀部署。
- 可观测性：每代理追踪、超级步骤审计、重试计数器。

## 构建它

`code/main.py` 实现：

- `CheckpointStore` —— 基于 SQLite 的检查点日志，以 thread-id 为键。每个超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` —— 模拟运行中途崩溃；第二个工作进程从最后一个检查点恢复。
- `AgentQueue` —— 每代理的空闲 / 处理中 / 响应状态机，带有小型工作队列。
- `demo_async_vs_threads()` —— 通过 asyncio 和线程运行 500 个并发的模拟"LLM 调用"；报告墙钟时间和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；async 版本在 < 1 秒内处理 500 个并发调用；线程版本需要数秒，且每并发单元使用数量级更多的内存。

## 交付它

规范的生产加固：

- **从简单开始（Bedi 规则）。** FastAPI + Postgres，直到你测量到它失败。
- **在优化之前对所有内容进行仪表化。** 每次运行的延迟直方图、每步骤时间、重试次数、失败分类。
- **副作用使用发件箱模式。** 尤其是支付和外部 API 调用。
- **彩虹部署。** 绝不在部署期间杀死运行中的代理运行。
- **当你遇到以下具体问题时采用持久化执行引擎（Temporal / LangGraph / Restate）：** 长达数小时的人在回路中等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层使用 async。** 线程仅用于 CPU 密集型后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复工作正常；测量 async vs 线程的并发差异。
2. 实现一个**发件箱**表：每个工具调用先写入发件箱，然后由单独的协程/任务执行。通过运行工具调用两次验证幂等性。
3. 模拟一次**彩虹部署**：两个并发的运行时版本；将一半新的 thread_id 路由到每个版本；确认旧版本上正在运行的线程未被中断。
4. 阅读 LangGraph 的运行时文档（链接见下）。确定运行时中哪些功能在手写 FastAPI + Postgres 版本中复制最耗时。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是明确的。勾画如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| 持久化执行 | "持久化程序状态" | 引擎在每个超级步骤后写入状态；崩溃恢复是确定性的。 |
| 超级步骤 | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "代理运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性 | "可安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| 发件箱模式 | "解耦副作用" | 将意图写入表；单独的执行器执行并标记完成。 |
| 至少一次投递 | "可能重复" | 消息队列语义；去重键使消费者有效一次。 |
| 彩虹部署 | "重叠版本" | 长时间运行工作负载期间多个运行时版本并发。 |
| 异步纤程 | "协作式让出" | 用户态并发；对 I/O 密集型负载比线程便宜得多。 |
| 检查点 | "状态快照" | 超级步骤边界处的序列化状态；恢复的关键。 |
