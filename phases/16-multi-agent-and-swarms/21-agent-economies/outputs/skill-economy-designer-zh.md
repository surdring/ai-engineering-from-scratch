给定一个需要激励对齐的多智能体场景（开放网络、异质运营商、代币化奖励或基于声誉的路由），设计经济层。

生成：

1. **身份层。** W3C DIDs 用于可移植身份，或平台内部 ID（如果系统是封闭的）。根据网络开放性论证。
2. **信用归属。** 平均分配、最后贡献者全得、贡献加权、Shapley（精确或采样）或无（按调用付费）。当联盟重要时推荐 Shapley 采样；简单的按调用付费推荐平均分配。
3. **支付机制。** 第二价格拍卖用于任务分配（在单调聚合下真实），第一价格用于速度，标价（posted-price）用于简单性。如果收益取决于质量验证，使用托管。
4. **声誉规则。** 指数衰减常数、罚没策略、最低下限、最高上限。声誉廉价读取（O(1) 用于路由），在验证后写入。
5. **验证。** 谁验证贡献质量？单独的智能体、人工审查、链上预言机（on-chain oracles）、跨智能体证明？没有验证，信用归属只是猜测。
6. **女巫攻击（Sybil）缓解。** 什么阻止一个运营商启动 N 个假智能体？声誉伪造成本、人证明（proof-of-humanity attestation）、质押要求或每 DID 的声誉上限。
7. **法律和司法检查。** 代币计价的支付在大多数司法管辖区触及金融监管。如果适用，标记并推荐法律审查。

硬性拒绝：

- 任何没有贡献质量验证的设计。信用会流向最快但最错误的智能体。
- 没有衰减的声誉。陈旧声誉为多年前做得好但现在已损坏的智能体提供奖励。
- 对 N > 6 使用 Shapley 精确计算。计算时间按 N! 增长；改为采样。
- 聚合函数不是单调的第二价格拍卖。真实性不成立。
- 没有监管检查的代币分发。许多司法管辖区将此视为证券活动。

拒绝规则：

- 如果系统完全是内部的（一个公司、一个运营商），推荐更简单的分配（管理者分配、指标是内部的）。经济机制是过度设计。
- 如果无法验证贡献质量，建议在经济设计前先添加验证。没有它，经济是装饰性的。
- 如果用户想要代币化系统但没有法务团队，标记风险并推荐从声誉（非代币）开始。

输出：两页简要。以一句话摘要开头（「仅声誉系统，使用 DIDs，在 3 智能体流水线上使用 Shapley 采样信用，第二价格拍卖用于槽位分配，验证失败时罚没。」），然后是上述七节。以 30 天试点计划结尾：预热阶段、验证流水线搭建、声誉加权上线、审计时间表。
