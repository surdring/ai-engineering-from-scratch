给定一个适合并行子智能体研究的用户查询，生成一个监督者模式设计，准备好接入任何框架（LangGraph、OpenAI Agents SDK、CrewAI Hierarchical）。

生成：

1. **复杂度估算。** 此查询是简单（1 个智能体，3-10 次工具调用）、中等（2-4 个 worker）还是复杂（5+ 个 worker）？使用 Anthropic 的规模-努力启发式用一句话论证。
2. **主导系统提示。** 必须包含：(a) 分解指令，(b) 综合指令，(c) 主导绝不读取原始源内容、只读 worker 摘要的明确规则。
3. **Worker 系统提示。** 每个角色一个，各自命名其狭窄范围以及主导期望的输出格式。
4. **子问题分解规则。** 主导如何拆分查询？先广后窄，还是直接分解？什么会使子问题不合格（与另一个重叠、太宽泛）？
5. **综合模板。** 明确的冲突处理规则：如果两个 worker 返回矛盾的事实，综合必须呈现分歧，而非静默选择一个。
6. **模型配对。** 主导用什么模型（推理层），worker 用什么（更快/更便宜的层）。解释权衡。
7. **可观测性要求。** 最低追踪点：计划、每个 worker 开始/结束、综合输入、综合输出。

硬性拒绝：

- 主导自身使用工具的任何设计。主导只做计划和综合。
- 允许范围漂移的 worker 提示（例如无边界地「研究与 X 相关的任何内容」）。
- 隐藏冲突的综合模板。

拒绝规则：

- 如果查询简单（估计总共不到 10 次工具调用），拒绝设计并推荐单智能体。引用 Anthropic 15 倍令牌成本发现。
- 如果查询是顺序的（第 2 步需要第 1 步的输出），拒绝并推荐流水线/链模式。
- 如果用户优化的是确定性和审计，拒绝监督者并推荐 LangGraph 静态图。

输出：一页设计简要。以复杂度估算和模式匹配裁决（「监督者适合」）开头。以如果系统将持续运行时发出 rainbow-deployment 提醒结尾。
