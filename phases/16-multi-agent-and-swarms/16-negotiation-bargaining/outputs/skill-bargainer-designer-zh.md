给定一个谈判或任务市场场景（两方讨价还价、N 方拍卖、contract-net 任务分配），设计协议。

生成：

1. **机制。** 两方讨价还价、N 竞标者拍卖、contract-net 广播或多方联盟。命名博弈类型。
2. **报价生成器。** 确定性（Zeuthen 式让步、Rubinstein 均衡、简单线性计划）或 LLM 提示。默认：确定性，除非报价必须是定性结构（提案、角色分配）。
3. **叙述层。** LLM 贡献什么：人性化表述、说服战术、人设。明确指出 LLM 不决定什么。
4. **私有 vs 公共通道。** 推理轨迹如何保持在对方上下文之外。「私有草稿」+「公共消息」作为两个字段。根据 arXiv:2503.06416，这是不可协商的。
5. **轮次上限。** 两方最多 3-5 轮。无界不是选项；它奖励从众并鼓励情绪化报价。
6. **保留价和 BATNA 纪律。** 双方必须知道自己的保留价。如果对方试探，LLM 叙述者不能透露。根据此规则验证每条发出消息。
7. **成交率监控。** 此协议的预期基准成交率（引用谈判基准的数字：27%-89% 范围，取决于 LLM 角色）。回归告警阈值。
8. **升级。** 低于阈值轮次、ZOPA 违规或对方违规路由到调解智能体或人类。

硬性拒绝：

- 任何 LLM 在没有确定性后备的情况下计算数值报价的设计。arXiv:2402.15813 显示这产生约 27% 的成交率。
- 任何没有分离私有和公共通道的设计。对方会读到你的推理。
- 任何无界轮次的设计。保证产生从众驱动的结果。
- 让单个智能体同时持有买方和卖方状态的设计（角色扮演讨价还价）。私有信息属性是机制的核心；合并角色会移除它。

拒绝规则：

- 如果任务没有数值收益（定性谈判、合同条款），OG-Narrator 分解可能不适用。推荐结构化提案 + 模式验证。
- 如果用户无法实现单独的草稿（单 LLM 调用架构），显式标记泄露风险并推荐两调用架构。
- 如果谈判是敌对的，一方可能说谎，推荐调解智能体加记录报价用于审计。

输出：一页简要。以一句话摘要开头（「两方讨价还价：Zeuthen 报价生成器 + LLM 叙述者，5 轮上限，单独草稿，成交率低于 85% 告警。」），然后是上述八节。以示例消息结尾：对方看到的内容 vs 私有草稿持有的内容。
