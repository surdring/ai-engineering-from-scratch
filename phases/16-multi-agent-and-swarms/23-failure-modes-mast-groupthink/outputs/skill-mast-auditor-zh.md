给定一个多智能体系统和采样的执行轨迹，运行失败模式审计。

生成：

1. **样本构建。** 至少 200 条来自生产的轨迹，跨任务类型和时间窗口均匀采样。记录采样方法和偏差风险。
2. **分类遍历。** 对每条轨迹，标记 `success | failure`。对失败，分配一个 MAST 类别（spec / coord / verify），以及适用时一个或多个群体思维（Groupthink）家族标签（monoculture / conformity / tom / mixed-motive / cascade）。
3. **分布表。** 按 MAST 类别和群体思维标签的计数和百分比。与 Cemri 2025 的参考分布（41.77 / 36.94 / 21.30）比较。严重偏离参考的系统通常有特定的薄弱层。
4. **主要失败模式。** 识别 3 种最频繁的特定模式（例如「两个智能体都进行审查」）。记录复现步骤。
5. **缓解排名。** 对每种主要模式，从标准库提出缓解措施：显式角色合约、版本化共享状态、独立验证器、熔断器（circuit breaker）、检测-诊断-验证（STRATUS）三重奏。按模式频率的预期失败减少率排名。
6. **静默失败风险。** 有多少失败产生看似合理但错误的输出 vs 明显的错误？静默率驱动验证层投资。
7. **慢失败代理指标。** 推荐 2-3 个能在漂移变成明显错误之前暴露的实时指标：一致率、重试率、输出长度分布、智能体间编辑距离。

硬性拒绝：

- 没有随机或分层样本的审计。手动挑选的失败过度代表戏剧性案例，遗漏慢失败漂移。
- 没有基线测量的缓解建议。「添加验证器」在当前失败率未知时毫无意义。
- 忽略 MAST-未知事件。如果轨迹不适合任何类别，分类不完整；建议扩展而非强制归类。
- 声称季度审计足够而没有运营性慢失败监控。季度审计遗漏审计之间的漂移。

拒绝规则：

- 如果轨迹缺少每智能体归因（谁写了什么、谁读了什么），审计无法区分协调失败和角色冲突。建议在重新审计前添加结构化每智能体日志。
- 如果系统总共少于 50 条失败轨迹，样本太小，无法产生分布估计。建议更长的观察窗口。
- 如果轨迹包含 PII，在分析前脱敏。

输出：三页报告。以一句话摘要开头（「41% 规范失败，12% 协调失败，39% 验证缺口，8% 未知；主要模式是双审查者冲突；最高 ROI 缓解是显式角色合约。」），然后是上述七节。以优先级行动列表结尾：三项缓解措施，附带估计实现成本和预期失败率降低。
