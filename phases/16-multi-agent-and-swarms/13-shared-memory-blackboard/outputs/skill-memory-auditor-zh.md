给定一个多智能体代码库或架构文档，审计共享记忆设计并标记记忆污染暴露。

生成：

1. **拓扑。** 完整消息池、按主题分区的黑板、投影到每个智能体的视图，还是混合模式？指出数据结构（list、dict、pandas frame、向量存储、SQL 表）。估算稳态下写者和读者的粗略上限。
2. **溯源字段。** 每次写入时，条目是否记录：写者 ID、时间戳、提示哈希或提示文本、工具调用追踪、源 URI 或工具名称？列出存在的字段和缺失的字段。
3. **更新模型。** 日志是仅追加，还是写者原地修改？如果是修改，并发控制机制是什么（锁、乐观版本控制、无）？修正应为替代条目，而非原地编辑——标记任何不这样做的设计。
4. **验证器隔离。** 是否有一个具有独立源访问权限的只读智能体？它可以写入主池吗（不应该）？其输出去哪里？
5. **投影模式。** 如果设计使用投影（LangGraph reducers、黑板主题、角色范围视图），模式是否文档化？新智能体如何声明它们消费的投影？
6. **污染风险评分。** 在每个轴上评分 1-5：[溯源完整性]、[替代优于修改]、[验证器独立性]、[投影模式清晰度]。任何轴上得分低于 3 的系统被标记。

硬性拒绝：

- 任何未标记缺失验证器的审计。具有独立源访问权限的不可写验证器是承重缓解措施；没有它，所有其他缓解措施都是装饰性的。
- 推荐「增加更多测试」的审计。测试无法捕获记忆污染，因为污染产生通过测试的看似合理的输出。
- 推荐将内容哈希作为唯一溯源的审计。哈希告诉你写入了*什么*，而非*谁*或*从哪里来*。

拒绝规则：

- 如果代码库将共享状态隐藏在外部服务中（Redis、Postgres、向量数据库），且没有检查工具，声明没有生产读访问权限无法完成审计。
- 如果系统智能体少于三个，注意记忆污染风险低，但溯源仍然是廉价的保险。
- 如果系统使用具有内置状态管理的框架（LangGraph checkpointer、AutoGen pool），审计框架的保证而非重新推导它们。

输出：两页报告。以一句话摘要开头（「共享状态是一个完整消息池，没有溯源，没有验证器——高污染风险。」），然后是上述六节。以优先级行动列表结尾：三项变更，每项标记为 [关键] [应当] 或 [锦上添花]，附带预计实现时间。
