# 层次化架构及其失败模式

> 层次化是嵌套的监督者。管理者代理之下是子管理者，子管理者之下是工作者。CrewAI 的 `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态地委托任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是真实的组织结构图时，这是自然的模式。这也是最可能陷入管理循环的模式——管理者代理分配工作不当、误解子输出或无法达成共识。顺序执行通常胜过它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第 16 阶段 · 05（监督者模式）
**时间：** 约 60 分钟

## 问题

一旦监督者模式理解了，自然的下一步是「如果工作者自身就是监督者会怎样？」团队有子团队；公司有部门的部门。层次化架构反映了这一点。

问题：LLM 管理者与人管理者不同。人管理者对自己的下属知道什么有稳定的先验。LLM 管理者每一轮都从其上下文中重新推理组织。该上下文中的微小漂移，整棵树就会错误分配工作。

## 概念

### 形状

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理者 A        子管理者 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点规划、委托和综合。只有叶子节点做实际工作。

### 哪里它闪耀

- **清晰的组织映射。** 如果真实任务是部门化的（「法务审查文档，财务审查文档，工程审查文档，然后为高管总结」），层次结构是显式的。
- **局部摘要。** 每个子管理者在顶层管理者看到之前综合其团队的输出。顶层管理者看到三个子管理者摘要，而非十五个工作者输出。

### 哪里它崩溃

2026 年事后分析持续发现的三种失败模式：

1. **任务分配错误。** 管理者阅读目标，幻觉出分解，并委托给错误的子管理者。因为子管理者顺从地处理分配给它的内容，错误只在顶层综合时才浮现——比人类本可以捕获的地方远离一个层级。
2. **输出误解。** 子管理者返回「无法验证声明 X」。顶层管理者总结为「声明 X 未确认」。含义在每个层级漂移。
3. **共识循环。** 两个子管理者意见分歧；顶层管理者让他们协调；他们重新向下委托；工作者重新运行；子管理者返回稍微不同的答案；循环。CrewAI 的 `Process.hierarchical` 通过步骤限制来防范这一点，但限制本身现在是一个超参数。

### 决定性问题

顺序（线性管道）vs 层次化：你的任务是否真的有独立的子团队，还是它是一个伪装成树的线性流程？如果是后者，使用顺序。如果是前者，使用层次化但要预算明确的协调规则。

### CrewAI 的实现

`Process.hierarchical` 在专业团队之上连接了一个管理者 LLM。管理者：

- 接收顶层任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定接受、重新委托还是迭代。

文档：https://docs.crewai.com/en/introduction（查找「Hierarchical Process」，在核心概念下）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这在调试方面比 CrewAI 更清晰（你可以单独逐步执行每个图），但更难表达动态重塑树结构。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 构建

`code/main.py` 运行一个 3 层层次结构：

- 顶层管理者：将任务拆分为「工程」和「法务」分支，
- 工程子管理者：拆分为「前端」和「后端」工作者，
- 法务子管理者：一个工作者。

演示对比**正常路径**（每个人都同意）与**扰动路径**——顶层管理者的分解错误地将「法务」标记为「财务」，观察错误如何级联——子管理者顺从地做财务工作，顶层综合者报告财务发现，原始的法务问题无人回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，并排清晰展示「要求了什么」vs「交付了什么」。

## 实践

`outputs/skill-hierarchy-fitness.md` 评估给定任务是应该使用层次化、顺序还是扁平监督者。输入：任务描述、组织结构、协调预算。输出：模式推荐以及需要防范的具体失败模式。

## 交付

如果你交付层次化：

- **树深度上限为 2。** 三级已经隐藏了大多数错误，使其无法观测。
- **明确的协调预算。** 设置顶层管理者必须提交之前的最大轮数。通常是 2。
- **每个综合的来源追溯。** 每个节点的摘要必须引用产生它的叶子输出。
- **分解漂移警报。** 记录每步管理者的分解；与用户查询进行对比。如果分解不再覆盖查询，触发警报。

## 练习

1. 运行 `code/main.py` 并比较正常 vs 扰动路径。需要多少层管理者交接才能使顶层输出与用户问题完全背离？
2. 添加第三层（顶层 → 子层 → 子子层 → 工作者）。测量随着深度增长，扰动路径自我纠正 vs 完全背离的频率。
3. 在每个子管理者处实现一个「金丝雀」工作者，始终被问及原始用户问题不变。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不一致时，管理者应如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体护栏（步骤限制、manager_llm 约束），并描述它针对什么失败模式。
5. 比较嵌套 LangGraph 监督者与 CrewAI 层次化。哪种让协调循环的检测更便宜？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Hierarchical | 「组织结构图模式」 | 监督者之上的监督者；只有叶子做工作。 |
| Manager LLM | 「老板」 | 在内部节点分解、分配和验证的 LLM。 |
| Decomposition drift | 「老板偏离了主题」 | 顶层管理者的拆分不再涵盖原始问题。 |
| Reconciliation loop | 「无尽会议」 | 子管理者意见分歧；顶层重新委托；工作者重新运行；循环直到预算耗尽。 |
| Depth-2 ceiling | 「不要超过 2 层深度」 | 经验护栏：3 层以上使可观测性崩溃。 |
| Canary question | 「每层的基准真相」 | 一个始终被问及原始查询不变的工作者，用于检测漂移。 |
| Provenance chain | 「谁说了什么」 | 从每个综合追溯到产生它的叶子输出。 |
