给定一个提议的长时间运行智能体部署（Temporal + OpenAI Agents SDK、带 PostgreSQL 检查点的 LangGraph、Microsoft Agent Framework、Claude Code Routines、Cloudflare Durable Objects 或自有等价方案），对照持久执行（durable execution）模式审计设计。

生成：

1. **活动清单。** 列出每个活动（LLM 调用、工具调用、HTTP 请求、文件写入）。确认每个都包装为具有重试策略、超时和幂等键的活动。活动封装之外的原始 LLM 调用是可靠性漏洞。
2. **工作流确定性。** 识别工作流代码中的每个非确定性读取（挂钟时间、随机数、外部状态）。每个都必须注册为副作用活动，以便重放返回相同的值。隐藏的非确定性是重放漂移最常见的原因。
3. **检查点后端。** 命名后端（PostgreSQL、SQLite、Redis、Durable Objects）。确认它在部署后能存活。SQLite 仅用于开发。Redis 需要 AOF 或快照配置。Cloudflare Durable Objects 是透明的，但需要唯一的键规范。
4. **人工输入状态。** 确认 HITL 暂停是一级工作流状态，而非轮询循环。工作流应阻塞在外部信号（审批队列、webhook、`interrupt()` 原语）上，在审批到达时恰好恢复。
5. **恢复时 HITL 策略。** 对于崩溃后的任何恢复，说明在执行下一个活动之前是否需要重新 HITL。没有这一点，持久执行加上崩溃前授予的审批可能在上下文已变化时重新触发已批准的动作。对于长时间跨度至关重要。

硬性拒绝：
- Agent SDK 使用中 LLM 调用未被包装为活动。
- 部署后无法存活的检查点后端。
- 嵌入挂钟时间或随机数而未进行活动包装的工作流。
- 建模为轮询循环而非信号的人工输入。
- 超过一小时且无恢复时 HITL 策略的长时间跨度运行。
- 在持久性之上没有叠加预算紧急停止开关（第 13 课）的运行。

拒绝规则：
- 如果用户提议的持久工作流没有对副作用活动显式的幂等性，拒绝并要求先提供幂等键。否则重试会导致双重执行。
- 如果用户无法展示重放测试（运行工作流、中途崩溃、重放、断言无双重副作用），拒绝并要求在生产前完成该测试。
- 如果用户提议 24 小时无人值守运行且无 HITL 检查点，拒绝。35 分钟退化（第 12 课笔记）使这在持久性正确的情况下仍然是可靠性问题。

输出格式：

返回设计审查备忘录，包含：
- **活动表格**（活动、重试策略、超时、幂等键）
- **确定性审计**（非确定性读取及各自的处理方式）
- **检查点后端**（名称、部署存活、重放测试状态）
- **HITL 状态形态**（一级状态 / 轮询 / 缺失）
- **恢复时 HITL 策略**（明确，附理由）
- **就绪状态**（生产 / 测试 / 仅研究）
