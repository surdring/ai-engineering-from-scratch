# 长时间运行的后台代理：持久执行

> 生产长周期代理不在 `while True` 中运行。每个 LLM 调用变成一个带有检查点、重试和重放的活动。Temporal 的 OpenAI Agents SDK 集成于 2026 年 3 月 GA。Claude Code Routines（Anthropic）在没有持久本地进程的情况下运行计划的 Claude Code 调用。会话在人工输入上暂停，在部署间幸存，并从以 `thread_id` 为键的最新检查点恢复。新的人体工程学背后坐着一个旧模式——工作流编排——带有一个新输入：LLM 调用作为非确定性活动，必须在恢复时确定性重放。

**类型：** 学习
**语言：** Python（标准库，最小持久执行状态机）
**前置条件：** Phase 15 · 10（权限模式），Phase 15 · 01（长周期代理）
**时间：** ~60 分钟

## 问题

考虑一个运行四小时的代理。它调用三个工具，提示用户两次，进行四十次 LLM 调用。在中途，它运行的主机重启。发生了什么？

- 在朴素的 `while True` 循环中：一切丢失。运行从头重新开始。三个工具调用（带有真实副作用）再次执行。用户再次被提示已批准的事情。四十次 LLM 调用被重新计费。
- 使用持久执行：运行从最近的检查点恢复。已完成的活动不重新执行；其结果从持久日志重放。用户不重新批准已批准的事情。已进行的 LLM 调用不重新计费。

这与工作流引擎已经交付了十年的模式相同（Temporal、Cadence、Uber 的 Cherami）。新的是 LLM 调用现在是一种活动——非确定性、昂贵、带有副作用——并且它们干净地适合此模式。

本课的主题贯穿：长周期可靠性衰减（METR 观察到"35 分钟退化"——成功率大约与时间周期二次方下降）。持久执行使得比可靠性剖面支持的更长的运行成为可能，如果设计正确则是安全失败的新方式，如果设计错误则是不安全的新方式。

## 概念

### 活动、工作流与重放

- **工作流（Workflow）**：确定性的编排代码。定义活动序列、分支、等待。必须是确定性的，以便可以从事件日志重放而不产生令人惊讶的偏离。
- **活动（Activity）**：非确定性、可能失败的工作单元。LLM 调用、工具调用、文件写入、HTTP 请求。每个活动记录其输入和（一旦完成）其输出。
- **事件日志（Event Log）**：持久后备存储。每次活动启动、完成、失败、重试以及每次工作流决策被记录。
- **重放（Replay）**：恢复时，工作流代码从头重新运行；每个已完成的活动返回其记录的日志结果而不重新执行。只有尚未完成的活动才实际运行。

这与 React 针对虚拟 DOM 重新渲染，或 Git 从提交重建工作树的形状相同。编排器中的确定性是使持久性廉价的原因。

### 为什么 LLM 调用适合该模式

LLM 调用是：
- 非确定性的（temperature > 0；即使 temperature 0 也跨模型版本漂移）。
- 昂贵的（金钱和延迟）。
- 可能失败的（速率限制、超时）。
- 有副作用（如果它们调用工具）。

这正是活动的特征。将每个 LLM 调用包装为活动给你指数退避重试、跨重启的检查点以及可重放的调试追踪。

### 以 `thread_id` 为键的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects 和 Claude Code Routines 都收敛到相同的 API 形状：一个 `thread_id`（或等价物）标识会话；每个状态转换持久化到后端（默认 PostgreSQL，开发用 SQLite，缓存用 Redis）；恢复读取最新检查点。

后端选择很重要：

- **PostgreSQL**：持久、可查询、在部署间幸存。LangGraph 的默认选择。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快但是短暂性的，除非配置 AOF/快照。
- **Cloudflare Durable Objects**：透明分布式；按唯一键限定范围；存活数小时到数周。

### 人工输入作为一等状态

提议-然后-提交（第 15 课）需要一个持久的"等待人工"状态。工作流暂停，外部队列持有待处理请求，批准从确切位置恢复。没有持久性这是尽力而为；有了它，过夜的批准到达，工作流在早上继续。

### 35 分钟退化

METR 观察到每个被测量的代理类别在连续运行约 35 分钟后都展示可靠性衰减。将任务持续时间翻倍大约将失败率翻四倍。持久执行不修复这一点；它让你运行超过可靠性剖面支持的时间。安全模式是结合持久性与需要重新进入时新的 HITL 的检查点，以及限制总计算（无论墙钟时间）的预算终止开关（第 13 课）。

### 持久执行是错误答案的场景

- 短于几分钟的运行，没有人工输入。开销 > 收益。
- 严格的只读信息检索。
- 在一个上下文窗口内需要端到端正确性的任务（某些推理任务；某些一次性生成）。

## 使用场景

`code/main.py` 在标准库 Python 中实现最小持久执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到 JSON 事件日志。
- 排序活动的工作流函数。
- `run_or_replay(workflow, event_log)` 函数，重放已完成的活动而不重新执行它们。

驱动程序模拟一个三活动工作流，在中途崩溃，并展示 (a) 朴素重试重新执行一切 vs (b) 重放只运行缺失的活动。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| 工作流（Workflow） | "代理的脚本" | 确定性编排代码；可从事件日志重放 |
| 活动（Activity） | "一个步骤" | 非确定性单元（LLM 调用、工具调用）；前后记录日志 |
| 事件日志（Event Log） | "后备存储" | 每次状态转换的持久记录 |
| 重放（Replay） | "恢复" | 重新运行工作流；已完成的活动返回记录的日志结果而不重新执行 |
| 检查点（Checkpoint） | "保存点" | 以 thread_id 为键的持久化状态；恢复时最新获胜 |
| thread_id | "会话键" | 限定持久状态范围的标识符 |
| 35 分钟退化 | "可靠性衰减" | METR：成功率约与时间周期二次方下降 |
| 非确定性（Non-Determinism） | "重放时的偏离" | 挂钟时间、随机数、LLM 输出；必须注册为副作用 |
