# AI Scientist v2——Workshop 级别自主研究

> Sakana 的 AI Scientist v2（Yamada 等人，arXiv:2504.08066）运行完整的研究循环：假设、代码、实验、图表、写作、投稿。它是首个生成的论文通过 ICLR 2025 workshop 同行评审的系统。独立评估（Beel 等人）发现 42% 的实验因编码错误失败，文献回顾经常将已确立的概念错误标注为新概念。Sakana 自己的文档警告代码库执行 LLM 编写的代码，建议 Docker 隔离。这幅图景的两半都是重点。

**类型：** 学习
**语言：** Python（标准库，研究循环状态机玩具）
**前置条件：** Phase 15 · 03（AlphaEvolve），Phase 15 · 04（DGM）
**时间：** ~60 分钟

## 问题

研究是一个开放式任务。与 AlphaEvolve 的算法搜索或 DGM 的基准受限自我修改不同，研究结果没有机器可检查的正确性标准。论文由审稿人评判，而非单元测试。这使得循环更难闭合——而如果闭合则更有价值，因为研究是复利进展所在之处。

AI Scientist v1（Sakana，2024）通过从人工编写的模板开始来闭合循环。LLM 在固定脚手架内填充实验。AI Scientist v2（Yamada 等人，2025）通过使用带有视觉语言模型（Vision-Language Model，VLM）评判循环的 Agent 树搜索，移除了模板要求。系统生成想法、实现实验、生成图表、撰写论文，并对审稿人反馈进行迭代。

同行评审裁决：一篇 v2 生成的论文在 ICLR 2025 workshop 被接收（有披露）。独立评估裁决：系统远不可靠。两者都是真实的。

## 概念

### 架构

1. **想法生成。** LLM 基于主题和先前文献提出研究想法。v1 使用模板；v2 使用在假设空间上的 Agent 搜索。
2. **新颖性检查。** 文献检索步骤检查该想法是否已发表。Beel 等人的评估在此步骤发现了错误标注——已确立的方法经常被归类为新方法。
3. **实验计划。** 代理起草实验协议并编写代码。
4. **执行。** 代码在沙箱中运行。失败被反馈回重试循环。在 Beel 等人的测量中，42% 的实验在此阶段因编码错误失败。
5. **图表生成。** 视觉语言模型读取生成的图表并重写以提高清晰度。这是 v2 的关键技术新增。
6. **写作。** LLM 起草论文，与内部审稿人迭代。
7. **可选：投稿。** 论文被投稿至会议。

### Workshop 接收结果意味着什么

一篇 v2 生成的论文在 ICLR 2025 workshop 通过了同行评审。作者向程序委员会披露了论文的起源。此次接收是一个数据点；它不是声称系统"能做研究"的许可证。

重要背景：workshop 论文比主会论文门槛低。同行评审有噪声；每天有一小部分投稿被接收。一次成功是概念验证，不是可靠性声明。Nature 2026 论文记录了端到端循环，本身由人类研究员共同撰写；它不是"系统写了一篇 Nature 论文"。

### 独立评估发现了什么

Beel 等人（arXiv:2502.14297）进行了外部评估。头条发现：

- **实验失败。** 42% 的实验因编码错误失败（错误导入、形状不匹配、未定义变量）。重试循环捕获了一部分，不是全部。
- **新颖性错误标注。** 文献检索步骤经常将已确立的概念标记为新概念。这是幻觉的研究等价物。
- **展示质量差距。** 视觉语言图表评判生成了发表级别的可视化，掩盖了底层的实验弱点。

最后一项发现对本阶段很重要。一个产生令人信服输出而非做令人信服研究的系统，比明显失败的系统更危险，而不是更安全。评估必须触及底层的声明，而不止步于图表。

### 沙箱逃逸问题

Sakana 自己的仓库 README 警告：

> 由于此软件执行 LLM 生成的代码的特性，我们无法保证安全。存在危险包、不受控的网络访问和意外进程生成的风险。使用风险自负，考虑 Docker 隔离。

这是未验证领域中自主性的操作性形态。LLM 编写代码；代码运行；代码可以做进程被允许做的任何事。没有对文件系统、网络和进程操作进行硬限制的沙箱，任何自主研究代理都可能外泄数据、浪费计算资源或重写自身。

AlphaEvolve 的沙箱故事更容易，因为其评估器是紧密的。AI Scientist v2 的循环运行具有开放式目标的开放式代码。这就是为什么它需要更强的隔离（最低 Docker；推荐 seccomp / gVisor）以及在每个输出离开系统之前的人工审查。

### v2 在前沿技术栈中的位置

| 系统 | 目标 | 输出类型 | 评估器 | 已知失败 |
|------|------|---------|--------|---------|
| AlphaEvolve | 算法 | 代码 | 单元 + 基准测试 | 受限于评估器严谨性 |
| DGM | 代理脚手架 | 代码 | SWE-bench | 奖励黑客 |
| AI Scientist v2 | 研究论文 | 文本 + 代码 + 图表 | 同行评审（弱） | 实验失败、错误标注、打磨掩盖弱点 |

v2 在三个系统中拥有最弱的自动评估器、最广的输出表面和最短的到达公共产物的路径。操作性控制（沙箱、审查、披露）承担了大部分安全工作。

## 使用场景

`code/main.py` 将 v2 循环模拟为状态机：想法 → 新颖性检查 → 实验 → 图表 → 写作 → 评审 → 接收或迭代。每个状态有从 Beel 等人发现中抽取的可配置失败概率。运行模拟器 N 个循环并计数：

- 多少想法到达投稿阶段。
- 多少投稿会有美化后的论文隐藏的关键实验缺陷。
- 重试预算如何在质量和产出率之间权衡。

## 部署

`outputs/skill-ai-scientist-sandbox-review.md` 是任何研究循环代理产物在离开沙箱前必须通过的双门控审查检查清单。

## 练习

1. 以默认参数运行 `code/main.py`。什么比例的循环运行产生"干净"的论文？什么比例的论文带有图表评判打磨过的实验失败缺陷？

2. 默认值已使用 Beel 等人的 42% / 25%。分别重新运行 `--experiment-failure 0.20 --novelty-mislabel 0.10` 和 `--experiment-failure 0.60 --novelty-mislabel 0.40`。打磨但有缺陷的份额在两次运行之间如何变化？

3. 阅读 Sakana 的 AI Scientist v2 仓库 README 中关于沙箱要求的内容。指出两项（除 Docker 外）你会为多天自主运行应用的额外限制。

4. 阅读 Beel 等人第 4 节关于展示质量差距的内容。设计一个额外的评估器，捕获看起来打磨良好但实验上有缺陷的论文。

5. 提出一个面向研究代理输出的人工审查协议，比"一个博士读每篇论文"更好地扩展。识别瓶颈并围绕它设计。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| AI Scientist v1 | "Sakana 的模板化研究代理" | 在固定脚手架中填充实验 |
| AI Scientist v2 | "无模板研究代理" | 带有 VLM 图表评判的 Agent 树搜索 |
| Agent 树搜索（Agentic Tree Search） | "分支研究代理" | 并行扩展多个实验计划；由内部评判剪枝 |
| 视觉语言评判（Vision-Language Critique） | "图表上的 VLM 打磨" | 多模态模型读取图表并重写以提高清晰度 |
| 文献检索（Literature Retrieval） | "新颖性检查" | 搜索先前工作以确认想法新颖性——有记录的错误标注 |
| 打磨掩盖（Polish Masking） | "漂亮的论文，破碎的研究" | 展示质量超过实验质量；隐藏弱点 |
| 沙箱逃逸（Sandbox Escape） | "LLM 代码突破" | 代理执行的代码做了循环设计者未意图的事 |
