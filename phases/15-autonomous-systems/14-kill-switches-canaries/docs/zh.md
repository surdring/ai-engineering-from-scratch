# 终止开关、熔断器与金丝雀 Token

> 终止开关（Kill Switch）是一个在代理编辑表面之外持有的布尔值——一个 Redis 键、一个功能开关、一个签名配置——完全禁用代理。熔断器（Circuit Breaker）更细粒度：它在特定模式（连续五次相同工具调用）上跳闸，暂停问题路径，并升级到人类。金丝雀 Token（Canary Token）继承自经典欺骗：一个代理没有正当理由触碰的假凭证或蜜罐记录，其访问触发警报。基于 eBPF 的数据路径（例如 Cilium）可以在内核层将隔离后 Pod 的出口流量重写到法庭取证蜜罐；已发表的 Cilium 基准测试报告在负载下 P99 数据路径延迟低于毫秒（你的传播预算取决于策略更新如何到达节点，而非数据路径本身）。适应移动基线的统计检测器（EWMA、CUSUM）将静默接受偏移——用不会弯曲的硬宪法限制来分层叠加它们。

**类型：** 学习
**语言：** Python（标准库，三检测器模拟器：终止开关、熔断器、金丝雀）
**前置条件：** Phase 15 · 13（成本治理器），Phase 15 · 10（权限模式）
**时间：** ~60 分钟

## 问题

成本治理器（第 13 课）限制了代理可以花费什么。它们不限制代理在预算内可以做什么。一个拥有 $50 速度限制的代理仍然可以外泄一个秘密、发布错误的帖子或删除一个资源——昂贵的操作在 Token 上往往是便宜的。

本课涵盖与成本层并列的三个检测器：

1. **终止开关**：在代理触及范围之外的布尔关闭按钮。
2. **熔断器**：暂停特定路径的操作模式检测器。
3. **金丝雀 Token**：代理没有正当理由触碰的诱饵，触碰即暴露自身。

三者都是 LLM 之前的工程。经典欺骗、速率限制断路器和功能开关终止早于自主代理。新的是攻击面：代理读取不可信内容（第 11 课），编辑自己的记忆，并可以将许多看起来安全的操作组合成一个不安全的操作。这里命名的检测器之所以有效，是因为它们不信任代理的自我报告。

## 概念

### 终止开关

终止开关是代理读取但不能写入的布尔值。实现选择：

- **托管服务中的功能开关。** LaunchDarkly、Statsig、Unleash。更新在数秒内传播。
- **代理轮询的 Redis 键。** 简单；需要代理进程在每一轮检查。
- **对象存储中的签名配置。** 代理在启动时验证签名；拒绝未签名状态。
- **OS 级别信号或容器生命周期终止。** Docker `kill`、Kubernetes `kubectl delete pod`、systemd stop。

正确终止开关具有的属性：

- 代理不能将其设为 `off`。（存在于代理凭证不能写入的系统中。）
- 在每个后果性操作上检查，而不仅是启动时。
- 当它是 off 时，代理不做任何外部可观察的事情，包括向代理可达的系统写日志。
- 重新启用是显式的人工操作，而非自动超时。

### 熔断器

熔断器暂停特定模式，而非整个代理。经典形状（来自 2007 年 Nygard 的书，仍然时效）：

- **闭合（Closed）**：操作被允许。
- **断开（Open）**：操作被阻止。
- **半开（Half-Open）**：冷却后，允许 1-3 次探测尝试（默认 1）；成功闭合断路器，任何剩余失败重新断开。

代理相关的触发器：

- 连续五次相同工具调用（重复循环）。
- 同一工具以不同输入连续五次失败（系统性故障）。
- 工具调用速率高于阈值（第 13 课速度限制）。
- 在不可信内容读取后调用特定工具（例如 `file.delete`）（第 11 课）。

### 金丝雀 Token

金丝雀 Token（也称为 honeytokens）是代理永远不应触碰的输入。访问就是警报。

代理的例子：

- 环境中一个没有实际权限的假 `AWS_SECRET_ACCESS_KEY`。外泄尝试即时可追溯。
- 一个标记有已知哨兵值的假数据库记录。该记录的任何读取或更新触发警报。
- 工作空间中的假文件。该文件的读取或编辑是可怀疑的。
- 会话开始时注入的假记忆条目。如果它在后续输出中出现，记忆已被外泄。

金丝雀设计特定于工作流。代理有正当理由触碰的金丝雀是噪声。

### 为什么分层统计和硬限制

统计检测器（EWMA、CUSUM、工具调用速率的 z-score）适应移动基线。这对于正当偏移是特性，对于对抗性偏移是 bug。耐心的攻击者可以将基线移动得足够慢，使任何警报都不触发。

硬宪法限制（第 17 课）不会弯曲。"永不发送邮件给 `secrets@`"作为宪法规则更简单、可审计，且不能通过缓慢偏移来游戏。分层模式：统计检测器捕获大多数噪声，硬限制捕获绕过检测器的攻击。

### 通过 eBPF 数据路径重定向进行隔离

Cilium 基于 eBPF 的网络策略可以在内核层将隔离后 Pod 的出口流量重写到法庭取证蜜罐。Cilium 公开的基准测试在 TCP_RR 上报告 P99 数据路径延迟低于毫秒；在集群中*触发*隔离的墙钟时间预算由策略更新如何从你的控制平面传播到节点主导（在 Kubernetes 上预加载操作器，数百毫秒是现实的；尾部延迟取决于集群大小和协调频率）。这给你两件事：代理不泄露任何真实东西，代理在蜜罐内的行为被完全观察。与金丝雀 Token 相同的模式，不同的层面：代理通过其在被遏制时试图做什么来揭示意图。

### 没有检测器能捕获什么

- **由允许原语组合的语义攻击。** 每个单独操作通过每个检测器；组合才是问题。这是第 10 课的 Auto Mode 组合失败。
- **将检测器作为负载一部分的攻击。** 如果代理读取一个写着"在行动前禁用金丝雀"的页面，且代理具有该能力，那么检测器就被损害了。金丝雀 Token 应该在代理无法修改的系统中。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| 终止开关（Kill Switch） | "关闭按钮" | 代理编辑表面之外的布尔值；在每个后果性操作上检查 |
| 熔断器（Circuit Breaker） | "模式暂停" | 基于重复、失败率或速率限制的特定操作跳闸 |
| 金丝雀 Token（Canary Token） | "蜜罐 Token" | 代理没有正当理由触碰的诱饵；访问触发警报 |
| 蜜罐（Honeypot） | "法庭取证沙箱" | 被隔离代理的重定向流量/工作空间，在此观察代理 |
| EWMA | "移动平均" | 指数加权；适应偏移（特性 + bug） |
| CUSUM | "累积和" | 检测从基线的持续偏移 |
| 硬限制（Hard Limit） | "宪法规则" | 不适应；无论历史如何保持恒定 |
| 宪法限制（Constitutional Limit） | "始终为真的规则" | 关联第 17 课的宪法；不能被代理编辑 |
