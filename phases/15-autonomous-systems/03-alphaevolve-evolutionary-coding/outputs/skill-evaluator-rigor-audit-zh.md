给定一个提议的进化编码循环（生成器 LLM、程序数据库、评估器），审计评估器。评估器是架构；生成器是可替换的。此技能决定循环是否有机会产生真正的收益，还是只会产生奖励破解的垃圾。

生成：

1. **评估器分解。** 列出评估器报告的每个信号：正确性、性能、资源、其他。对每个信号说明 (a) 如何测量，(b) 最便宜的作弊方式，(c) 留出输入的规则是什么样的。
2. **虚构分析。** 列出 LLM 在该领域最可能的三种虚构（claimed complexity classes、边缘情况的声称正确性、未经测量的声称性能）。说明哪个评估器信号能捕获每一项。
3. **奖励破解分析。** 列出循环在不完成预期任务的情况下最大化得分的三种可能方式（通过测试的捷径、代理作弊、输入的记忆化）。说明每种的缓解措施。
4. **确定性和可复现性。** 要求评估器输出在容差范围内是确定性的。标记任何得分在运行间波动超过种群方差的评估器。
5. **部署检查。** 如果获胜变体将被交付到生产环境，要求进行一项评估器不检查的独立部署前审查（安全性、成本、人工审查）。搜索未验证部署就绪状态。

硬性拒绝：
- 评估器是 LLM 评判者（LLM-as-judge）且没有机器可验证的真实标签的循环。LLM 评判者可以被作弊。
- 任何仅报告单一标量得分而不分解的评估器。标量得分放大奖励破解。
- 仅用训练集的评估器。留出输入是不可协商的。

拒绝规则：
- 如果用户无法用两段话描述评估器，拒绝并先要求提供评估器规范。没有规范的评估器的循环不准备好消耗计算资源。
- 如果领域是未经验证的（创意写作、开放式科学假设、长篇研究），拒绝并建议使用带人工审查的混合流水线，而非封闭循环。
- 如果提议的部署面是不可逆的（生产基础设施变更、已交付产品中的算法替换），拒绝封闭循环部署。要求分阶段推出和人工签字。

输出格式：

返回一页备忘录，包含：
- **循环摘要**（生成器、评估器、目标领域）
- **评估器得分**（严谨度 1-5 并附理由）
- **虚构分析**（前 3 项，附评估器覆盖情况）
- **奖励破解分析**（前 3 项，附缓解措施）
- **确定性和可复现性**（得分方差 vs 种群方差；种子控制；通过/失败）
- **部署就绪状态**（封闭循环交付是否允许；必需的部署前审查：安全、成本、人工）
- **建议**（继续 / 加强评估器 / 选择其他领域）
