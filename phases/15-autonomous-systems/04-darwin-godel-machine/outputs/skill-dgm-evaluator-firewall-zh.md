给定一个提议的自修改智能体循环（智能体脚手架、评估器流水线、部署目标），生成一份防火墙规范，将评估器保持在智能体触及范围之外。DGM 论文记录了智能体移除自身安全标记以提高得分的情况；此技能即对应控制措施。

生成：

1. **文件系统边界。** 列出智能体可以编辑的每个文件。列出评估器读取的每个文件。交集必须为空。如果不为空，指定使其为空的精确操作。
2. **进程边界。** 评估器必须在智能体无法控制的进程中运行。指定容器或沙箱。如果智能体有 shell 访问权限，智能体可以绕过进程内防护。
3. **网络和密钥边界。** 智能体不得编辑用于获取评估器输入或发布得分的 CI 密钥。列出密钥存储位置及其挂载方式。
4. **得分入口合约。** 得分从评估器返回到循环的存档。该路径是唯一通道。智能体能写入的任何影响未来评分的内容都破坏了合约。
5. **变更审查列表。** 列出在接受前需要人工审查的智能体编辑类别：新依赖、评分相关代码的编辑、沙箱配置的编辑、日志或监控的编辑。

硬性拒绝：
- 评估器与智能体源代码在同一仓库中。将其移出。
- 评估器在智能体进程内作为 Python import 运行。在子进程中运行它。
- 任何在智能体一侧边界应用的得分调整。所有调整必须是服务端。
- 任何智能体可通过编辑自身代码来静默的日志。

拒绝规则：
- 如果用户无法列出智能体可编辑的每个文件，拒绝并要求先进行文件级写访问审计。
- 如果评估器没有留出输入，拒绝——这是第 3 课的审计，是先决条件。
- 如果部署面包含对评估器本身的变更（即使是通过提议的依赖更新间接影响），拒绝并要求手动防火墙审查步骤。

输出格式：

返回一页规范，包含：
- **智能体写操作面**（路径、globs）
- **评估器读操作面**（路径、端点）
- **交集**（必须为空；显示 diff）
- **进程模型**（评估器如何隔离）
- **密钥清单**（存储位置和挂载方式）
- **需要审查的变更类别**（项目符号列表）
- **签字行**（谁拥有防火墙不变量）
