# 长视频理解：百万 Token 上下文

> 一段 1 小时、24 FPS 的 4K 视频，经分块和嵌入后，产生约 6000 万个 Token。一段 2 小时的播客文字记录是 3 万个 Token。一部完整的蓝光电影，即使经过激进池化压缩后，仍有数十万个 Token。Google 的 Gemini 1.5（2024年3月）以 1000 万 Token 上下文开启了这一时代，在小时级视频上实现了可靠的大海捞针（Needle-in-a-Haystack）召回。LWM（Liu et al.，2024年2月）展示了环形注意力（Ring Attention）的扩展路径。LongVILA 和 Video-XL 进一步扩展了摄入能力。VideoAgent 用智能体检索（Agentic Retrieval）替代了原始上下文。每种方法在计算、召回和工程复杂性上做出了不同的权衡。本课将它们并排解读。

**类型：** 构建
**语言：** Python（标准库，大海捞针模拟器 + 智能体检索路由器）
**前置要求：** 第12阶段 · 17（视频时序 Token）
**时间：** ~180分钟

## 学习目标

- 计算不同 FPS 和池化下长视频的视觉 Token 总量。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、Token 压缩（LongVILA / Video-XL）。
- 在准确率和延迟方面比较原始上下文视频 VLM 与智能体检索视频 VLM（VideoAgent）。
- 为一段 30 分钟的视频设计大海捞针测试，并测量特定分钟的召回率。

## 问题

Qwen2.5-VL 尺寸的图像块在 384 原生分辨率下每帧约 729 Token。3x3 池化后每帧 81 Token。一段 30 分钟的片段以 1 FPS = 1800 帧 = 145,800 Token。2025 年的开源 VLM 可以处理，但较为紧张。以 2 FPS 计算，则为 291,600 Token——只有最大的上下文窗口才能容纳。

一部 2 小时的电影以 1 FPS 计算为 583k Token。超出大多数 2026 年开源模型的能力范围；需要 Gemini 2.5 Pro 或更激进地池化。

出现了三种扩展路径。

## 核心概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

以硬件投入解决问题。将上下文扩展到数百万 Token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 首发 1M Token；Gemini 1.5 Ultra 达到 10M；2026 年的 Gemini 2.5 Pro 可以可靠处理数小时的视频。论文（arXiv:2403.05530）记录了在最长约 950 万 Token 下达到 99.7% 的大海捞针召回率。

工程实现：自定义注意力机制，配备内存层次结构（局部 + 全局 + 稀疏）以及 MoE 专家路由用于长上下文效率。完整细节未公开，也不开源。

### 路径 2：环形注意力（LWM、LongVILA）

环形注意力将长序列分布到多台设备上，形成一个"环"，每台设备持有一个片段。对完整序列的注意力计算通过每台设备以环形模式将其片段发送到下一台设备、计算部分注意力并聚合来实现。

LWM（Liu et al.，2024）以此方式训练了一个 1M Token 上下文模型。训练计算随序列长度线性增长，而非平方增长——注意力的平方复杂度通过环形中的设备分摊。

LongVILA（arXiv:2408.10188）将这一模式适配到 VLM。1400 帧视频，每帧 192 Token = 268k 上下文，通过 8 路并行的环形注意力训练。

### 路径 3：Token 压缩（Video-XL、LongVA）

比暴力上下文更便宜：在 LLM 看到序列之前进行激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 Token：每 N 帧的片段产生一个单一的"摘要"Token，该 Token 对 N 帧进行注意力。推理时，LLM 看到每个片段一个摘要 Token，大幅缩减上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从 200k 扩展到 2M。在长上下文文本上训练，通过共享表示迁移到长上下文视频。

Token 压缩以特定时间戳的召回率为代价换取可扩展性。模型知道大致发生了什么，但有时会错过精确的帧。

### 路径 4：智能体检索（VideoAgent）

不将完整视频输入 LLM。而是将视频视为数据库，使用 LLM 来查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 读取问题。
2. LLM 向检索工具查询相关片段（"显示有猫的片段"）。
3. 工具返回匹配片段的时间戳。
4. LLM 通过 VLM 读取这些片段。
5. LLM 组合答案或提出后续查询。

这是应用于长视频的 LLM 作为智能体（LLM-as-Agent）模式。推理更便宜（只编码相关片段），但工程更复杂（检索质量成为瓶颈）。

### 大海捞针基准测试

标准的长上下文测试：在视频的随机位置插入一个独特的视觉或文本标记，然后提出一个需要回忆它的查询。

指标：按视频长度和标记位置统计 Recall@k。

Gemini 2.5 Pro 在最长 90 分钟的视频上得分 >99% 召回率。开源 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟时得分约 85-90%，超过 60 分钟后下降。

VideoAgent 在 2 小时以上的视频上可以匹敌甚至超越原始上下文模型，因为如果工具足够好，检索能命中目标标记。

### 选择哪种路径

对于 15 分钟片段，追求前沿准确率：开源 72B + 原生上下文通常可行。选择 Qwen2.5-VL-72B。

对于 30 分钟到 1 小时的内容：开源选 LongVILA 或 Video-XL；闭源选 Gemini 2.5 Pro。质量标准很重要——前沿选闭源。

对于 2 小时以上的内容：VideoAgent 或类似的检索模式。另一种方案是摘要为较小的块并输入分层摘要。

### 2026 年生产模式

在实践中，生产级长视频流水线是混合的：

1. 在整个视频上运行动态 FPS 采样 + 激进池化（获得 100k Token 的全局表示）。
2. 传入 72B VLM 获得全局摘要。
3. 如果用户提出详细问题，使用摘要作为索引运行智能体检索。

这结合了暴力上下文用于全局理解和检索用于局部细节。

## 实践

`code/main.py`：

- 计算从 1 分钟到 3 小时视频在不同 FPS + 池化下的 Token 预算。
- 模拟大海捞针运行：在随机时间戳注入标记，提出问题，评分召回率。
- 包含一个智能体检索路由器模拟器，选择特定片段输入下游 VLM。

运行预算表，感受规模差距。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 暴力上下文 | "就是更多 Token" | 将 LLM 上下文扩展到数百万 Token；一次处理所有内容 |
| 环形注意力 | "LWM 风格并行" | 分布式注意力模式，每台设备持有一个片段并轮转 |
| Token 压缩 | "摘要 Token" | 在进入 LLM 之前通过学习的压缩器减少每个片段的 Token |
| 大海捞针 | "NIH 测试" | 在随机位置插入唯一标记，测试时要求模型回忆它 |
| 智能体检索 | "LLM 作为查询规划器" | LLM 向检索工具查询相关片段，通过 VLM 读取，组合答案 |
| VideoAgent | "视频检索模式" | 经典的智能体检索设计：问题 -> 工具 -> 片段 -> 答案 |
