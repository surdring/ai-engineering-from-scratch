# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像不是 224x224 的方块。收据是 9:16，图表是 16:9，医学扫描可能是 4096x4096，手机截图是 9:19.5。2024 年之前的 VLM 答案——将所有内容缩放到固定方块——丢弃了使 OCR、文档理解和高分辨率场景解析得以工作的信号。NaViT（Google，2023）证明你可以将可变分辨率的块打包到一个带有块对角掩码（Block-Diagonal Masking）的 Transformer 批次中。Qwen2-VL 的 M-RoPE（2024）彻底抛弃了绝对位置表。LLaVA-NeXT 的 AnyRes 将高分辨率图像平铺为基础图像 + 子图像。SigLIP 2 的 NaFlex 变体（2025）现在是希望用单个检查点服务所有宽高比的开源 VLM 的默认编码器。本课端到端实现 patch-n'-pack。

**类型：** 构建
**语言：** Python（标准库，块打包器 + 块对角掩码）
**前置条件：** Phase 12 · 01（ViT 块）、Phase 12 · 05（LLaVA）
**时间：** 约 120 分钟

## 学习目标

- 将来自一批可变分辨率图像的块打包为一个序列，并构建块对角注意力掩码。
- 在给定任务下，在 AnyRes 平铺（LLaVA-NeXT）、NaFlex（SigLIP 2）和 M-RoPE（Qwen2-VL）之间做出选择。
- 计算 OCR、图表和摄影在不缩放情况下的 token 预算。
- 指出正方形缩放（Square-Resize）的三种失败模式：文本被挤压、内容被裁剪、填充浪费 token。

## 问题所在

Transformer 期望一个序列。批次是一堆相同长度的序列。如果你的图像是 224x224，每次都得到 196 个 patch token，无需填充，搞定。在 224 上训练，在 224 上推理，永远不再考虑分辨率。

但世界不配合。文档是纵向（8.5x11 英寸，约 2:3）。图表截图是横向（16:9）。收据是又高又窄（1:3）。医学成像以 2048x2048 或更大的尺寸交付。移动设备截图是 1170x2532（0.46:1）。

三种 2024 年之前的方案及其为何失败：

1. 缩放到固定正方形（224x224 或 336x336）。挤压扭曲文本和人脸。降采样破坏图表标签和 OCR 内容。直到 LLaVA-1.5 之前是标准做法。
2. 裁剪为固定宽高比。你丢弃了图像的大部分内容，而选择裁剪位置本身就是一个视觉问题。
3. 填充到最长边。修复了扭曲问题，但对纵向图像浪费了 50%+ 的 token 在填充上。注意力计算在所有填充 token 上是二次方的。

2024-2025 年的答案：让 Transformer 以图像的原生分辨率吃入块，并想办法将异构批次打包到一个序列中而没有浪费的计算。

## 核心概念

### NaViT 与 Patch-n'-Pack

NaViT（Dehghani 等人，2023）是证明这在大规模下可行的论文。思路是机械的：

1. 对于批次中的每张图像，按选定的块大小（比如 14）计算其原生块网格。
2. 将每张图像的块展平为自己的可变长度序列。
3. 将所有图像的块拼接为批次的一个长序列。
4. 构建块对角注意力掩码，使图像 A 的块只关注图像 A 内部。
5. 携带每块的位置信息（2D RoPE 或分数位置嵌入）。

三张图像的批次，分辨率分别为 336x336（576 个 token）、224x224（256 个 token）和 448x336（768 个 token），变成一个 1600 token 的序列，带有 1600x1600 的块对角掩码。没有填充。没有浪费的计算。Transformer 处理任意宽高比。

NaViT 还引入了训练期间的分数块丢弃（Fractional Patch Dropping）——随机丢弃批次中 50% 的块——既正则化又加速训练。SigLIP 2 继承了这个做法。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes 是务实的替代方案。给定高分辨率图像和固定编码器（CLIP 或 SigLIP @ 336），将图像平铺：

1. 从预定义的集合——（1x1）、（1x2）、（2x1）、（1x3）、（3x1）、（2x2）等——中选择最适合图像宽高比的网格布局。
2. 将完整图像平铺到网格中；每个瓦片成为 336x336 的裁剪。
3. 同时生成缩略图：整张图像缩放到 336x336 作为全局上下文 token。
4. 通过冻结的 336 编码器编码每个瓦片。拼接瓦片 token + 缩略图 token。

对于 672x672 的图像，2x2 网格加缩略图：4 * 576 + 576 = 2880 个视觉 token。昂贵但有效——LLM 同时看到局部细节和全局上下文。

当你的编码器冻结且只支持一个分辨率时，AnyRes 是首选路径。对于大图它会爆炸 token 数（1344x1344 的图像在 4x4 网格下为 9216 + 576 ≈ 9800 个 token，填满了大部分 8k LLM 上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置嵌入（Multimodal Rotary Position Embedding）。与 NaViT 的分数位置或 AnyRes 的平铺+缩略图不同，每个块携带一个 3D 位置（时间、高度、宽度）。查询/键旋转处理任意 H、W 和时序长度。

M-RoPE 无需重新训练即可提供原生动态分辨率。推理时你输入任意 HxW 图像，块嵌入器产生 H/14 x W/14 个 token，每个 token 获得其（t=0，r=行，c=列）位置，RoPE 以正确的频率旋转注意力，完成。Qwen2.5-VL 和 Qwen3-VL 延续了这一做法。InternVL3 的 V2PE 是相同思路，每种模态有不同的编码。

与 AnyRes 不同，M-RoPE 以原生分辨率产生 O(H x W / P^2) 个 token——没有乘性的瓦片开销。与 NaViT 不同，它仍然期望每次前向传播一张图像。跨分辨率批处理仍然需要在其之上使用 patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex 是 SigLIP 2 检查点的原生灵活（Native-Flex）模式。单个模型在推理时服务多种序列长度（256、729、1024 个 token）。内部它在训练期间使用 NaViT 风格的 patch-n'-pack 和每块的绝对分数位置。卖点：一个检查点，根据任务在推理时选择你的 token 预算。

对于语义任务（分类、检索），256 个 token。对于 OCR 或图表理解，1024 个 token。无需重新训练。

### 打包掩码

块对角掩码是大多数实现出错的地方。对于覆盖图像 `i=0..B-1`（长度为 `n_i`）的长度为 `N_total` 的打包序列，形状为 `(N_total, N_total)` 的掩码 `M` 在两者索引都落在同一图像块内时为 1，否则为 0。你可以从累积长度列表构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 当且仅当存在 b 使得 offsets[b] <= i < offsets[b+1] 且 offsets[b] <= j < offsets[b+1]
```

这在 PyTorch 中用 `torch.block_diag` 或显式 gather 只需一行。FlashAttention 的可变长度路径（`cu_seqlens`）完全跳过掩码，直接使用累积长度张量在序列内做注意力——对于典型批次，比密集掩码快约 10 倍。

### Token 预算

按任务选择你的策略：

- OCR / 文档：1024-4096 个 token。SigLIP 2 NaFlex @ 1024，或 AnyRes 3x3 + 缩略图。
- 图表和 UI：原生 384-448 下 729-1024 个 token。Qwen2.5-VL 动态分辨率带最大像素上限。
- 自然照片：256-576 个 token 足够。下游 LLM 看到的内容足够了。在内容密度高的地方为 token 付费。
- 视频：空间池化后每帧 64-128 个 token，2-8 FPS。第 12.17 课覆盖此内容。

2026 年生产规则：为每个任务选择一个最大像素上限，以原生宽高比编码到此上限，打包批次，跳过填充。Qwen2.5-VL 正是通过 `min_pixels` 和 `max_pixels` 暴露了这个旋钮。

## 使用指南

`code/main.py` 为具有整数像素坐标的异构图像批次实现了 patch-n'-pack。它：

- 接受（H, W）图像尺寸的列表。
- 以块大小 14 计算每张图像的块序列长度。
- 将它们打包到总长度 `sum(n_i)` 的一个序列中。
- 构建块对角注意力掩码（密集，为清晰起见）。
- 将打包成本与正方形缩放和 AnyRes 平铺进行比较。
- 为混合批次（收据、图表、截图、照片）打印 token 预算表。

运行它。产出的数字是每个 2026 年开源 VLM 使用 patch-n'-pack 的原因。

## 交付物

本课产出 `outputs/skill-resolution-budget-planner.md`。给定混合宽高比的工作负载（OCR、图表、照片、视频帧）和总 token 预算，它选择正确的策略（NaFlex、AnyRes、M-RoPE 或固定正方形）并生成每次请求的配置。在为产品调整 VLM 规模时使用此技能——它防止静默的 10 倍 token 膨胀，这会摧毁延迟预算。

## 练习

1. 一张收据是 600x1500（1:2.5）。在块大小 14 下，原生分辨率有多少个 token？正方形缩放到 336 后有多少个？实践中哪种丢失更多的 OCR 准确率？

2. 为包含四个图像的批次构建块对角掩码，长度分别为 256、576、729、1024。验证注意力矩阵是 2585x2585，且恰好有 `256^2 + 576^2 + 729^2 + 1024^2` 个非零元素。

3. 对于 1792x896 的图像，块大小 14，比较：（a）正方形缩放到 336 然后编码，（b）AnyRes 2x1 + 缩略图，（c）原生 M-RoPE。哪种使用最少的 token？哪种保留最多的细节？

4. 实现分数块丢弃：给定一个打包序列，均匀随机丢弃 50% 的 token，并相应更新块对角掩码。测量掩码的稀疏性变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）第 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为什么两个边界都很重要。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| Patch-n'-pack | 「NaViT 风格打包」 | 将来自不同图像的可变长度块序列拼接到一个批次维度中 |
| 块对角掩码（Block-diagonal mask） | 「打包掩码」 | 注意力掩码，限制每张图像的块只关注自己，而非打包中的邻居 |
| AnyRes | 「LLaVA-NeXT 平铺」 | 将高分辨率图像分割为固定大小瓦片网格加上全局缩略图；用固定编码器编码每个瓦片 |
| NaFlex | 「SigLIP 2 原生灵活」 | 单个 SigLIP 2 检查点，在推理时服务 256/729/1024 token 预算，无需重新训练 |
| M-RoPE | 「多模态 RoPE」 | 3D 旋转位置编码（时间、行、列），无需位置表即可处理任意 H、W、T |
| cu_seqlens | 「FlashAttention 打包」 | FlashAttention varlen 路径使用的累积长度张量，替代密集块对角掩码 |
| min_pixels / max_pixels | 「分辨率边界」 | Qwen2.5-VL 每请求旋钮，限制极小型或极大型输入的 token 数 |
| 视觉 token 预算 | 「每张图像多少 token」 | 每张图像发出的 patch token 大致数量；设置 LLM 的提示词预算和注意力成本 |
