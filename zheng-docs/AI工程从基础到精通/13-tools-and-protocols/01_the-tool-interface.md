# 工具接口 —— 为什么 Agent 需要结构化 I/O

> 语言模型产生 Token。程序执行动作。两者之间的鸿沟由工具接口（Tool Interface）填补：一种契约，让模型请求动作，由宿主程序去执行。2026 年的每一个技术栈 —— OpenAI、Anthropic 和 Gemini 上的函数调用（Function Calling）；MCP 的 `tools/call`；A2A 的 Task Parts —— 都是同一个四步循环的不同编码形式。本课命名这个循环，并展示运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，不使用 LLM）
**前置知识：** Phase 11（LLM 补全 API）
**时间：** ~45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法自行对真实世界执行操作。
- 绘制四步工具调用循环（描述 → 决策 → 执行 → 观察），并指明每一步由谁负责。
- 将工具描述写为三个部分：名称、JSON Schema 输入和确定性执行器函数。
- 区分纯工具（Pure Tool）和副作用工具（Side-Effecting Tool），并说明这种区分对安全性的重要性。

## 问题

LLM 输出的是对下一个 Token 的概率分布。这就是它的全部输出面。如果你问一个聊天模型"班加罗尔现在的天气如何"，它能写出一句看似合理的回答，但无法拨入天气 API。这句回答可能碰巧正确，也可能已经过时三天。

弥合这一鸿沟正是工具接口的目的。宿主程序——你的 Agent 运行时、Claude Desktop、ChatGPT、Cursor，或自定义脚本——向模型宣告一个可调用工具的列表。当模型判断需要执行某个动作时，它会输出一个结构化负载（Structured Payload），指名要调用的工具及其参数。宿主解析这个负载，真正执行该工具，并将结果反馈回去。这个循环一直持续，直到模型判定不再需要更多调用。

这个契约的第一个版本于 2023 年 6 月作为 OpenAI 的"functions"参数发布。Anthropic 在 Claude 2.1 中紧随其后推出了 `tool_use` 块。几个月后 Gemini 增加了 `functionDeclarations`。如今每个提供商都暴露同样的形态：输入一个 JSON-Schema 类型的工具列表，输出一个 JSON 负载的工具调用。模型上下文协议（Model Context Protocol，MCP）于 2024 年 11 月推广了这个契约，使一个工具注册中心可以为所有模型服务。A2A（2026 年 4 月，v1.0）则在同一原语之上叠加了 Agent 到 Agent 的委托层。

四步循环是所有这些形式之下不变的内核。Phase 13 中其他所有内容都是对此的展开。

## 核心概念

### 第一步：描述（Describe）

宿主用三个字段声明每个工具。

- **名称（Name）。** 一个稳定的、机器可读的标识符。用 `get_weather`，而不是"weather thing"。
- **描述（Description）。** 一段自然语言的简要说明。"当用户询问特定城市的当前天气状况时使用。不要用于历史数据。"
- **输入 Schema（Input Schema）。** 一个 JSON Schema 对象（draft 2020-12），描述工具的参数。

模型接收这个列表。现代提供商使用提供商特定的模板将这些声明序列化到系统提示词中，因此作为调用者，你只需要处理结构化的形式。

### 第二步：决策（Decide）

根据用户的消息和可用工具，模型选择以下三种行为之一。

1. **直接用文本回答。** 不进行工具调用。
2. **调用一个或多个工具。** 输出结构化调用对象。在 `parallel_tool_calls: true` 下（OpenAI 和 Gemini 的默认值，Anthropic 需要主动开启），模型可以在一个回合中输出多个调用。
3. **拒绝（Refuse）。** 严格模式（Strict Mode）下的结构化输出可以产生一个类型化的 `refusal` 块，而不是调用。

一个工具调用负载有三个稳定字段：调用 `id`、工具 `name` 和一个 JSON `arguments` 对象。id 的存在是为了让宿主能够将后续结果与特定调用相关联——当并行调用以乱序返回时，这一点至关重要。

### 第三步：执行（Execute）

宿主接收调用，对照声明的 Schema 验证参数，然后运行执行器。无效的参数意味着模型幻想了某个字段或使用了错误的类型——这在弱模型上是非常常见的失败模式。生产级宿主在遇到无效参数时有三种处理方式：快速失败并将错误呈现给模型、用约束解析器修复 JSON，或将验证错误包含在提示词中重试模型。

执行器本身是普通代码。Python、TypeScript、Shell 命令、数据库查询。它产生一个结果，通常是字符串，但也可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察（Observe）

宿主将工具结果追加到对话中（作为带匹配 `id` 的 `tool` 角色消息），并重新调用模型。模型现在上下文中有工具输出，可以生成最终答案或请求更多调用。这个过程一直持续，直到模型停止产生调用，或宿主达到安全上限的迭代次数。

### 信任划分

从安全性角度，工具分为两类。

- **纯工具（Pure）。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地进行投机性调用。
- **后果性工具（Consequential）。** 变更状态、花费金钱、触及用户数据。`send_email`、`delete_file`、`execute_trade`。必须设门控。

Meta 2026 年的 Agent 安全"二法则（Rule of Two）"规定，单个回合最多可以结合以下两者中的两项：不可信输入、敏感数据、后果性操作。工具接口正是你执行这条规则的地方——通过拒绝调用、要求用户确认或升级权限范围。完整的安全章节见 Phase 13 · 15，Agent 级的权限策略见 Phase 14 · 09。

### 循环存在于哪里

| 上下文 | 谁描述 | 谁决策 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM（通过 MCP 客户端） | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用 Agent | 被调用 Agent |
| Web 浏览器（函数调用 Agent） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无论在何处，始终是相同的四个步骤。列名变化，但结构不变。

### 为什么不直接让模型输出 JSON？

"让模型以 JSON 格式回复"是函数调用出现之前的模式。在前沿模型上大约有 5% 到 15% 的失败率，在更小的模型上失败率更高。失败模式包括缺少大括号、多余逗号、幻想的字段和错误的类型。你需要一个 JSON 修复步骤、重试或约束解码器。

原生函数调用更好，有三个原因。第一，提供商对模型进行了端到端的精确调用格式训练，因此在严格模式下有效 JSON 的比率攀升至 98% 到 99%。第二，调用负载位于自己的协议槽中，而不是自由文本内部——因此工具调用绝不会泄露到用户可见的回复中。第三，提供商通过约束解码（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）强制执行 Schema 合规。输出保证能够通过验证。

### 断路器（Circuit Breakers）

当模型停止产生调用或宿主达到最大回合数时，循环终止。生产级宿主将此限制设置为 5 到 20 个回合。超过此限制，几乎肯定陷入了模型无法退出的循环。Claude Code 默认为 20；OpenAI Assistants 为 10；Cursor 的 Agent 模式为 25。

另一种选择——无界循环——每隔六个月就会作为"Agent 一夜之间花费了 400 美元 API 调用费"的事后分析出现。不要在未设上限的情况下发布产品。

## 使用它

`code/main.py` 在不使用 LLM 的情况下运行四步循环。一个假的"决策器"函数通过对用户消息进行模式匹配来模拟模型；执行器、Schema 验证器和观察步骤都是真实的。运行它以查看完整的请求/响应编排，以及可打印的中间状态，然后在后续课程中将假的决策器替换为任何真实的提供商。

需要关注的点：

- 工具注册中心为每个工具保存三个字段：名称、描述、Schema 和执行器引用。
- 验证器是一个只使用标准库的最小 JSON Schema 子集（types、required、enum、min/max）。Phase 13 · 04 会提供一个更完整的版本。
- 循环将迭代次数限制为 5。生产级 Agent 正是需要这种断路器。

## 交付成果

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定一个草稿工具定义（名称 + 描述 + Schema + 执行器概要），该技能审核其是否适合循环使用：名称是否机器稳定、描述是否提供了完整的使用说明、Schema 是否正确使用了 JSON Schema 2020-12、以及纯/后果性分类是否明确。

## 练习

1. 向 `code/main.py` 中添加第四个工具 `get_stock_price(ticker)`。将其描述写为"当用户通过股票代码询问当前股价时使用。不要用于历史价格或市场摘要。"运行测试程序，确认假决策器能将提及股票代码的查询路由到新工具。

2. 破坏 Schema 验证器。传递一个缺少必填字段的调用，确认宿主在执行前拒绝它。然后传递一个包含额外未知字段的调用。做出决定：宿主应该拒绝还是忽略？以安全性论证来证明你的选择。

3. 将测试程序中的每个工具分类为纯工具或后果性工具。向需要标记的注册条目添加 `consequential: true` 标志，并修改循环，使其在选中后果性工具时打印"将请求用户确认"的提示。这是每个生产级宿主都需要的确认门控的形状。

4. 在纸上绘制四步循环，并针对你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义技术栈）填写上面的提供商列表。与 Phase 13 · 06 中 MCP 特定的变体进行交叉对照。

5. 从头到尾阅读 OpenAI 的函数调用指南。找出请求中存在但未在此处介绍的四步循环中出现的一个字段。解释它增加了什么，以及为什么它是便利性的而非必要的。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| 工具（Tool） | "模型可以调用的东西" | 由名称 + JSON-Schema 类型化输入 + 执行器函数组成的元组 |
| 函数调用（Function Calling） | "原生工具使用" | 提供商级 API 支持，使模型能够输出结构化工具调用而非自由文本 |
| 工具调用（Tool Call） | "模型请求执行操作" | 模型输出的一个 JSON 负载，包含 `id`、`name`、`arguments` |
| 工具结果（Tool Result） | "工具返回的内容" | 执行器的输出，包装在带匹配 id 的 `tool` 角色消息中 |
| 并行工具调用（Parallel Tool Calls） | "一次发出多个调用" | 模型中一个回合包含多个调用对象，相互独立且可按 id 排序 |
| 严格模式（Strict Mode） | "保证 JSON 合规" | 约束解码，强制模型输出必须通过声明的 Schema 验证 |
| 纯工具（Pure Tool） | "只读工具" | 无副作用；可以安全地重复运行 |
| 后果性工具（Consequential Tool） | "操作工具" | 改变外部状态；需要门控、审计或用户确认 |
| 四步循环（Four-Step Loop） | "工具调用周期" | 描述 → 决策 → 执行 → 观察 |
| 宿主（Host） | "Agent 运行时" | 持有工具注册中心、调用模型并运行执行器的程序 |
