# 监督者 / 编排者-工作者模式

> 一个主导代理规划和委托；专业工作者在并行上下文中执行并报告回来。这是 Anthropic 研究系统（Claude Opus 4 为主导，Sonnet 4 为子代理）背后的模式，在内部研究评估中比单代理 Opus 4 高出 +90.2%。Anthropic 的工程文章报告，BrowseComp 上 80% 的方差仅由 token 使用量解释——多代理胜出主要是因为每个子代理获得一个全新的上下文窗口。本课从原语构建监督者模式，并涵盖 2026 年生产部署的工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库、`threading`）
**前置知识：** 第 16 阶段 · 04（原语模型）
**时间：** 约 75 分钟

## 问题

研究是单代理系统失败的原型任务。你问「2023 年到 2026 年间多代理系统发生了什么变化？」一个单代理顺序读取五篇论文，填满了一半上下文，然后必须对所有这些进行推理。到第五篇时它已经忘记了第一篇。它无法并行化。

监督者模式修复了这一点：一个主导代理规划搜索，将每个子问题委托给一个工作者，然后综合。每个工作者为狭窄的问题获得自己的 20 万 token 窗口。主导代理永远不会看到原始论文——只看到工作者摘要。

Anthropic 的生产研究系统报告，在内部研究评估中比单个 Opus 4 高出 +90.2%。同一篇文章指出，BrowseComp 上 80% 的方差仅由 *token 使用量*来解释。每个子代理的全新上下文是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   主导代理   │  规划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1 │  │ 工作者2 │  │ 工作者3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
        全新          全新          全新
        上下文        上下文        上下文
```

主导代理从不阅读原始材料。在主导代理综合之前，工作者从不看到彼此的工作。每个箭头都是一次带有狭窄工件的交接。

### 为什么它获胜

三个机制：

1. **每个子代理的全新上下文。** 探索「FIPA-ACL 遗产」的工作者不携带主导代理在规划时花费的 4 万 token。它为一个问题获得一个 20 万 token 的窗口。
2. **通过提示进行专业化。** 主导的提示是「分解并综合」，不是「研究」。每个工作者的提示很狭窄：「找到 X 中有什么变化」。专注的提示产生专注的输出。
3. **并行性。** 工作者并发运行。墙钟时间大约为 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。

### 工程经验（Anthropic 2025）

Anthropic 的文章列出了几个到 2026 年仍然相关的生产经验：

- **根据查询复杂度调整投入规模。** 简单查询：一个代理，3-10 次工具调用。复杂查询：10+ 个代理。主导代理必须估算这一点，而不是由调用者估算。
- **先宽后窄。** 首先分解为宽泛的子问题，如果答案需要深度，则为每个子问题生成更多工作者。
- **彩虹部署（Rainbow Deployment）。** 代理是长时间运行且有状态的。传统的蓝绿部署（Blue-Green）不适用。Anthropic 使用彩虹部署：新版本逐步推出，而旧版本逐步淘汰。
- **Token 使用量占主导。** 多代理大约是单代理的 15 倍 token。仅在任务价值证明成本合理时才运行。

### LangGraph 的转变

LangGraph 最初发布了一个 `langgraph-supervisor` 库，带有高级的 `create_supervisor` 辅助函数。2025 年，LangChain 将推荐改为直接通过工具调用实现监督者模式，因为工具调用对*监督者看到什么*（上下文工程）提供了更多控制。该库仍然可用；文档现在推荐工具调用形式。

### 失败模式

- **主导代理幻觉出计划。** 如果主导生成的子问题没有分解真实问题，工作者就在错误目标上进行精确研究。
- **工作者过度探索。** 没有明确的边界范围，工作者会偏离其分配的子问题，污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导必须要么重新提问（增加一轮），要么明确指出分歧。悄无声息地选择一方是最糟糕的失败：用户永远不会知道发生了分歧。

### 什么时候监督者是错误的

- **顺序任务。** 如果步骤 2 确实需要步骤 1 的输出，并行性毫无益处。使用管道（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单代理更快更便宜地处理它们。在生成工作者之前使用主导代理的「调整规模」检查。
- **严格确定性。** 监督者使用 LLM 选择的委托。当审计/重放比适应性更重要时，静态图更好。

## 构建

`code/main.py` 使用 `threading` 实现了一个三个并行工作者的监督者。主导代理将查询分解为子问题，工作者在每个子问题上并发运行，主导代理进行综合。没有真实的 LLM——工作者被脚本化以模拟获取并总结。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假的摘要（在生产中可以是任何使用工具的代理）。
- `Lead.run(query)` 在线程中启动工作者，等待完成，并进行综合。

运行：

```
python3 code/main.py
```

输出显示计划、带有开始/结束时间戳的并行工作者追踪，以及最终综合。你可以看到墙钟时间的优势：三个 0.3 秒的工作者在大约 0.35 秒内运行，而不是 0.9 秒。

## 实践

`outputs/skill-supervisor-designer.md` 接收一个用户查询，并生成监督者模式设计：主导系统提示、工作者角色、子问题分解规则和综合模板。在构建新的研究风格代理系统之前使用此工具。

## 交付

部署监督者模式前的检查清单：

- **模型配对。** 主导代理使用推理层模型（Opus 级别、`o3` 级别）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何超过中位数运行时间 2 倍的工作者被终止；主导要么以更窄范围重新生成，要么在没有它的情况下继续。
- **每个工作者的 token 上限。** 硬限制（如预期综合输入的 10 倍）防止失控工作者耗尽预算。
- **可观测性。** 追踪主导的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹推出。** 有状态的长时间运行代理需要渐进版本过渡，而非热替换。

## 练习

1. 运行 `code/main.py`，然后修改主导生成为 5 个工作者而非 3 个。观察墙钟时间效果。在此演示中，工作者数量达到多少时生成开销超过并行节省？
2. 实现工作者超时：终止任何运行超过 0.5 秒的工作者，让主导综合剩余结果。你需要什么可观测性来知道一个工作者被切断了？
3. 在主导的综合中加入冲突检测步骤：如果两个工作者返回矛盾的答案，主导记录分歧而非选择其一。如何在不调用 LLM 的情况下检测矛盾？
4. 阅读 Anthropic 研究系统工程文章。列出此玩具演示在投入生产时需要采纳的三个实践。
5. 比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用推荐。哪种让你更好地控制监督者看到什么？为什么 Anthropic 显式地只将子答案而不是原始工作者上下文传递给综合？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Supervisor | 「主导代理」 | 一个编排者代理，负责规划、委托和综合。不自己做实际工作。 |
| Worker | 「子代理」 | 由监督者调用的专注代理，具有狭窄范围和自己的上下文窗口。 |
| Orchestrator-worker | 「监督者模式」 | 同一件事，不同名称。2026 年文献两者都用。 |
| Fresh context | 「干净窗口」 | 工作者的上下文从其系统提示和分配的问题开始，而非主导的历史。 |
| Rainbow deployment | 「渐进式推出」 | 长时间运行的有状态代理需要版本化的淘汰和替换，而非蓝绿部署。 |
| Token dominance | 「上下文是变量」 | 根据 Anthropic，80% 的研究评估方差来自使用的总 token，而非模型选择。 |
| Scale effort | 「将代理数量与复杂度匹配」 | 主导估算查询难度，据此生成 1 个或 10+ 个工作者。 |
| Synthesis conflict | 「工作者意见分歧」 | 两个工作者返回矛盾的事实；主导必须呈现分歧，而非悄无声息选择一方。 |
