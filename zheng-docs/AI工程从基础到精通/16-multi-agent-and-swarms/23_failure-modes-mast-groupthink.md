# 失败模式——MAST、群体思维、单文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等人，NeurIPS 2025，arXiv:2503.13657），源自 7 个最先进开源 MAS 的 1642 条执行追踪，显示 **41–86.7% 的失败率**。三个根类别：**规范问题（Specification Problems）**（41.77%）——角色模糊、任务定义不清；**协调失败（Coordination Failures）**（36.94%）——通信故障、状态不同步；**验证缺口（Verification Gaps）**（21.30%）——缺少验证、缺乏质量检查。**群体思维（Groupthink）** 家族（arXiv:2508.05687）增加：单文化崩溃（Monoculture Collapse）（相同基座模型 → 相关失败）、从众偏差（Conformity Bias）（代理相互强化错误）、心智理论（Theory of Mind）不足、混合动机动态、级联可靠性失败。级联示例：支付失败触发订单重试，订单重试触发库存重试，库存重试压垮库存服务（数秒内负载 10 倍——需要断路器（Circuit Breaker））。记忆投毒（Memory Poisoning）：一个代理的幻觉进入共享记忆，下游代理将其视为事实；准确率逐渐下降，使根因诊断极其困难。**STRATUS**（NeurIPS 2025）报告通过专门的检测/诊断/验证代理，缓解成功率提升 1.5 倍。本课将失败模式作为一等工程目标来处理。

**类型：** 学习
**语言：** Python（标准库）
**前置条件：** Phase 16 · 13（共享记忆）、Phase 16 · 14（共识与拜占庭容错）、Phase 16 · 15（投票与辩论拓扑）
**时间：** ~75 分钟

## 问题

多代理系统在真实任务上的失败率为 41-86.7%（Cemri 等人 2025 年在 7 个开源 MAS 上测量）。这不是"加更多代理"就能调试的。这些失败有结构性原因。MAST 分类法为你提供了类别。本课将每个类别映射到具体的检测、诊断和缓解模式，让数字不再是随机的。

2026 年的生产实践是将失败模式视为设计输入。你的架构只有在你能指出每个 MAST 类别并说出你部署的缓解措施时才算"足够好"。

## 概念

### MAST 类别

**规范问题（41.77% 的失败）。** 代理的任务定义不够严密。示例：

- 角色模糊：两个代理都认为自己是审查者。
- 任务规格不足："总结这个"，但用户想要特定的角度。
- 成功标准隐含：代理无法判断自己是否成功。

缓解措施：
- 编写显式角色契约。每个代理的提示词说明它做什么*以及不做什么*。
- 每任务的验收测试。在代理开始前，定义"完成的样子是 X"。
- 预飞行规范检查：一个独立代理在分发前审查任务定义。

**协调失败（36.94%）。** 通信或状态故障。

示例：
- 两个代理在没有同步的情况下更新共享状态。
- 代理之间丢失消息（队列故障、超时）。
- 状态漂移：代理 A 认为任务完成；代理 B 仍在执行。

缓解措施：
- 带乐观并发的版本化共享状态。
- 关键消息的显式确认（重试直到确认）。
- 定期状态同步检查点；及早检测漂移。

**验证缺口（21.30%）。** 对输出没有独立检查。

示例：
- 一个代理声称成功；无人验证。
- 代理链中的每个代理都信任前一个的输出。
- 对涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立验证者代理（第 13 课）。只读、独立源访问。
- 显式交接契约："A 的输出必须通过检查器 C 后 B 才能开始。"
- 用于事后分析的结果日志记录。

### 群体思维家族（arXiv:2508.05687）

当代理同质化或相互模仿时出现的五种相关失败：

**单文化崩溃。** 相同基座模型或训练数据 → 相关错误。当三个代理共享一个 LLM 时，它们共享其幻觉。

**从众偏差。** 代理向最响亮或最自信的同伴调整，即使对方是错误的。

**心智理论不足。** 代理未能建模彼此的信念；协调崩溃（第 18 课）。

**混合动机动态。** 具有部分对齐激励的代理漂向妥协中间点，谁都不满意。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例——重试风暴

2026 年经典的事故模式：

```
支付服务 10% 的请求失败
   ↓
订单代理重试支付（指数退避但 naive）
   ↓
每次重试都是一次新的订单-库存检查
   ↓
库存服务看到 2 倍正常负载
   ↓
库存服务开始超时
   ↓
每个订单重试库存检查
   ↓
库存服务看到 10 倍正常负载
   ↓
集群宕机
```

修复方法是经典手段：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。加上有上限的每请求重试预算。

断路器是少数可以直接从分布式系统借用而无需修改的多代理失败缓解措施之一。

### 记忆投毒（回顾）

来自第 13 课：一个代理的幻觉变成共享记忆事实；下游代理在被投毒的事实上推理。在 MAST 术语中，这是共享记忆层的验证缺口。

逐渐准确率下降是其症状。你不会得到崩溃；你会得到难以根因定位的缓慢漂移。

缓解：仅追加日志、来源追溯、不可写验证者。第 13 课已涵盖。

### STRATUS——用于失败检测的专门化代理

STRATUS（NeurIPS 2025）报告当你部署以下角色时缓解成功率提升 1.5 倍：

- **检测代理。** 观察症状模式（高分歧、重试激增、准确率漂移）。
- **诊断代理。** 给定症状，从 MAST 分类法推断可能的根因。
- **验证代理。** 在应用缓解措施后，检查症状是否消除。

这是 SRE 风格的事故响应，应用于代理系统。这三个角色都可以是具有专门提示词的 LLM 代理。

### 失败模式审计

2026 年的最佳实践是每年（或每主要版本）进行一次失败模式审计：

1. **追踪采样。** 收集约 1000 条真实执行追踪。
2. **分类。** 对于每条追踪中的失败，映射到 MAST + 群体思维类别。
3. **计算按类别的失败率。** 哪些类别主导你的系统？
4. **排名缓解措施。** 哪个修复能消除最多的失败？
5. **选择 2-3 个缓解措施。** 实施；下季度重新审计。

纪律比具体选择更重要。没有审计，失败会融入噪音，永远得不到系统性解决。

### 当系统静默失败时

最危险的失败类别是静默正确性失败。大声失败（崩溃、异常、告警）的系统可以被监控。产生看似合理但错误的输出的系统无法通过异常日志检测。这就是为什么验证缺口按每次失败算是最昂贵的类别，尽管在数量上仅占 21.30%。

投资于：
- 基于采样的人工审查。
- 黄金数据集回归测试。
- 对重要输出的跨代理交叉检查。

### 失败 vs 缓慢失败

有些失败是即时的；有些是缓慢的。即时失败（超时、模式不匹配、认证错误）检测成本低。缓慢失败（记忆投毒、单文化漂移、角色模糊）检测和预防成本高。

2026 年的工程手段：对缓慢失败代理进行仪表化，以便在漂移变成可见错误之前捕获它。一致性率、重试率、输出长度分布以及连续代理版本之间的编辑距离都是有用的代理指标。

## 构建它

`code/main.py` 实现：

- `FailureTaxonomy` —— 将模拟事故分类到 MAST + 群体思维类别。
- `CircuitBreaker` —— 经典模式；错误率超过阈值时打开。

运行：

```
python3 code/main.py
```

预期输出：
- 无断路器时的重试风暴：库存错误爆炸（模拟）。
- 有断路器时：在阈值处上限；提供降级模式响应。
- 检测代理标记该模式并命名 MAST 类别。

## 使用它

`outputs/skill-mast-auditor.md` 在多代理系统上运行 MAST 风格失败模式审计。追踪 → 分类 → 缓解排名。

## 交付它

生产中的失败模式纪律：

- **每季度 MAST 审计。** 不是每年。类别随系统增长而变化。
- **处处断路器。** 对任何依赖服务的每次出站调用。默认在 5-10% 错误率时打开阈值。
- **黄金数据集。** 小型、高质量、人工审计。每周对其进行回归测试。
- **STRATUS 三重奏。** 检测 + 诊断 + 验证代理监控生产。先从检测代理开始；当症状嘈杂时再增加诊断。
- **失败预算。** 按类别的显式 SLO 失败率。超出预算触发停止发布的对话。

## 练习

1. 运行 `code/main.py`。确认断路器限制了重试风暴。改变失败阈值并观察权衡。
2. 实现一个**缓慢失败代理指标**：3 个并行代理之间的一致性率。当它急剧下降时触发告警。通过逐渐关联代理输出来模拟单文化漂移。
3. 阅读 Cemri 等人（arXiv:2503.13657）。选择其 7 个 MAS 系统之一，映射其前 3 个失败类别。这些与 MAST 预测的相比如何？
4. 阅读群体思维论文（arXiv:2508.05687）。确定五种模式中哪种在生产中最难检测。提出一个代理指标。
5. 为你了解的特定多代理系统设计 STRATUS 风格的检测-诊断-验证三重奏。检测观察哪些症状？诊断推荐哪些缓解措施？验证如何确认它们有效？

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|----------------|------------------------|
| MAST | "2026 年分类法" | Cemri 2025；3 个根类别 + 14 个失败子类型。 |
| 规范问题 | "角色模糊" | 任务或角色定义不足；代理不知道做什么。 |
| 协调失败 | "状态漂移" | 代理之间的通信或同步故障。 |
| 验证缺口 | "没人检查" | 输出在没有独立验证的情况下被接受。 |
| 群体思维家族 | "同质性失败" | 单文化、从众、心智理论不足、混合动机、级联。 |
| 单文化崩溃 | "相同模型，相同幻觉" | 来自共享基座模型或训练数据的相关错误。 |
| 重试风暴 | "级联错误放大" | 一个失败触发重试，放大下游负载。 |
| 断路器 | "错误率过高时快速失败" | 错误率超过阈值时打开；用默认值短路。 |
| STRATUS | "事故响应三重奏" | 检测 + 诊断 + 验证代理。1.5 倍缓解成功率。 |
| 记忆投毒 | "幻觉传播" | 共享记忆事实被污染；下游代理在被投毒的内容上推理。 |
