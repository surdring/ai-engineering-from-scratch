# 共享记忆与黑板模式

> 2026 年多 Agent 系统中有两种共存的方式：**消息池（Message Pool）**（每个人都看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板（Blackboard with Subscription）**（代理订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者是多 Agent 系统中唯一的有状态部分——这意味着两者是有趣 Bug 的所在地。参考失败模式是**记忆投毒（Memory Poisoning）**：一个代理幻觉出一个「事实」，其他代理将其视为已验证，准确性以比即时崩溃更难调试的方式逐渐衰减。本课从标准库构建两种结构，注入投毒攻击，并展示三种在生产中实际有效的缓解措施。

## 问题

多 Agent 系统需要一个代理共享事实的地方。一个字面选择是「在消息中传递一切」——但那是以额外复制重新发明共享状态。另一个是「给每个人一个全局日志」——但全局日志无限增长且容易被投毒。第三个是「为每个代理投影视图」——可扩展但 Schema 重。

当其中一个代理幻觉并将幻觉写入共享状态时，每个读取该状态的下游代理都将幻觉采纳为事实。到人类注意到时，推理链已经五步深了，根本原因是写入的第三条消息。调试多 Agent 准确性衰减比调试崩溃更难。

这就是记忆投毒。它是 MAST 分类法（Cemri 等人，arXiv:2503.13657）中记录第二多的失败族，而且是结构性的：任何没有来源追溯和不可写验证者的共享记忆设计最终都会出现它。

## 概念

### 两种主要拓扑

**全量消息池。** 每个代理读取每条消息。AutoGen GroupChat 和 MetaGPT 使用它。简单、透明、可检查，但不能扩展到约 10 个代理以上，因为每个代理的上下文被其他代理的工作填满。

```
agent-A ──写入──▶ ┌────────────────┐ ◀──读取── agent-D
                   │   消息池       │
agent-B ──写入──▶ │                │ ◀──读取── agent-E
                   │ (全局日志)     │
agent-C ──写入──▶ └────────────────┘ ◀──读取── agent-F
```

**带订阅的黑板。** 代理声明对主题的兴趣；底层只路由相关消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用它。可进一步扩展，但需要预先设计 Schema 以使订阅有意义。

```
                   ┌─ 主题: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (已订阅)
                   ├─ 主题: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (已订阅)
                   ├─ 主题: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (已订阅)
                   └──────────────────┘
```

### 各自优势

- **全量池**在代理少（< 10）、异构且对话时间跨度短时胜出。当每个人都看到一切时，推理谁说了什么很简单。
- **黑板**在代理多、角色同质但实例众多（集群）、且对话长时间运行时胜出。路由节省了 token 成本和上下文污染。

生产系统通常混合：顶层一个小全量池（规划层），下面放黑板（工作者层）。

### 记忆投毒，一个场景

三个代理处理一个研究任务。Agent A 是检索代理。Agent B 是摘要者。Agent C 是分析师。

1. A 获取一个页面并写入共享状态：「该研究报告了 42% 的准确率提升。」
2. 获取的页面实际上写的是「4.2% 提升」。A 幻觉了一个小数点。
3. B 读取共享状态后写道：「报告了 42% 的大幅准确率提升（来源：A）。」
4. C 读取共享状态后写道：「建议采用——42% 的提升是变革性的。」
5. 最终报告引用了一个从未存在过的 42% 数据。

没有代理崩溃。没有测试失败。系统「工作了」。幻觉通过共享状态从一个代理的上下文跨越到每个下游代理的推理中。

### 为什么这是结构性的

没有共享状态，代理 A 的幻觉停留在 A 的上下文中。下游代理会重新获取或重新推导，可能会捕获错误。有了朴素的共享状态，A 的上下文变成了每个人的上下文，幻觉被洗白为事实。

问题不在于共享状态本身——而是**没有来源追溯和独立验证者的**共享状态。三种缓解措施解决这个问题：

1. **为每次写入附加来源追溯。** 共享状态中的每个条目记录谁写的、何时、在什么提示下、以及（如适用）代理引用了什么来源。下游代理根据来源追溯带怀疑地阅读。
2. **版本化写入；将其视为仅追加。** 更正是取代旧条目的新条目，而非就地更新。审计追踪被保留。
3. **保留至少一个不能写入共享状态的代理。** 一个只读验证者代理抽样条目、重新获取来源并标记不一致。因为它不能写入池中，它不能被池投毒。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 代理早四个十年。Hayes-Roth（1985 年，「A Blackboard Architecture for Control」）描述了专业的知识源（Knowledge Source），它们观察全局黑板、贡献部分解决方案并触发其他源。2026 年的黑板（CA-MCP、Matrix）是相同的模式，以 LLM 代理作为知识源，以 JSON 块作为部分解决方案。旧文献有关于写入争用、机会主义控制和一致性的文档化解决方案，现代系统在重新发现这些。

### 投影 vs 全量视图

纯黑板给每个订阅者相同的投影（主题范围限定）。更激进的设计是**按代理投影（Per-Agent Projection）**：每个代理获得一个根据其角色定制的视图。LangGraph 的状态归约器是 2026 年的规范实现——归约器函数将全局状态折叠为角色特定的切片。

按代理投影进一步扩展但需要 Schema。没有 Schema，你在每个代理的提示中重建特设投影。

### 写入争用模式

多个代理同时写入是并发问题，不仅仅是 LLM 问题。三种模式有效：

- **顺序写入者（单生产者）。** 所有写入通过一个协调代理序列化。简单，但是瓶颈。
- **带版本控制的乐观并发。** 每个条目有版本；写入者在版本不匹配时失败并重试。经典数据库技术。
- **主题分区。** 不同代理拥有不同主题。没有跨主题争用。需要设计的分区边界。

大多数 2026 框架默认为顺序写入者，因为 LLM 调用足够慢，争用很少且瓶颈无害。

### 不可写验证者

最承重的缓解是不可写验证者。实现规则：

- 验证者与团队共享状态（读取黑板或池）。
- 验证者没有共享状态的写入句柄——只有单独的验证通道。
- 验证者独立获取写入中引用的来源。标记分歧。
- 验证者自身的输出被路由到人类或单独的决策代理，从不反馈回池中。

没有这种分离，验证者的输出成为池中的新条目，这意味着被投毒的池会毒害验证者，进而毒害其验证。

## 构建

`code/main.py` 在标准库 Python 中实现两种拓扑，外加一个玩具投毒攻击和三种缓解措施。

- `MessagePool`——线程安全的仅追加日志，全量读出。
- `Blackboard`——主题键控的发布/订阅，带按代理订阅。
- `ProvenanceEntry`——每次写入记录（写入者、时间戳、提示哈希、来源 URI）。
- `PoisoningScenario`——运行一个三代理研究任务，其中代理 A 幻觉了一个小数点。打印最终报告。
- `Verifier`——一个重新获取来源并标记不一致的只读代理。在有验证者存在的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证者）：幻觉出的 42% 传播到最终报告。
- 运行 2（有验证者）：验证者标记不一致，池被标记为「已标记」，最终报告包含撤回。

## 交付

对于任何共享记忆设计：

- 记录每次写入的来源追溯：`(写入者, 时间戳, 提示哈希, 引用的工具调用, 来源 URI)`。
- 使日志仅追加。更正是引用被取代条目的新条目。
- 部署至少一个具有独立来源访问权限的只读验证者代理。
- 将验证者输出路由到单独的通道，不返回共享池。
- 记录取代写入的比例——上升比例是幻觉模式的早期证据。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Message pool | 「共享聊天历史」 | 每个代理都读取的仅追加日志。完全透明，扩展性差。 |
| Blackboard | 「共享工作空间」 | 主题键控的发布/订阅。代理订阅相关主题。可进一步扩展。 |
| Provenance | 「谁写了什么」 | 每次写入的元数据：写入者、时间戳、提示、来源。 |
| Memory poisoning | 「幻觉传播」 | 一个代理的错误进入共享状态，下游代理将其采纳为事实。 |
| Append-only | 「无就地更新」 | 更正是取代旧条目的新条目。保留审计追踪。 |
| Unwritable verifier | 「独立审计员」 | 重新获取来源并标记不一致的只读代理。 |
| Projection | 「限定视图」 | 从全局状态计算出的按代理视图。LangGraph 归约器是规范案例。 |
| Knowledge Source | 「专业代理」 | Hayes-Roth 1985 年对黑板参与者的术语。 |
