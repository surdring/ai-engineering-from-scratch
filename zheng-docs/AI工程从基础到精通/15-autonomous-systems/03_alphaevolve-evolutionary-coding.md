# AlphaEvolve——进化式编码代理

> 将一个前沿编码模型与一个进化循环和一个机器可检查的评估器配对。让循环运行足够久。它发现了一个使用 48 次标量乘法的 4x4 复数矩阵乘法过程——56 年来对 Strassen 算法的首次改进。它还在 Google 生产环境中找到了一个 Borg 调度启发式算法，回收了约 0.7% 的集群计算资源。架构故意保持简单。胜利来自评估器的严谨性。

**类型：** 学习
**语言：** Python（标准库，进化循环玩具）
**前置条件：** Phase 15 · 01（长周期框架），Phase 15 · 02（自我教学的推理）
**时间：** ~60 分钟

## 问题

大语言模型可以写代码。进化算法可以搜索代码。两者都已分别尝试了数十年；两者都遇到了天花板。LLM 的天花板是虚构（Confabulation）：模型写出看似合理但不做其声称之事的代码。进化的天花板是搜索成本：对语法的随机变异很少产生可编译的程序，更不用说更好的。

AlphaEvolve（Novikov 等人，DeepMind，arXiv:2506.13131，2025 年 6 月）将两者结合。LLM 向程序数据库提出有针对性的编辑；自动评估器对每个变体评分；高分变体成为未来世代的父代。LLM 处理编写看似合理代码的昂贵步骤；评估器捕获虚构。循环运行数小时到数周。

报告的结果：48 次标量乘法的 4x4 复数矩阵乘法（Strassen 1969 年的界限是 49），Google 生产中的 Borg 调度启发式算法，32.5% 的 FlashAttention 内核加速，Gemini 训练吞吐量改进。

该架构有效是因为评估器是机器可检查的。在评估器不是这样的地方就不工作。这种不对称性是课程的要点。

## 概念

### 循环

1. 从一个正确但次优的种子程序 `P_0` 开始。
2. 维护一个变体程序数据库，每个由评估器评分。
3. 从数据库中采样一个或多个父代（MAP-Elites 风格或岛屿模型）。
4. 提示 LLM（Gemini Flash 用于许多候选，Gemini Pro 用于困难的）产生父代的修改变体。
5. 在留出评估器上编译、运行和评估变体。
6. 按其分数和特征向量为键插入数据库。
7. 重复。

两个细节很重要。首先，LLM 的提示包含的不仅仅是父程序——通常是数据库中的几个顶级变体，加上评估器签名，加上简短的任务描述。模型的工作是提出一个可能改进分数的有针对性的更改。其次，数据库是结构化的（MAP-Elites 网格，岛屿模型），使循环探索多样性，而不仅是当前领先者。

### 是什么让评估器不可商议

AlphaEvolve 的胜利全部来自评估器快速、确定且难以游戏的领域：

- **矩阵乘法算法**：一个将矩阵相乘并逐位检查相等性的单元测试。
- **Borg 调度启发式算法**：一个生产级模拟器，重放历史集群负载并测量浪费的计算。
- **FlashAttention 内核**：一个正确性测试加上真实硬件上的挂钟基准测试。
- **Gemini 训练吞吐量**：测量的每步 GPU 秒。

在每种情况下，评估器捕获了否则会占主导的一类 LLM 错误：虚构的正确性声明、在硬件上消失的性能声明以及边缘情况失败。移除评估器，循环就为漂亮的代码优化。

### 奖励黑客是这一陈述的另一面

进化为评估器测量的任何东西优化。如果评估器不完美，循环会发现不完美之处。在未验证的领域，循环会为表面特征优化，而非预期行为。DeepMind 在论文中明确标记了这一点：AlphaEvolve 的成功仅转移到评估器严谨性与搜索雄心匹配的领域。

2025-2026 年代码搜索循环中奖励黑客的具体例子：

- 奖励"完成时间"的优化目标奖励提交空解决方案。
- 奖励测试下正确性的基准分数奖励记忆测试和过拟合。
- "代码质量"代理奖励删除注释和重写变量名，没有语义变化。

AlphaEvolve 中的修复：交付一个 LLM 从未见过的留出评估器，输入在评估时生成。即便如此，DeepMind 建议对任何提议的部署进行严格审查。

### 为什么 LLM + 搜索胜过单独任何一种

LLM 可以产生可编译的、语义上合理的修改。对 2000 行的 Python 文件进行随机变异遗传算法几乎总是产生语法错误。LLM 还将搜索集中在合理的邻域（改变一个函数，而非随机字节），这大大减少了浪费的评估器调用。

反过来，评估器捕获 LLM 的虚构。LLM 会自信地声称一个函数"在极限情况下是 O(n log n)"而实际上是 O(n²)；挂钟基准测试使问题得到定论。

### AlphaEvolve 在前沿技术栈中的位置

| 系统 | 生成器 | 评估器 | 领域 | 示例胜利 |
|------|--------|--------|------|---------|
| AlphaEvolve | Gemini | 正确性 + 基准测试 | 算法、内核、调度器 | 48 次乘法 4x4 矩阵乘 |
| FunSearch（DeepMind，2023） | PaLM / Codey | 正确性 | 组合数学 | cap-set 下界 |
| AI Scientist v2（Sakana，第 5 课） | GPT/Claude | LLM 评判 + 实验 | ML 研究 | ICLR workshop 论文 |
| Darwin Godel Machine（第 4 课） | 代理脚手架 | SWE-bench / Polyglot | 代理代码 | 20% → 50% SWE-bench |

四个都是相同配方的变体：生成器加评估器，循环。差异在于评估器评分什么以及有多严谨。

## 使用场景

`code/main.py` 在玩具符号回归问题上实现最小 AlphaEvolve 风格的循环。"LLM"是一个标准库代理，对计算目标函数的程序提出小的语法变异。"评估器"在留出测试点上测量均方误差。

观察：

- 最佳分数如何在世代中改进。
- MAP-Elites 网格如何保持多样化解决方案存活，使循环不收敛到局部最小值。
- 移除留出测试（仅训练评估器）如何让循环壮观地过拟合。

## 部署

`outputs/skill-evaluator-rigor-audit.md` 是在新领域考虑 AlphaEvolve 风格循环的前提条件：你的评估器是否实际捕获了你关心的失败？

## 练习

1. 运行 `code/main.py`。注意最佳分数轨迹。禁用留出评估器（标志 `--no-holdout`）并重新运行。量化过拟合。

2. 阅读 AlphaEvolve 论文第 3 节关于 MAP-Elites 网格的内容。为新问题（例如编译器优化传递）设计一个特征向量描述符，保持搜索多样。

3. 48 次乘法 4x4 结果在 56 年后改进了 Strassen 的 49 次乘法界限。阅读论文附录 F，用三句话解释为什么这个问题的评估器特别容易做对，以及为什么大多数领域不像它。

4. 提出一个 AlphaEvolve 会失败的领域。识别评估器确切在哪里崩溃以及为什么。

5. 对于你了解的领域，写出你会使用的评估器签名。包括 (a) 正确性条件，(b) 性能指标，(c) 留出输入生成规则，(d) 至少一个反奖励黑客检查。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| AlphaEvolve | "DeepMind 的进化式编码代理" | Gemini + 程序数据库 + 机器可检查的评估器 |
| MAP-Elites | "保持多样性的存档" | 按特征向量为键的网格；每个单元保存具有该描述符的最佳变体 |
| 岛屿模型（Island Model） | "并行进化的子种群" | 定期迁移的独立种群；防止过早收敛 |
| 机器可检查的评估器 | "确定性预言机" | LLM 无法伪造的单元测试、模拟器或基准测试——此循环的前提条件 |
| 奖励黑客（Reward Hacking） | "优化指标而非目标" | 循环找到最大化分数而不做预期任务的方法 |
| 种子程序（Seed Program） | "起点" | 循环从中进化的初始正确但次优的程序 |
| 留出评估器（Held-Out Evaluator） | "LLM 从未见过的评估数据" | 在评估时生成的输入以防止记忆 |
