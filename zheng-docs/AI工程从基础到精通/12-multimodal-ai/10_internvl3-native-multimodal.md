# InternVL3：原生多模态预训练

> InternVL3 之前的每一个开源 VLM 都遵循相同的三步方案：拿一个在数万亿文本 token 上训练的文本 LLM，外挂一个视觉编码器，然后微调接缝。这有效，但有对齐债务（Alignment Debt）——文本 LLM 将其全部预训练预算花在了纯文本上，并不能原生理解视觉 token。当你事后添加视觉时，LLM 必须在不遗忘文本的情况下重新学习如何将视觉输入与其文本推理关联起来。InternVL3（Zhu 等人，2025 年 4 月）拒绝了事后方法：一次预训练运行，从第一步起文本和多模态交错。结果是 78B 参数的开源模型在 MMMU-Pro 上匹敌 Gemini 2.5 Pro。本课阅读原生预训练的理由以及当你采用它时发生了什么变化。

**类型：** 学习
**语言：** Python（标准库，训练语料混合器）
**前置条件：** Phase 12 · 05、Phase 12 · 07（方案）
**时间：** 约 120 分钟

## 学习目标

- 解释为什么事后 VLM 训练会积累对齐债务，引用三种可测量的症状（灾难性遗忘、答案漂移、视觉-文本不一致）。
- 描述 InternVL3 的原生预训练语料混合以及为什么文本 : 交错 : 描述的比例很重要。
- 将 V2PE（可变视觉位置编码）与 Qwen2-VL 的 M-RoPE 进行比较。
- 指出视觉分辨率路由器（ViR）和解耦视觉-语言部署（DvD）优化。

## 问题所在

事后 VLM 训练是默认做法。LLaVA、BLIP-2、Qwen-VL、Idefics——都拿一个已经预训练的 LLM（Llama、Vicuna、Qwen、Mistral）并添加视觉。训练阶段通常看起来像：

1. 冻结 LLM + 冻结视觉编码器 + 可训练投影器，在描述对上训练以对齐嵌入。
2. 解冻 LLM，在指令数据上训练（LLaVA-Instruct、ShareGPT4V）。
3. 可选的特定任务微调。

对齐债务的三种症状出现：

- 灾难性遗忘。事后 VLM 忘记了纯文本技能。GSM8K 分数下降 5-10 个点。Hellaswag 分数下降。纯文本代理退化。
- 答案漂移。相同视觉问题的细微措辞变化得到不同答案。视觉编码器与 LLM 的连接比 LLM 自己的 token 更弱。
- 视觉-文本不一致。VLM 可以正确描述一张图像，然后回答一个与其自己的描述相矛盾的问题。视觉 token 不像文本那样参与 LLM 的内部一致性检查。

这些症状有充分的文献记录。MM1.5 第 4 节量化了它们。LLaVA-OneVision 的消融暗示了它们。原生预训练是答案。

## 核心概念

### 原生多模态预训练

InternVL3 在从第一步起就原生多模态的语料上从零开始训练。混合是：

- 40% 纯文本数据（FineWeb、Proof-Pile-2 等）
- 35% 交错图像-文本数据（OBELICS、MMC4 风格）
- 20% 成对图像-描述数据
- 5% 视频-文本数据

视觉 token、文本 token 和跨模态交互从第一个梯度步骤起都参与相同的损失。没有对齐预训练，没有投影器冻结阶段，没有需要恢复的灾难性遗忘。

基础模型的训练是单一阶段。指令微调跟随其后，但基础模型已经将视觉 token 理解为一等公民。

### V2PE（可变视觉位置编码）

Qwen2-VL 使用带有固定轴分配的 M-RoPE。InternVL3 引入了 V2PE：位置编码按模态类型（文本、图像、视频）变化，具有可学习的缩放。实践中：

- 文本 token 获得 1D 位置（文本索引）。
- 图像块获得 2D 位置（行、列）。
- 视频帧获得 3D 位置（时间、行、列）。

三者共享相同的 RoPE 频率基础，但每个频带的隐藏维度分配是可学习的参数，而非固定分割。在预训练期间可以自由地在时间 vs 空间频率分辨率之间进行权衡。

V2PE 的消融声称：在相同算力下，视频基准上比 M-RoPE 高 1-2 个点。不是革命，但更干净。

### 视觉分辨率路由器（ViR）

部署优化。不是所有图像都需要全分辨率编码。一张细节不丰富的单物体照片在用 1280px 原生编码时浪费 token。ViR 是一个小型分类器，在编码之前预测回答所需的最小分辨率。

路由有三个层级：低分辨率（256 个 token）、中（576 个）、高（2048+ 个）。对于生产流量中 60% 的查询，低或中等就足够了。净效果：同等质量下吞吐量提升 2-3 倍。

### 解耦视觉-语言部署（DvD）

当你服务一个大型 VLM 时，视觉编码器每张图像运行一次，但 LLM 为每个输出 token 自回归运行。两个组件有不同的瓶颈（视觉 = GPU 内存带宽用于卷积 + 注意力；LLM = KV 缓存）。DvD 将它们拆分到不同的 GPU 上，之间进行流式传输。

对于 8B + 4 亿编码器的模型，DvD 大约使每节点吞吐量比共置翻倍。

### 单阶段 vs 多阶段质量

InternVL3 的主要基准声称：78B 参数时匹敌 Gemini 2.5 Pro 的 MMMU-Pro。38B 时匹敌 GPT-4o。8B 时引领开源 8B 排行榜。全部在单阶段预训练 + 指令微调方案上。

对齐债务假说是可测量的：InternVL3-8B 在每单位视觉基准增益下比 Qwen2.5-VL-7B 丢失更少的文本基准分数（MMLU、GSM8K）。模型更多的是通才，因为训练是一体而非两块。

### InternVL3.5 与 InternVL-U

InternVL3.5（2025 年 8 月）扩展了该方案。相同的原生预训练方法，更多数据，更多参数。MMMU 改进是增量的。

InternVL-U（2026 年）增加了统一生成——通过在相同骨干之上的 MMDiT 头实现图像输出。「U」代表「理解 + 生成」，追逐 Transfusion 风格的统一模型（第 12.13 课）。相同的原生预训练骨干同时支持理解和生成头。

### 原生预训练的权衡

原生预训练不是免费的：

- 算力。从零开始训练新 VLM 与训练文本 LLM 成本相同——数百万 GPU-小时。事后适配重用现有 LLM 权重，节省大部分成本。
- 数据。大规模交错图像-文本语料很稀缺。OBELICS 有 1.41 亿文档；MMC4 有 5.71 亿。纯文本就能交付 15T token。多模态预训练数据稀缺是一个硬约束。
- 基础 LLM 重用。原生预训练放弃了以后替换新 LLM 的选择。事后可以让你通过仅重新训练适配器来将 Llama-3.1 换成 Llama-4。

InternVL3 下的赌注：对齐债务比重用损失更糟糕。基准支持这一声称。生产成本阻止了未来实验室低成本复制。事后 VLM 将继续存在，因为它们对大多数项目来说仍然更便宜。

## 使用指南

`code/main.py` 是一个训练语料混合器和 ViR 路由器模拟器。它：

- 获取目标语料混合（%文本、%交错、%描述、%视频）并计算每种模态的期望步骤数。
- 在一批查询上模拟 ViR 路由（分布：50% 低细节、30% 中等、20% 高细节）并报告平均 token 数。
- 在给定编码器 vs LLM FLOPs 的情况下报告 DvD 吞吐量估计。
- 打印事后 vs 原生预训练在参数、算力、数据和期望的对齐债务症状方面的并排对比。

## 交付物

本课产出 `outputs/skill-native-vs-posthoc-auditor.md`。给定一个提议的 VLM 训练计划，它审核是走原生还是事后路径，标记对齐债务风险，并推荐语料混合。当你为新的开源 VLM 项目确定规模并需要选择训练策略时使用它。

## 练习

1. 估计 InternVL3-8B（原生预训练）和 LLaVA-OneVision-7B（事后）之间的算力差值。GPU-小时比率大约是多少？什么解释了差距？

2. InternVL3 报告 40% 文本 / 35% 交错 / 20% 描述 / 5% 视频。如果你的目标任务是视频密集型的，提出一个新的比例，并论证为什么基础模型仍然需要大量的文本和描述数据。

3. 阅读 MM1.5 第 4 节关于遗忘的内容。指出事后训练显示最大退化的确切基准。退化有多大？

4. ViR 将 60% 的流量路由到低分辨率编码。什么样的查询它可能错误路由（当需要高分辨率时却发送到低分辨率）？提出三种路由器失败模式。

5. DvD 将视觉和 LLM 拆分到不同的 GPU 上。在什么样的流量模式下 DvD 反而损害而非帮助吞吐量？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| 原生多模态预训练 | 「从头一起训练」 | 文本 + 图像 + 视频 token 从步骤 1 起参与损失，不是后来外挂的 |
| 对齐债务（Alignment debt） | 「事后惩罚」 | 将视觉外挂到冻结 LLM 上产生的文本技能和答案一致性方面的可测量退化 |
| V2PE | 「可变视觉位置编码」 | 按模态的可学习位置编码分配；InternVL3 的 M-RoPE 继任者 |
| ViR | 「分辨率路由器」 | 小型分类器，在编码之前为每个查询选择所需的最小分辨率，节省推理 token |
| DvD | 「解耦部署」 | 视觉编码器在一块 GPU 上，LLM 在另一块上，流式交接；对大型 VLM 使吞吐量翻倍 |
| InternVL-U | 「统一理解 + 生成」 | 2026 年后续版本，在原生预训练骨干之上添加图像生成头 |
| 交错语料 | 「OBELICS / MMC4」 | 以自然阅读顺序包含文本和图像的文档；原生预训练的原材料 |
