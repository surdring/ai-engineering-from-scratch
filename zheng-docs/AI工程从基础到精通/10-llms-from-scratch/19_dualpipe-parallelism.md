# DualPipe 并行

> DeepSeek-V3 在 2,048 块 H800 GPU 上训练，MoE 专家分散在各节点上。跨节点专家全对全通信每 1 GPU-小时的通信对应 1 GPU-小时的计算。GPU 有一半时间空闲。DualPipe（DeepSeek，2024 年 12 月）是一个双向流水线，将前向和后向计算与它们触发的全对全通信重叠。气泡减少，吞吐量攀升，而保持两份模型参数副本（得名"双"）在专家并行已将专家分散到各 rank 的情况下成本很低。本课是一个 Learn 类型的遍历，讲解 DualPipe 实际做了什么，以及为什么 Sea AI Lab 的 DualPipeV 优化以略大的气泡为代价去掉了 2x 参数成本。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**前置知识：** 第十阶段 · 05（分布式训练、FSDP、DeepSpeed），第十阶段 · 14（开放模型架构和 MoE）
**时间：** 约 60 分钟

## 学习目标

- 说出 DualPipe 前向-后向块的四个组成部分，以及为什么每个部分获得自己的重叠窗口。
- 解释大规模下的流水线气泡问题，以及"无气泡"在实践中与营销中的实际含义。
- 为 8 个 PP rank 和 16 个微批次手动追踪 DualPipe 调度，并确认前向和反向流填充彼此的闲置槽位。
- 陈述 DualPipeV（Sea AI Lab，2025）所做的权衡：以专家并行不活跃时略大的气泡为代价，去掉 2x 参数复制。

## 问题

在 2k H800 GPU 上训练 671B MoE 模型遇到三个复合瓶颈：

1. **内存压力。** 每个 GPU 持有模型的一个切片。61 层上序列长度 8k 的 128 头激活内存是巨大的。
2. **流水线气泡。** 传统流水线并行（GPipe、1F1B）在等待其阶段的输入或梯度时让 GPU 闲置。在 8 个阶段下，即使使用 1F1B 调度，大约 12% 的 GPU 时间可能是气泡。
3. **跨节点全对全。** 带有专家并行的 MoE 将专家分散到各节点。每次前向传播触发一个全对全来将 token 调度到其专家，再一个全对全来组合。在 2k GPU 下这很容易变成 1:1 的计通比。

这些每个都有单独的解决方案：梯度检查点用于内存，Zero Bubble（Sea AI Lab，2023）用于流水线气泡，专家并行通信内核用于全对全。DualPipe 所做的是让它们一起运作。调度在单个前向-后向块内将计算和通信重叠，从流水线两端同时注入微批次，并使用产生的调度在全对全窗口中隐藏通信。

报告结果：近乎消除流水线气泡，DeepSeek-V3 的 14.8T token 训练运行中 GPU 利用率超过 95%。

## 核心概念

### 流水线并行复习

将一个 N 层模型拆分到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。微批次前向流经设备 0 到 P-1，然后后向从 P-1 到 0。每个设备只能在前一设备发送其输出时开始其前向阶段，并且只能在下游设备发送上游梯度时开始后向。

GPipe（Huang 等人，2019）一次调度一个微批次，这会浪费大部分 GPU 时间。1F1B（Narayanan 等人，2021）将多个微批次的前向和后向传播交错安排。Zero Bubble（Qi 等人，2023）将后向传播分为两部分——输入后向（B）和权重后向（W）——并调度它们填充气泡。在 Zero Bubble 之后，流水线几乎紧凑。

DualPipe 是下一步。它在此基础上添加了两个想法：

### 想法 1：块分解

每个前向块被拆分为四个组成部分：

- **注意力。** Q/K/V 投影、注意力、输出投影。
- **全对全分发。** 将 token 发送到其专家的跨节点通信。
- **MLP。** MoE 专家计算。
- **全对全组合。** 将专家输出带回的跨节点通信。

后向块添加每个部分的梯度版本。DualPipe 调度它们使得全对全分发与下一个块的注意力计算并行发生，全对全组合与后续块的 MLP 计算并行发生。

### 想法 2：双向调度

大多数流水线调度从阶段 0 注入微批次并向阶段 P-1 流动。DualPipe 从两端注入微批次。阶段 0 看到源自该处的前向微批次；阶段 P-1 也看到源自该处的前向微批次。两个流在中间相遇。

为使这可行，设备 `i` 必须同时持有早期流水线层 `i` 和晚期流水线层 `P - 1 - i`。这就是 DualPipe 的"双"部分：每个设备保留其需要服务的两份模型层副本（每个方向各一份）。在 DeepSeek-V3 的规模下，这是 2x 参数复制成本。这是可承受的，因为专家并行已经将 MoE 专家分散得如此之薄，以至于复制两份非专家层的开销很小。

关键的是，一个方向的前向流和另一个方向的后向流在单向调度中原本是气泡的位置精确重叠。气泡消失。

### 手动追踪调度

考虑 P = 4 个 rank，8 个微批次，分为 4 个前向 / 4 个反向。时间从左到右；行是设备 rank。

```
           时间 →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

解读 "F4/F5R" 标记：rank 1 在同一时间槽中运行微批次 4 的前向（在流水线中从左到右）和微批次 5 的前向（从右到左）。这就是"双向"在操作上的含义。

在 rank 2，交叉流更早重叠，在 rank 0 和 P-1 最晚重叠。在调度的稳定中期阶段，每个 rank 运行 X-方向的前向与 Y-方向的后向重叠。计算繁忙。前向传播的全对全分发隐藏在后向计算中。全对全组合隐藏在前向计算中。气泡被挤出。

### 气泡核算

标准 1F1B 流水线气泡（每个 rank 浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

Zero Bubble 优化将其降低但不是零。DualPipe 在稳定阶段，如果微批次计数被 2 倍流水线深度整除，具有零气泡。在稳定阶段之外（预热和冷却），有一些气泡但它不随微批次数量增长——这是论文强调的关键属性。

用营销术语说："无气泡"。用技术术语说：气泡不随微批次数量增长。Sea AI Lab 的后续分析（DualPipeV / Cut-in-half）显示完全零气泡仅在专家并行不是瓶颈时才成立；在 EP 驱动的全对全下，总是存在一些调度妥协。

### DualPipeV——优化版本

Sea AI Lab（2025）观察到当 EP 通信重叠不是重点时，2x 参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠为在单参数副本上运行的"V 形"调度。气泡比 DualPipe 稍大，但内存节省显著。DeepSeek 在其开源 DualPipe 实现中采纳了 DualPipeV 作为 EP-off 模式。

权衡：

| 特性 | DualPipe | DualPipeV | 1F1B | Zero Bubble |
|---------|---------|-----------|------|------------|
| 每设备参数副本 | 2 | 1 | 1 | 1 |
| 气泡 vs 微批次 | 常数 | 小增长 | 增长 | 增长 |
| 计算-通信重叠 | 完全 | 部分 | 最小 | 部分 |
| 使用场景 | EP-heavy MoE | 稠密或 EP-light | 基线 | 任意流水线 |

### 对 14.8T token 运行意味着什么

DeepSeek-V3 的预训练在 2,048 H800 GPU 上消耗 14.8T token，大约 2.8M GPU-小时。使用朴素的 1F1B，他们会因流水线气泡损失其 12-15%——340-420K GPU-小时，足以训练一个完整的 70B 模型。DualPipe 回收了其中大部分。在没有内部日志的情况下直接量化其贡献是困难的，但论文中的声称是训练期间 GPU 利用率平均超过 95%。

对于较小运行（少于 1k GPU），DualPipe 过度设计——流水线气泡相对于总成本较小，稠密模型训练很少遇到全对全瓶颈。对于数千 GPU 级别的前沿 MoE 训练，它实际上是必需的。

### 在栈中的位置

- 与 **FSDP**（第十阶段 · 05）互补。FSDP 将模型参数分片到各 rank；DualPipe 调度各 rank 的计算。它们组合。
- 兼容 **ZeRO-3** 梯度分片。两份副本的簿记需要与 ZeRO 的分片梯度协作。
- 需要针对特定集群拓扑调优的**自定义全对全内核**。DeepSeek 的开源内核是参考实现。

## 使用方式

`code/main.py` 是一个流水线调度模拟器。它接收 `(P, n_micro_batches, schedule)` 并打印 1F1B、Zero Bubble、DualPipe 和 DualPipeV 各自在稳定阶段的利用率。这是一个教学工具——数字与论文中的定性声称匹配，它们不是对生产实测加速的声称。

模拟器的价值：使用不同的 P 和微批次计数运行它，观察气泡比率如何对 1F1B 增长而对 DualPipe 不增长。

真实训练运行的集成考量：

- 选择一个能整除你的微批次计数的流水线并行深度。
- 确保你的专家并行网格支持双向全对全。DeepSeek 的内核是参考实现。
- 预期第一次要花费一周的调试时间在调度本身上。簿记是繁琐的。
- 监控每个 rank 的 GPU 利用率，而不仅仅是聚合。DualPipe 的好处来自收紧落后节点。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 流水线气泡 | "每个 rank 的空闲时间" | GPU 周期浪费，因为流水线阶段在等待其输入或梯度 |
| 1F1B | "默认流水线调度" | 一个前向/一个后向交错调度；DualPipe 击败的基准 |
| Zero Bubble | "Sea AI Lab 2023" | 将后向分为 B（输入梯度）和 W（权重梯度）；几乎完全收紧流水线 |
| DualPipe | "DeepSeek-V3 调度" | 双向流水线 + 计算-通信重叠；气泡不随微批次计数增长 |
| DualPipeV | "减半" | V 形优化，以略大气泡为代价去掉 2x 参数复制 |
| 块 | "流水线工作单元" | 一个微批次通过一个流水线阶段的一次前向或后向传播 |
| 全对全分发 | "将 token 发送给专家" | 将 token 路由到其指定的 MoE 专家的跨节点通信 |
| 全对全组合 | "将专家输出带回" | 在 MLP 之后收集专家输出的跨节点通信 |
| 专家并行（EP） | "跨 GPU 的专家" | 将 MoE 专家分片到各 rank，使不同 GPU 持有不同专家 |
| 流水线并行（PP） | "跨 GPU 的层" | 将模型层分片到各 rank；DualPipe 所调度的维度 |
| 气泡比率 | "浪费的 GPU 时间" | (bubble_time / total_time)；DualPipe 将其趋近于零的比率 |
