# 推测解码与 EAGLE-3

> 第七阶段 · 第 16 课证明了数学原理：Leviathan 拒绝规则精确地保持了验证器的分布。本课是 2026 年生产级推测解码的训练栈视角。EAGLE-3 将草稿模型从廉价近似转变为一个在验证器自身隐藏状态上训练的专用小型网络，然后添加了一个训练时测试循环，使训练和推理分布对齐。结果：端到端 3× 到 6.5× 的加速，聊天场景下每 token 接受率超过 0.9，且没有分布性权衡。2026 年每个生产推理栈都默认搭载它。

## 学习目标

- 用一句话陈述 Leviathan 定理，并证明推测循环产生的样本与验证器的分布完全相同。
- 遍历从原始推测解码（Leviathan 2023）到 EAGLE、EAGLE-2 再到 EAGLE-3 的两年演进历程，说出每一步移除的具体限制。
- 从接受率 `α` 和草稿与验证器成本比 `c` 计算期望加速比，并为每种情形选择最优草稿长度 `N`。
- 从头实现完整的推测循环：草稿生成、验证、从残差分布进行拒绝采样、在被拒绝时回滚 KV 缓存、在完全接受时输出奖励 token。

## 问题

在 H100 上对 70B 模型进行自回归解码，每秒大约只能生成 35 个 token。GPU 远未饱和。内存带宽是瓶颈：每个 token 需要从 HBM 加载 70B 的权重，执行一步算术运算，产生一个浮点数。计算单元大部分时间处于空闲状态。

推测解码将其转化为一个真正可以解决的吞吐量问题。一个廉价的草稿模型通过 `N` 次小前向传播提出 `N` 个 token。验证器对前缀加上全部 `N` 个草稿 token 运行一次。如果验证器在位置 `i` 的分布与草稿一致（在我们即将精确说明的统计意义上），则接受；如果不一致，则拒绝并从残差分布中采样一个修正。一次大模型前向传播最多产生 `N+1` 个被接受的 token，而不是 1 个。

核心定理是 Leviathan、Kalman、Matias（ICML 2023）提出的：输出分布与直接从验证器采样的结果完全相同。不是近似。是完全相同。这正是推测解码在生产环境中可被接受的唯一原因——它是一个纯粹的延迟优化，没有任何质量上的权衡。

第七阶段 · 第 16 课给出了数学。本课给出的是训练栈。一个好的草稿带来的加速比廉价草稿多 2×。EAGLE、EAGLE-2 和 EAGLE-3（Li 等人，2024-2025）将"草稿 = 同一模型的较小版本"转变为了一门精确的工程学科。2026 年的生产推理服务器默认使用 EAGLE-3。

## 核心概念

### 不变性：Leviathan 拒绝采样

令 `p(t)` 为草稿模型对给定前缀的下一个 token 的分布，`q(t)` 为验证器的分布。采样一个草稿 token `d ~ p`。以概率 `min(1, q(d) / p(d))` 接受。如果拒绝，则从残差分布 `(q - p)_+ / ||(q - p)_+||_1` 中采样。结果样本服从分布 `q`。无论 `p` 有多差，这一点都成立——`p` 越差，拒绝的频率越高，但输出保持精确。

将 `N` 次这样的调用背靠背地堆叠起来，使用对 `前缀 + d_1 + ... + d_N` 的一次验证器前向传播。验证器同时返回 `q_1, q_2, ..., q_{N+1}`。从左到右遍历。在位置 `j` 发生第一次拒绝时，从 `residual(q_j, p_j)` 采样并停止。如果完全接受，则从 `q_{N+1}` 采样一个奖励 token。

### 什么决定了加速比

令 `α` 为每个草稿 token 的期望接受率。令 `c = cost(草稿) / cost(验证器)` 为成本比。每次验证器前向传播的期望接受 token 数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个被接受 token 的期望总墙钟时间为 `(N * c + 1) / E[accepted]`。对 `N` 最小化该值，就得到了最佳点。对于 `α = 0.8, c = 0.05`：最优 `N` 约为 5-7，加速比为 3.2×。对于 `α = 0.95, c = 0.02`：最优 `N` 约为 8-10，加速比可达 5×。

最大的杠杆是 `α`。从 `α = 0.6`（原始草稿）提升到 `α = 0.9`（EAGLE-3），在固定 `N = 5` 的情况下，每次验证器前向传播的期望接受 token 数从 2.2 增加到 4.1。在同一个验证器上获得近 2× 的吞吐量提升。

### 两年演进历程

**原始推测解码（Leviathan, 2023）。** 草稿模型是同一个模型家族中独立训练的较小 LLM。容易接入，`α ≈ 0.6`，加速比最多约 2×。

**EAGLE-1（Li 等人，2024）。** 草稿是一个微型 transformer——通常只有一到两层——以验证器的最后一层隐藏状态作为输入，直接预测下一个 token。因为草稿看到了验证器的特征表示，其分布与验证器的分布接近得多。`α` 提升到 0.7-0.8。

**EAGLE-2（Li 等人，2024）。** 添加了动态草稿树：不是提出一个 `N` 个 token 的单一序列，而是提出一个小型候选树，在一次验证器前向传播中用树注意力对每个候选打分，然后沿最高概率路径前进。草稿长度在每个步骤自适应调整。被接受路径上每个 token 的 `α` 超过 0.85。

**EAGLE-3（Li 等人，2025，NeurIPS）。** 两个额外的改变。第一，完全丢弃特征预测损失——EAGLE-1/2 训练草稿来匹配验证器的隐藏状态，这限制了数据能提供的帮助。EAGLE-3 直接训练 token 预测。第二，训练时测试（TTT）：在草稿训练期间，将草稿自身之前的预测在多步中作为输入反馈回来，与推理时的操作方式相同。这使训练和测试分布对齐，并阻止了误差累积。实测加速比：聊天场景下高达 6.5×，在 H100 上 SGLang 批量大小 64 时吞吐量提升 38%。

### KV 缓存回滚

验证在一次传播中将验证器的 KV 缓存扩展 `N` 个条目。如果拒绝发生在位置 `j`，则位置 `j-1` 之后的缓存内容不再正确。两种常见实现：写入临时缓冲区并在接受时提交（vLLM、TensorRT-LLM），或者维护物理 KV 缓存加逻辑长度并在拒绝时截断。无论哪种方式，回滚成本每个头每层只有若干字节，与前向传播成本相比可以忽略不计。

对于 EAGLE-2 的树搜索，验证器使用遵循树拓扑的非因果掩码运行注意力。工程实现较为复杂，但计算上就是一次带自定义掩码的标准 flash-attention 调用。

### 2026 年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练成本 |
|----------|-----------|-----|---------|---------------|
| 原始 | 单独的较小 LLM | 0.55-0.70 | 1.8-2.3× | 无（复用已有小模型） |
| Medusa | 验证器上的额外 LM 头 | 0.65-0.75 | 2-3× | ~1B SFT token |
| EAGLE-1 | 基于隐藏状态的 1 层 transformer | 0.70-0.80 | 2.5-3× | ~60B token |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80-0.88 | 3-4× | ~60B token |
| EAGLE-3 | 多层特征融合 + TTT | 0.88-0.92 | 3.5-6.5× | ~60-200B token |
| Lookahead | 无草稿（Jacobi 迭代） | N/A | 1.3-1.6× | 无 |

2026 年生产实践中：vLLM 和 SGLang 在可用时默认使用 EAGLE-3，否则使用 EAGLE-2。TensorRT-LLM 对 Meta 和 NVIDIA 公共模型具有最快的 Medusa 路径。llama.cpp 为 CPU 部署提供原始草稿解码。

## 构建实现

见 `code/main.py`。这是完整的 Leviathan 推测循环，包含所有组件：N 草稿、验证器并行传播、逐位置拒绝、残差采样、奖励 token、KV 回滚，以及输出分布与 `q` 直接采样匹配的经验验证。

### 步骤 1：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 步骤 2：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 步骤 3：完整的推测步骤

`spec_step` 函数从 `p` 草拟 `N` 个 token，然后在一次并行 `q` 评估中验证全部。对每个草稿 token 应用拒绝规则，在第一次拒绝时从残差中采样修正。如果全部接受，则从 `q_{N+1}` 输出一个奖励 token。

### 步骤 4：KV 回滚簿记

模拟器为每个 worker 跟踪一个逻辑 `kv_length`。当接受 `k` 个草稿时，`kv_length += k`。当在位置 `j` 拒绝时，缓存已经写入到 `j` 之后，但逻辑长度被设置为 `prefix_length + j + 1`——修正 token 之后一个位置。后续读取截断到逻辑长度。

### 步骤 5：Leviathan 检验

运行 50,000 个推测步骤。统计被接受 token 的经验分布。与 50,000 个 `q` 的直接采样比较。卡方统计量应远低于临界值。该定理在实践中通过。

### 步骤 6：加速比 vs α

通过以不同幅度将 `p` 扰动远离 `q` 来扫描草稿质量。测量 `α`，然后将每次验证器调用的期望 token 数绘制为 `α` 和 `N` 的函数。代码打印一个表格，显示 EAGLE-3 级别的草稿质量（`α ≈ 0.9`）如何在每次验证器调用中解锁 4-5 个 token。

## 使用方式

使用 EAGLE-3 的生产级 `vllm serve`：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

在 H100 批量 64 下使用 SGLang with EAGLE-3：根据 EAGLE-3 论文，吞吐量比批量 64 的原始解码高出约 1.38×。

何时使用推测解码：

- 任何 p50 延迟比峰值吞吐量更重要的交互式聊天工作负载。
- 代码生成和结构化输出（JSON、SQL）。`α` 超过 0.9，因为目标分布高度可预测。
- 长文本生成（数千个 token）。分摊的加速持续产生收益。

何时不使用：

- 非常小的模型（< 3B）。草稿并不比验证器便宜多少。
- 微小的 batch-1 CPU 部署。草稿模型的内存开销可能不值得。
- 极高温度的创意采样，此时 `α` 会崩溃。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Leviathan 规则 | "min(1, q/p)" | 以概率 `min(1, q(d)/p(d))` 进行伯努利接受/拒绝，当从残差采样时精确保持验证器分布 |
| 残差分布 | "(q 减 p) 的正部，归一化" | `(q - p)_+` 在零处截断并重新归一化——拒绝时从中采样的正确分布 |
| 接受率 α | "草稿正确的频率" | 拒绝规则下每个 token 的期望伯努利成功概率；决定所有加速比数学 |
| EAGLE-1 | "隐藏状态草稿" | 以验证器最后一层隐藏状态为条件的微型 transformer 草稿（Li 等人，2024） |
| EAGLE-2 | "动态草稿树" | EAGLE-1 加上在一次验证器传播中用树注意力打分的候选延续树 |
| EAGLE-3 | "训练时测试" | 丢弃特征预测损失，在训练期间将草稿自身的输出反馈给它自己，直接训练 token 预测 |
| 训练时测试（TTT） | "曝光偏差修复" | 在训练期间自回归运行草稿，使训练和测试输入分布一致——计划采样的直接类比 |
| KV 回滚 | "撤销被拒绝的草稿" | 拒绝后将验证器 KV 缓存重置为被接受前缀长度的簿记操作 |
| 奖励 token | "免费的那一个" | 当全部 `N` 个草稿都被接受时，从 `q_{N+1}` 额外采样一个，无需额外验证器成本 |
| 树注意力 | "一次验证多个候选项" | 使用遵循草稿树拓扑的非因果掩码的注意力；在一次前向传播中为树中每个节点计算 `q_i` |
