# RAG vs 长上下文 vs LLM Wiki：三种知识管理范式

## 三种范式

| 维度 | RAG | 长上下文 | LLM Wiki (Karpathy) |
|------|-----|----------|---------------------|
| 知识处理时机 | 查询时（每次） | 查询时（每次） | 录入时（一次） |
| 存储位置 | 外部向量库 | Prompt 中 | 持久化 Markdown Wiki |
| 交叉引用 | 每次临时发现 | 完全由 LLM 推理 | 预建立并持续维护 |
| 矛盾检测 | 基本不做 | 不做 | 录入时标记 |
| 知识积累 | 无 | 无 | 随每次录入复合增长 |
| 延迟 | 检索额外开销 | Token 处理开销 | Wiki 读取开销（低） |
| 上下文长度需求 | 低 | 极高 | 低 |
| 更新成本 | 向量库重索引 | 无（每次 Prompt） | LLM 编辑多个 Wiki 页 |
| 可审计性 | 低（向量不透明） | 中（Prompt 可见） | 高（Git 版本控制） |
| 人可阅读 | 否 | 是（Prompt 中） | 是（结构化 Markdown） |
| 最佳规模 | 大规模 | 单文档 | 中等（~1000 页） |

## 适用场景

### RAG
- 需要查询海量文档（百万级）
- 知识频繁更新
- 用户随机、一次性查询
- 典型产品：NotebookLM、企业知识库搜索

### 长上下文
- 单篇/少量文档深度分析
- 需要完整的文档上下文
- 法律合同审查、论文精读
- 代表：Gemini 1.5 Pro (1M tokens)

### LLM Wiki
- 个人/团队持续积累的知识
- 需要交叉引用和综合分析
- 长期项目（数周到数月）
- 典型场景：研究笔记、课程学习、读书笔记

## 互补关系

三者不是对立的，可以组合使用：

- **LLM Wiki + RAG**：Wiki 存储编译后的知识，RAG 查询高频更新的外部源
- **LLM Wiki + 长上下文**：Wiki 提供知识索引，长上下文用于单文档深度分析
- **三者合一**：Wiki 是知识骨架，RAG 是动态接口，长上下文是深度透镜
