# AI SRE — 多代理事件响应、运维手册、预测性检测

> AI SRE 使用基于基础设施数据（日志、运维手册、服务拓扑）的 LLM，通过 RAG 实现调查、文档化和协调阶段的自动化。2026 年的架构模式是多代理编排（Multi-Agent Orchestration）——由监督者（Supervisor）协调的专门化代理（日志、指标、运维手册）；AI 提出假设和查询，人类批准判断性决策。Datadog Bits AI 和 Azure SRE Agent 将其作为托管产品提供。运维手册正在演进：NeuBird Hawkeye 使用对抗性评估（两个模型分析同一事件；一致 = 置信度，不一致 = 不确定性）；运作记忆（Operational Memory）在团队人员更替中持续存在。自动修复保持谨慎：AI 建议，人类批准。完全自主操作是狭窄的（重启 Pod、回滚特定部署），需配合紧密的护栏（Guardrails）——任何声称"设置后即忘"的人都在夸大。新兴前沿：事件前预测。MIT 研究报告称一个基于历史日志 + GPU 温度 + API 错误模式训练的 LLM 提前 10-15 分钟预测了 89% 的中断。预测：到 2026 年底 95% 的企业 LLM 具备自动故障转移。

## 学习目标

- 画出多代理 AI SRE 架构图：监督者 + 专门化代理（日志、指标、运维手册）+ 人类批准关卡。
- 解释为什么自动修复是狭窄的（重启 Pod、回滚部署）而非广泛的（重新设计服务架构）。
- 说出对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 置信度；不一致 = 升级。
- 引用 MIT 89% 的早期检测结果及其运维约束：没有行动措施的预测只是仪表板。

## 问题

一位值班（On-Call）工程师在凌晨 3 点被呼叫。"结账服务错误率高。"他们检查 Datadog、Loki、三本运维手册、部署日志。30 分钟后他们意识到根因是 KV 缓存峰值导致的 vLLM OOM。他们重启了 Pod；错误清除。

在 2026 年，调查的前 20 分钟是可自动化的。按服务分组日志、与最近的部署关联、与运维手册匹配——这些都是 RAG + 工具使用。一个受监督的代理可以在人类打开 Datadog 之前完成初步分类并提出假设。

完全自主修复是另一个问题。重启 Pod：安全。扩展 GPU 池：如果策略允许则安全。重新设计服务架构：绝对不行。这门纪律是划定那条狭窄的分界线。

## 概念

### 多代理架构

```
          事件
           │
           ▼
        监督者
        /    |    \
       ▼     ▼     ▼
  日志代理   指标代理   运维手册代理
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
        人类批准
             │
             ▼
        行动（狭窄集合）
```

监督者将事件分解为子查询。专门化代理有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合信息，向人类呈现假设 + 证据。人类批准或重定向。

### 自动修复范围

**安全（狭窄）**：重启 Pod，回滚特定部署，在预批准边界内扩展池，启用预批准的特性开关。

**不安全（广泛）**：更改服务拓扑，修改资源限制，部署新代码，更改 IAM，修改数据库。

任何声称"设置后即忘"的人都在夸大。安全集合随着 AI SRE 的成熟而增长，但边界是真实的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们在根因上达成一致，置信度就高。如果不同意，两者假设均可见地升级给人类。简单模式，有效过滤幻觉根因。

### 运作记忆

团队人员更替是传统 SRE 的无声杀手——部落知识流失。AI SRE 将运维手册 + 事后总结（Post-Mortem）存储在向量数据库中；代理在每次新事件时检索。当新工程师加入时，AI 拥有完整的历史记录。

### 事件前预测

MIT 2025 年研究：基于历史日志、GPU 温度、API 错误模式训练的 LLM 在测试集上提前 10-15 分钟预测了 89% 的中断。

现实检验：没有行动措施的预测只是仪表板。运维问题是"当我们预测时，我们做什么？"预防性排空？呼叫？自动扩展？答案是策略特定的。

### 2026 年的产品

- **Datadog Bits AI** — Datadog 内的托管 SRE 副驾驶。
- **Azure SRE Agent** — Azure 原生。
- **NeuBird Hawkeye** — 对抗性评估 + 运作记忆。
- **PagerDuty AIOps** — 分类 + 去重。
- **Incident.io Autopilot** — 事件指挥官 + 协调。

### 运维手册即代码

运维手册从 Confluence 页面演进为带结构化章节（症状、假设、验证、行动）的版本化 Markdown。结构化运维手册提供更好的 RAG 检索。任何 AI-SRE 推出的第一步都是将非结构化运维手册转为结构化。

### 应记住的数字

- MIT 早期检测：89% 的中断，10-15 分钟提前期。
- 多代理分类：监督者 +（日志、指标、运维手册）+ 人类。
- 安全自动修复集合：重启 Pod、回滚部署、在边界内扩展。
- 对抗性评估：两个模型独立；一致 = 置信度。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| AI SRE | "值班代理" | LLM 支持的事件调查 + 协调 |
| Supervisor agent | "编排者" | 将事件分解为子查询的顶层代理 |
| Specialized agent | "领域代理" | 有工具访问权限的子代理（日志、指标、运维手册） |
| Auto-remediation | "AI 修复" | 狭窄的预批准操作；非广泛重新设计 |
| Operational memory | "向量运维手册" | 事后总结 + 运维手册在向量 DB 中以供 RAG |
| Adversarial eval | "双模型检查" | 独立分析；一致 = 置信度 |
| NeuBird Hawkeye | "对抗性那款" | 带有对抗性评估 + 记忆模式的产品 |
| Bits AI | "Datadog 的 SRE 代理" | Datadog 托管的 AI SRE |
| Pre-incident prediction | "早期检测" | 中断预测提前 10-15 分钟 |
