# 批处理 API — 50% 折扣的行业标准

> 每个主要供应商都提供异步批处理 API，带有 50% 折扣和约 24 小时周转时间。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 批处理层、Together 批处理）都实现了相同的模式。将批处理与提示词缓存叠加，夜间管线的成本降至同步无缓存成本的约 10%。规则残酷而简单：如果不是交互式的，就属于批处理。内容生成管线、文档分类、数据提取、报告生成、批量标注、目录标记——任何能容忍 24 小时延迟的工作负载，只要还没迁移到批处理，就是在白白浪费金钱。2026 年的生产模式是将每个新的 LLM 工作负载分流到三条通道：交互式（带缓存的同步调用）、半交互式（带回退的异步队列）、批处理（夜间运行，叠加缓存输入）。那些假装是交互式但可以容忍分钟级延迟的工作负载浪费最多。

## 学习目标

- 说出三个供应商的批处理 API（OpenAI、Anthropic、Google）以及通用的 50% 折扣 + 24 小时周转保证。
- 计算在一个夜间分类工作负载上叠加批处理 + 缓存输入的成本，并与同步无缓存基准对比。
- 将工作负载分流为交互式 / 半交互式 / 批处理，并说明每个通道的理由。
- 说出两个陷阱：部分交互性（用户期望快于 24 小时）和输出格式漂移（批处理文件格式因供应商而异）。

## 问题

你的团队交付了一个夜间报告生成管线。50,000 篇文档，每篇摘要，聚类摘要，撰写执行简报。同步运行需要 4 小时，花费 $2,000/晚。你听说了批处理 API。

批处理给你 50% 折扣。你还在系统提示词上启用了提示词缓存（所有 50k 次调用共享）。叠加后，账单降到 $180/晚——约为基准的约 9%。同样的管线，三个配置改动。

批处理是 LLM 成本工具包中最便宜但没人用的杠杆。原因主要是组织性的：团队认为"实时"但实际上 SLA 是"明早前"。本课旨在不让 90% 的账单白白浪费。

## 概念

### 三个批处理 API

**OpenAI Batch API**：上传 JSONL 文件，包含请求列表。承诺 24 小时周转（实践中通常约 2-8 小时）。输入和输出 token 享受 50% 折扣。`/v1/batches` 端点。符合条件的输入也会叠加缓存输入定价。

**Anthropic Message Batches**：JSONL 上传。24 小时周转。50% 折扣。支持 `cache_control`——缓存写入是显式的，批处理内读取自动发生。

**Google Vertex AI Batch Prediction**：BigQuery 或 GCS 输入。Gemini 类似 50% 折扣。与 Vertex 管线集成。

### 语义：异步，而非缓慢

批处理是"我承诺 24 小时内返回"——而不是"这将花费 24 小时"。典型 P50 是 2-6 小时。供应商在 GPU 库存利用率较低的非高峰窗口调度你的批处理。

### 与缓存叠加

一个有相同 4K token 系统提示词的 50k 文档摘要任务：

- 同步无缓存：50000 ×（$input × 4000 + $output × 200）按全价计费。
- 同步缓存：系统提示词在第一次写入后缓存；剩余 49999 次以十分之一价格获得输入。
- 批处理缓存：以上所有加上读写均享受 50% 折扣。

叠加：批处理 + 缓存 = 同步无缓存账单的约 10%。任何夜间运行且有共享系统提示词的工作负载都应使用此方案。

### 工作负载分流

**交互式** — 用户等待响应。TTFT 重要。带提示词缓存的同步调用。不可批处理。

**半交互式** — 用户提交任务，几分钟后回来查看。异步队列，批处理不可用时回退到同步。例如中等量的 RAG 索引。

**批处理** — 用户期望"明早前"或"下一小时内"出结果。内容管线、大规模分类、离线分析。始终批处理，始终叠加缓存。

常见错误：将一切归类为交互式，因为管线是生产级别的。生产不是延迟规格——SLA 才是。

### 部分交互性陷阱

有些功能看起来是交互式的，但可以容忍 5-10 分钟。例如：带"刷新"按钮的夜间客户健康报告。用户点击刷新；等 10 分钟没问题。团队将其作为同步交付。50 个并发刷新比批处理后通过邮件交付贵 10 倍。

要问的问题是："24 小时对这个用户意味着什么？"如果答案是"他们不会注意到"，就批处理它。

### 输出格式陷阱

批处理文件格式因供应商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入其中。
- Vertex：BigQuery 表或 GCS 前缀的 TFRecord。

编写跨供应商的"单一批处理客户端"意味着每个供应商需要适配代码。那些宣传多供应商批处理的网关（Portkey、LiteLLM 部分层级）仍然是对原始格式的薄封装。

### 应记住的数字

- 跨供应商批处理折扣：输入 + 输出统一 50%。
- 周转 SLA：保证 24 小时，典型 P50 2-6 小时。
- 叠加批处理 + 缓存输入：同步无缓存成本的约 10%。
- 工作负载分流规则：如果 24 小时延迟可接受，始终批处理。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| Batch API | "异步折扣" | 50% 折扣，24 小时周转 |
| JSONL | "批处理格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批处理" | Anthropic 的批处理 API 产品名 |
| Batch prediction | "Vertex 批处理" | Vertex AI 的批处理 API 产品 |
| Turnaround SLA | "24 小时承诺" | 保证值，而非典型值；典型是 2-6 小时 |
| Workload triage | "交互性决策" | 交互式 / 半交互式 / 批处理路由决策 |
| Output schema | "响应格式" | 各供应商的 JSONL 布局；不可移植 |
| Stacked discount | "批处理 + 缓存" | 两者同时适用时约为无缓存同步账单的 10% |
